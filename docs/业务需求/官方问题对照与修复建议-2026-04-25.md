## 2. 当前回归现状总结

基于 `docs/业务需求/问数回归测试结果-2026-04-25.md`，当前结论可以先归纳为：

- TiDB workspace 建模主链路已经能跑通，包括：
  - 新建 workspace
  - 新建知识库
  - 连接 TiDB 数据源
  - 生成 suggested questions
  - modeling AI assistant 生成语义描述
  - modeling AI assistant 生成关联关系
  - 导入分析规则
  - 导入 SQL 模板
- 规则类问题已经明显改善，尤其是：
  - `R02 首存定义`
  - `R13 缺失外部数据源处理`
  - `R14 ES 数据使用限制`
- 模板类问题已部分稳定命中 `sql_pairs`，但复杂模板仍然失败较多。
- 当前最核心的问题已经不是“能不能导入 / 能不能走通链路”，而是：
  - 对明确业务口径模板，是否能**严格按模板语义锚定或参数化执行**
  - correction 阶段是否会**改坏原本正确的模板骨架**
  - schema retrieval 不足时，是否还会**把本该命中的模板题带偏**

一句话总结：**链路已基本打通，但“复杂模板问数的稳定准确执行”仍未达标。**

---

## 3. 失败的主要原因

### 3.1 最主要问题：明确业务口径模板仍按普通参考样例处理

当前系统虽然已经能命中部分 `sql_pairs`，但命中之后会把模板当作 LLM 参考样例，而不是区分“普通参考样例”和“明确业务口径模板”。

这种参考生成逻辑本身没有问题，适合开放式问数；问题在于 `docs/业务需求` 这批模板中有一部分已经具备固定业务口径、固定聚合结构和预期结果，更适合进入可选的 L2 锚定模板或 L3 参数化执行。

这会导致两个后果：

- 简单题可以借助样例成功生成正确 SQL；
- 一旦问题涉及多层聚合、分桶、cohort、TOPN 边界、按日分桶等复杂逻辑，LLM 会在模板基础上进行自由改写，最后把原始模板口径改偏。

这也是为什么以下问题即便已经进入 `sql_pairs` 路径，仍然会失败：

- `T08 首存 cohort 续存`
- `T09 所有用户区间汇总`
- `T10 首存用户日龄趋势`
- `T13 首存金额分桶`

### 3.2 correction 阶段会把复杂 SQL 改坏

当前 ask 流程中，SQL 生成失败后会进入 correction 路径。这个机制对简单 SQL 有帮助，但对复杂模板题风险很高：

- correction 往往不知道哪些部分是“业务骨架”，哪些只是“语法细节”；
- 一旦把 `CASE WHEN`、日期分桶、cohort 过滤、TOP3/非TOP3 边界等结构重写，结果就会偏离业务口径；
- 于是表面上看是“修 SQL”，实际上是在“重写模板”。

这也是本轮里 `T01`、`T04`、`T09`、`T10` 一类问题容易出现 correction 后语法或口径继续漂移的原因。

### 3.3 路由与召回仍然不够稳定

当前系统仍有一部分问题会出现：

- 本应走 `sql_pairs`，却掉到 `general`
- 本应按业务规则回答，却掉到 `USER_GUIDE`
- 命中 `sql_pairs` 后，schema retrieval 结果又不足，最终生成链路缺上下文

这说明路由问题虽然已经改善，但还没有完全收敛。

### 3.4 schema retrieval 对复杂中文业务问法支撑不足

复杂中文业务问题往往同时包含：

- 时间粒度
- 渠道条件
- cohort / 日龄 / 首存 / ROI 等业务术语
- 分层规则

如果 schema retrieval 只取回很少的表上下文，后续 LLM 就会更依赖“猜”，进而放大 SQL 漂移和 correction 漂移。

### 3.5 一部分问题是我们当前分支特有的 runtime / workspace 逻辑问题

本轮已经修掉了一批并非官方主仓共性、而是我们当前知识库 workspace 化之后引入的问题，例如：

- `thread` 主链路读取异常
- instruction / sql_pair 随 `deployHash` 漂移，导致部署后“看不见”
- TiDB/MySQL 风格 SQL 模板被导入时的 dry-run 校验直接拦住

这些问题会严重影响测试执行，但它们不是“文本到 SQL 能力本身”的通用问题，而是我们当前分支的 runtime scope / KB scope 设计问题。

## 5. 官方逻辑与我们当前代码逻辑是否相同

结论先说：

- **一部分是同源问题，逻辑接近；**
- **一部分已经不是同一套实现，我们当前分支有明显 fork / 重构后的特有逻辑。**

### 5.1 ask 路由与 general / instructions 相关逻辑：同源，但我们已做本地增强

在 `wren-ai-service` 侧，以下两个文件与官方主仓仍然是同一路径、同一类职责：

- `wren-ai-service/src/pipelines/generation/data_assistance.py`
- `wren-ai-service/src/pipelines/generation/intent_classification.py`

对比官方主仓后可以看到：

- 我们本地已经增加了 `USER INSTRUCTIONS` 注入；
- 我们本地增加了“业务规则优先走 `GENERAL`，不要误判成 `USER_GUIDE`”的约束；
- 我们本地还加强了“缺失外部数据源时主动索要，而不是擅自替换”的规则。

这说明：

- **这里的基础逻辑与官方是同源的；**
- **但我们已经针对本轮业务规则场景做了本地补丁。**

也正因为它们同源，所以“general 吃掉业务问题、instructions 不足、路由误判”这类问题本质上属于官方也存在的共性方向。

### 5.2 runtime scope / knowledge workspace / deployHash 资产可见性：不相同，属于我们当前分支特有逻辑

我们当前实际修改的是这些文件：

- `wren-ui/src/server/repositories/sqlPairRepository.ts`
- `wren-ui/src/server/repositories/instructionRepository.ts`
- `wren-ui/src/server/services/sqlPairService.ts`
- `wren-ui/src/server/services/instructionService.ts`
- `wren-ui/src/server/controllers/modelControllerRuntimeActions.ts`

而官方主仓对应的历史实现仍主要在：

- `wren-ui/src/apollo/server/repositories/sqlPairRepository.ts`
- `wren-ui/src/apollo/server/repositories/instructionRepository.ts`
- `wren-ui/src/apollo/server/services/sqlPairService.ts`
- `wren-ui/src/apollo/server/services/instructionService.ts`

官方主仓这些文件的特点很明显：

- 资产基本以 `projectId` 为主 scope；
- repository/service 逻辑相对简单；
- 不存在我们现在这套以 `workspaceId + knowledgeBaseId + kbSnapshotId + deployHash` 为核心的运行时身份设计。

这意味着：

- `thread` 读取异常
- instruction / sql_pair 随 deployHash 漂移
- KB scope 下资产部署后不可见

这些问题**不是官方主仓当前同一实现里直接存在的问题**，而是我们当前知识库 workspace 化和 runtime scope 化之后新增的复杂度。

### 5.3 “命中 sql_pair 但仍自由生成 SQL”这一点：方向相同，限制也相同

无论是官方主仓，还是我们当前分支，`sql_pair` 目前更像是：

- 检索增强样例
- SQL generation / correction 的参考上下文

而不是：

- 具备可选的“业务模板锚定 / 参数化执行”能力

这点非常关键。

也就是说，**在“模板不是模板引擎，只是模板样例”这一根本设计上，我们和官方逻辑是相同方向的。**

这也是为什么官方虽然有：

- instructions 注入
- ask experience 改进
- SQL knowledge 注入

但依然不能直接解决我们当前 `T08/T09/T10/T13` 这一类复杂业务模板问数失败。

## 6. 本轮判断：哪些问题是“官方共性”，哪些是“我们分支特有”

### 6.1 官方共性问题

以下问题可以判断为官方也存在或至少同方向存在：

- general / user guide / text-to-sql 的边界不稳定
- instructions 已能传入 SQL generation / correction，但仍属于 prompt 约束，不是确定性执行约束
- LLM 生成 SQL 时对目标数据库方言适配不稳定
- 复杂 SQL 在 correction 或底层处理阶段容易漂移
- 通过“知识注入”提升质量有帮助，但仍不足以替代确定性的业务模板锚定 / 参数化执行

### 6.2 我们当前分支特有问题

以下问题更像我们当前分支的 runtime / workspace 特有问题：

- `thread` 主链路异常
- knowledge workspace 场景下 instruction / sql_pair 的 scope 绑定与读取逻辑复杂化
- `deployHash` / `kbSnapshotId` 切换后资产可见性异常
- TiDB 回归依赖的导入 API / KB API 行为与官方 project-scope 旧实现不完全一致
- `/knowledge` 页面请求风暴和 `ERR_INSUFFICIENT_RESOURCES`

## 7. 建议方案

### 7.1 核心原则：保留 `sql_pair` 参考生成，不把所有模板硬执行

当前 `sql_pair` 作为 SQL generation 的参考样例是合理的，不应该整体改成硬模板执行。

更合适的方向是：**保留现有 `sql_pair` 参考生成逻辑，同时新增一个可选的业务模板锚定 / 参数化执行 lane。**

这个 lane 只处理满足高置信条件的业务模板，尤其是 `docs/业务需求/knowledge-base/sql-templates/` 中这类已经明确业务口径、固定聚合结构、可回归验证的模板。其他普通 SQL pair 继续作为参考样例参与生成。

注意：正式导入来源也只是高置信候选，不代表自动进入 L3 执行；仍必须通过参数完整度、口径兼容、schema / dialect 验证和 dry-run。

### 7.2 P0：定义 SQL pair / 模板的分层行为

建议把 SQL pair / SQL 模板分成四层，而不是“参考生成”和“硬执行”二选一：

| 层级 | 名称 | 行为 |
| --- | --- | --- |
| L0 | 普通 SQL pair | 只作为样例参考，保留当前逻辑 |
| L1 | 可信参考 | 提高召回权重和提示权重，但仍允许 LLM 生成 SQL |
| L2 | 锚定模板 | SQL 关键骨架不能改，只允许补参数、轻微适配方言或标识符 |
| L3 | 可执行模板 | 参数抽取后直接渲染 SQL 执行，LLM 不重写 SQL 骨架 |

本轮失败的 `T08/T09/T10/T13` 更像 L2 / L3 业务模板，但系统目前仍按 L0 参考样例处理，所以会出现“命中了模板，但结果仍然错”的情况。

### 7.3 P0：定义“高置信模板”的门槛

这里的“高置信”不是单纯 embedding 相似度高，而是多个信号同时满足：

1. **来源可信**
   - 来自正式业务需求导入；
   - 或由管理员 / 建模人员显式标记为业务模板；
   - 或由用户保存后进一步确认“以后类似问题按这个口径算”；
   - 或经过多次问数复用并确认结果正确。
2. **语义命中接近**
   - 当前问题和模板问题高度相似；
   - 第一候选与第二候选模板有明显差距；
   - 用户问题没有明显超出模板能力范围。
3. **参数可以完整抽取**
   - 日期范围、渠道、游戏类型、TOPN、分桶条件等参数可明确抽取；
   - 没有缺失关键参数；
   - 没有出现模板无法表达的新条件。
4. **业务口径兼容**
   - 指标、分组、过滤条件、时间粒度、cohort / 分桶 / TOPN 口径与模板一致；
   - 用户没有新增会改变 SQL 骨架的分析维度。
5. **SQL 骨架适合模板化**
   - 适合 cohort、分桶、TOPN / 非TOPN、首存日龄、日期分桶等固定业务结构；
   - 不适合开放探索式问题。
6. **执行前验证通过**
   - 渲染后的 SQL dry-run 通过；
   - 当前 TiDB workspace 下表、字段、方言均兼容；
   - 如有基准结果，能通过回归校验。

建议使用 gating，而不是只看一个分数：

```text
template_confidence =
  source_score
  + retrieval_score
  + margin_score
  + parameter_coverage_score
  + intent_compatibility_score
  + schema_compatibility_score
  - ambiguity_penalty
  - extra_dimension_penalty
```

初始建议阈值：

- `< 0.75`：普通参考生成；
- `0.75 - 0.9`：锚定生成，模板作为强约束；
- `>= 0.9`：参数化模板执行；
- 参数缺失或口径冲突时，无论相似度多高，都不能进入模板执行。

这些阈值只是初始建议，不能直接当作最终产品阈值。落地前应基于 `docs/业务需求` 回归问题、历史问数问题和人工构造的近似问题集校准召回分布、候选差距和误触发率。

### 7.4 P0：用户保存的 SQL 不默认等于高置信模板

用户在对话过程中保存的 SQL 可以作为强来源信号，但不能默认等于 L3 可执行模板。

建议产品语义分层：

| 用户动作 | 建议系统层级 | 说明 |
| --- | --- | --- |
| 保存 SQL | L0 / L1：参考样例 | 以后作为相似问题的参考 |
| 保存并确认“以后类似问题按这个口径算” | L2：锚定模板 | 后续相似问题优先按该口径生成 |
| 保存并完成参数槽识别、dry-run、回归验证 | L3：可执行模板 | 可进入参数抽取 + 模板渲染执行 |

因此，“用户保存”应进入 `trusted_reference`，只有额外完成口径确认和参数化验证后，才升级为 `anchored_template` 或 `executable_template`。

### 7.5 P0：复杂模板命中后，限制 correction 对骨架 SQL 的改写

对于 L2 / L3 模板，correction 只能修局部兼容问题，不能重写业务骨架。

允许修：

- 轻微语法问题；
- 标识符引用问题；
- quote / alias / limit 等方言适配问题；
- 当前数据源下可证明等价的函数替换。

禁止改：

- 聚合结构；
- 分桶逻辑；
- cohort 层级；
- TOP3 / 非TOP3 `CASE` 逻辑；
- 日期粒度；
- 业务过滤条件；
- 指标公式。

如果 correction 需要修改这些结构，应降级为普通生成或要求用户确认，而不是静默重写。

### 7.6 P0：继续保留并加强 GENERAL 路径的业务规则约束

这块本轮已经有明显收益，不建议回退。

建议继续保持：

- `GENERAL` 可显式读取业务规则 instructions；
- 业务定义 / 缺失外部数据源处理 / ES 使用限制一律优先按规则回答；
- 不再默认回到 Wren AI 用户指南。

这块是规则题正确性的基础盘。

### 7.7 P1：补充 ask diagnostics，先把失败链路“可观测化”

建议在 ask 响应或日志里增加这些诊断信息：

- 命中的 `sql_pair` / template id、title、score、候选差距
- instruction 命中数
- schema retrieval 的表数 / ddl 数
- 当前 ask path
- 是否进入 correction
- correction 重试次数
- 最终 SQL 来源：普通生成、锚定模板、可执行模板、correction 改写
- 参数抽取结果与缺失参数

没有这些诊断，后续每次回归只能凭现象猜原因，修复效率会很低。

### 7.8 P1：吸收官方 #1930 的数据库过滤思路

这不是最急的问题，但值得做：

- setup / 连接器导入时更严格按指定数据库过滤资产；
- 避免 TiDB workspace 引入不相关 schema，降低后续建模与 retrieval 噪音。

### 7.9 P1：补齐 TiDB / MySQL SQL knowledge 内容

官方 #2043 已经提供动态 SQL knowledge 注入方向，本地也已经有 `sql_knowledge_retrieval` 接入生成 / 纠错链路。下一步更应该检查内容层：

- TiDB / MySQL 是否有明确方言规则；
- 业务模板渲染后的 SQL 是否符合当前数据源方言；
- SQL knowledge 是否覆盖日期函数、窗口函数、CTE、limit、identifier quote 等常见差异。

这能提高普通 NL2SQL 和 L1/L2 锚定生成质量，但仍不能替代 L3 可执行模板。

### 7.10 P2：再评估官方 #1874 的 ask experience 大改

#1874 适合后续体验层面再评估，不建议作为当前复杂业务模板准确性的主修复路径。

### 7.11 前端改动建议：MVP 只做轻量标记和可解释展示

业务模板锚定 / 参数化执行不是纯后端能力，前端也需要提供最小表达入口和结果解释。

前端改动的目标不是一开始做复杂模板管理器，而是先解决两个问题：

1. 用户 / 管理员保存 SQL 时，能表达这个 SQL 的用途；
2. 问数回答时，能看出本次 SQL 是普通参考生成、业务口径锚定，还是参数化模板执行。

#### 7.11.1 保存 SQL 时增加用途选择

第一阶段建议只做最小选项：

```text
保存为：
- 参考样例（默认）
- 业务口径
```

后续再扩展：

```text
保存为：
- 参考样例
- 业务口径
- 参数化模板
```

建议默认值：

- 普通用户点击“保存 SQL”：默认保存为 `reference`；
- 用户明确选择“以后类似问题按这个口径算”：保存为 `business_template`；
- 只有完成参数槽识别、dry-run、回归验证后，才升级为 `executable_template`。

#### 7.11.2 SQL pair / 知识库列表展示模板类型

在 SQL pair / SQL templates 管理列表中展示类型：

```text
类型：参考样例 / 业务口径 / 参数化模板
```

第一阶段只需要展示，不强制做复杂筛选。后续可增加：

```text
全部 / 参考样例 / 业务口径 / 参数化模板
```

#### 7.11.3 问数回答区展示执行来源

在 thread answer 中增加轻量 badge 或调试信息：

```text
已参考 SQL 样例生成
已按业务口径模板生成
已执行参数化模板
```

如果模板未被直接采用，可以在 diagnostics 或 hover 中展示原因：

```text
未直接套用模板：问题新增了模板不支持的“渠道”维度
未直接套用模板：缺少必填参数 start_date / end_date
未直接套用模板：候选模板置信度不足
```

第一阶段可以先只对管理员 / debug 模式展示，避免干扰普通问数体验。

#### 7.11.4 后续模板管理器再分阶段建设

后续可建设完整模板详情页，展示：

- 模板名称；
- 模板类型；
- 适用问题；
- 参数槽；
- SQL 骨架；
- 最近命中次数；
- 最近成功率；
- 最近失败原因。

也可以支持参数槽配置：

- 参数名，例如 `start_date`、`end_date`、`channel`、`top_n`、`game_type`、`bucket_rule`；
- 参数类型；
- 是否必填；
- 默认值；
- 可选值来源；
- 示例问法。

当某个普通 SQL pair 多次被复用成功时，系统可以提示：

```text
这个 SQL pair 已被多次相似问题复用成功，是否升级为业务口径？
```

但这些都不应阻塞第一阶段 MVP。

#### 7.11.5 可能涉及的前端区域

第一阶段可能涉及：

- 保存 SQL 的 modal，例如 `QuestionSQLPairModal` / `AdjustSQLModal` 一类组件；
- 知识库 SQL pair / SQL templates 管理列表；
- 问数 thread answer 区域；
- knowledge asset wizard / 导入入口，如果要在导入时区分“参考样例 / 业务模板”。

建议第一阶段只做：

1. 保存 SQL 时选择“参考样例 / 业务口径”；
2. SQL pair 列表展示类型；
3. 回答区展示模板来源 badge 或 debug diagnostics。

### 7.12 数据模型与 API 字段建议

为避免 L0/L1/L2/L3、`reference`、`business_template`、`anchored_template`、`executable_template` 等概念在实现时混用，建议先统一数据模型。

建议 SQL pair / template 增加这些核心字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| `assetKind` | `sql_pair` / `sql_template` | 区分普通样例和业务模板资产 |
| `templateLevel` | `L0` / `L1` / `L2` / `L3` | 当前资产可被系统使用的最高等级 |
| `templateMode` | `reference` / `trusted_reference` / `anchored_template` / `executable_template` | 对外更易理解的模式名 |
| `sourceType` | `user_saved` / `admin_marked` / `business_import` / `system_promoted` | 来源与基础可信度 |
| `scopeType` | `personal` / `workspace` / `knowledge_base` | 影响范围 |
| `parameterSchema` | JSON schema | 参数槽定义，例如日期、渠道、TOPN、游戏类型 |
| `businessSignature` | hash / JSON | 业务骨架签名，用于判断 correction 是否改坏核心结构 |
| `templateVersion` | `1` / `2` | 模板版本 |
| `status` | `draft` / `active` / `deprecated` | 生命周期状态 |
| `effectiveFrom` / `effectiveTo` | 日期 | 业务口径生效区间 |
| `approvedBy` / `approvedAt` | 用户 / 时间 | 审批记录 |

ask 运行时建议返回这些诊断字段：

| 字段 | 说明 |
| --- | --- |
| `templateDecision.mode` | 本次是普通生成、可信参考、锚定模板还是可执行模板 |
| `templateDecision.templateId` | 命中的模板 id |
| `templateDecision.score` | 召回得分 |
| `templateDecision.margin` | 第一候选和第二候选差距 |
| `templateDecision.parameters` | 参数抽取结果 |
| `templateDecision.missingParameters` | 缺失参数 |
| `templateDecision.decisionReason` | 进入 / 未进入模板 lane 的原因 |
| `templateDecision.fallbackReason` | 降级原因 |
| `templateDecision.sqlSource` | `generated` / `anchored_generated` / `rendered_template` / `corrected` |

这些字段应优先服务 diagnostics 和回归测试，不一定第一阶段全部展示给普通用户。

### 7.13 模板决策、冲突处理与降级策略

业务模板 lane 必须支持可解释的降级，而不是只有“命中 / 不命中”。

建议决策顺序：

1. 召回候选 SQL pair / template；
2. 过滤掉 `templateLevel` 不足的候选；
3. 计算候选差距和业务口径兼容性；
4. 抽取参数并校验参数完整度；
5. 校验 schema / dialect 兼容性；
6. 根据等级进入 L0/L1/L2/L3；
7. 如果任一硬 gate 不通过，则降级或追问。

硬 gate 建议包括：

- 参数缺失：不进入 L3；
- 口径冲突：不进入 L2/L3；
- 候选差距不足：不自动执行模板；
- 用户新增模板不支持的维度或分析动作：降级到参考生成或追问；
- dry-run 失败：不执行 L3；
- 模板已 deprecated 或不在生效期：不命中。

候选冲突处理：

- 如果两个模板得分接近，且都能解释用户问题，优先追问或降级参考生成；
- 不应只因为第一名略高就进入 L3；
- diagnostics 中记录冲突候选和未执行原因。

追问场景处理：

- 如果初问使用 L2/L3，追问可以继承模板上下文；
- 但每次追问仍必须重新做参数完整度、口径兼容和新增维度判断；
- 如果追问只是修改日期、渠道、TOPN 等参数，可继续使用同一模板；
- 如果追问新增模板不支持的维度、指标或分析动作，应降级为 L1 参考生成或向用户澄清。

L2 骨架保护建议：

- 不允许改变 CTE 层级；
- 不允许改变聚合层级和 `GROUP BY` 粒度；
- 不允许改变窗口函数分区 / 排序口径；
- 不允许改变 `CASE` 分桶 / TOPN / cohort 逻辑；
- 不允许改变核心业务过滤条件；
- 只允许局部标识符、quote、函数名、limit、alias 等兼容性修正。

### 7.14 安全、权限与审计约束

参数化模板执行会直接生成可执行 SQL，因此需要比普通参考生成更明确的安全边界。

权限建议：

| 操作 | 权限建议 |
| --- | --- |
| 保存为参考样例 | 普通用户可保存到个人或当前知识库允许范围 |
| 标记为业务口径 | workspace owner / knowledge admin / 有知识库写权限的角色 |
| 升级为可执行模板 | 管理员或经过审批流程 |
| 修改参数槽 / SQL 骨架 | 管理员或模板 owner |
| 废弃模板 | 管理员或模板 owner |

安全约束：

- 参数必须有类型定义，例如 date、enum、number、identifier list；
- 不允许把未验证自由文本直接拼进 SQL；
- 枚举参数应来自白名单或当前 schema / 业务配置；
- 日期、数字、TOPN 等参数要做范围校验；
- identifier 类参数必须来自已知表字段或模板允许列表；
- L3 渲染 SQL 后必须 dry-run；
- 执行时保留现有 limit / preview / 权限控制；
- 所有模板决策、参数抽取、渲染 SQL、降级原因都要写入审计或 trace。

审计建议记录：

- 谁保存 / 标记 / 升级模板；
- 谁修改参数槽或 SQL 骨架；
- 本次问数是否进入 L2/L3；
- 命中模板、参数、SQL 来源、执行结果和 fallback reason。

### 7.15 阈值校准与版本管理

高置信阈值需要通过数据校准，而不是一次性写死。

建议准备三类校准样本：

1. 正例：明确应命中某个业务模板的问题；
2. 近似负例：看起来相似但口径不同的问题；
3. 扩展问题：在模板基础上新增维度、指标或分析动作的问题。

每次调整阈值时至少观察：

- 模板正确命中率；
- 误执行率；
- 降级率；
- 追问触发率；
- T03/T06/T11/T12 等原本通过模板是否回退；
- T09/T10/T13 等失败模板是否改善。

版本管理建议：

- 业务模板修改后生成新 `templateVersion`；
- 旧版本可以 deprecated，但保留历史问数可追溯性；
- 模板可设置生效时间，避免新旧业务口径混用；
- 回归报告中记录使用的模板版本。

## 8. 建议的实施顺序

建议按下面顺序推进：

1. **先补最小 diagnostics 和数据字段**
   - 让 ask 结果能记录命中模板 id、score、margin、参数抽取、决策原因和 fallback reason；
   - 这一步应先于复杂 gating，否则后续无法判断是召回、参数、口径还是 correction 问题。
2. **做业务模板锚定 / 参数化执行最小闭环**
   - 第一批覆盖最关键失败模板：
     - `T09 所有用户区间汇总`
     - `T10 首存用户日龄趋势`
     - `T13 首存金额分桶`
   - 目标不是替换所有 `sql_pair`，而是让这些明确业务口径模板从 L0 升级到 L2 / L3；
   - `T08 首存 cohort 续存` 作为第二批，原因是它更依赖 cohort 跨日续存口径和追问式解释，适合在第一批模板 lane 稳定后纳入。
3. **补模板高置信 gating**
   - 实现来源、相似度、候选差距、参数完整度、口径兼容、schema / dialect、dry-run 的综合判断；
   - 阈值先作为配置项，不写死。
4. **做 correction 保护**
   - 防止 L2 / L3 复杂模板命中后被 correction 改坏；
   - correction 超出局部兼容修正时必须降级、失败或要求确认。
5. **补前端 MVP，可与第 2-4 步并行**
   - 保存 SQL 时支持“参考样例 / 业务口径”；
   - SQL pair 列表展示模板类型；
   - 问数回答区展示本次 SQL 来源；
   - 回答区 badge 可后置，但保存用途字段应尽早接入。
6. **补完整 diagnostics 与阈值校准**
   - 建立正例、近似负例、扩展问题样本集；
   - 记录误执行率、降级率、追问率和原通过模板回退情况。
7. **最后再吸收上游可借鉴项**
   - 如 #1930 的数据库过滤思路、#2043 的 SQL knowledge 内容机制、#1874 的 ask experience 改进。

### 8.1 P0 验收标准与回归测试矩阵

P0 不应只以“功能跑通”为验收，而应以“复杂业务模板准确率提高且既有通过场景不回退”为验收。

建议 P0 验收标准：

| 验收项 | 标准 |
| --- | --- |
| 第一批模板 | `T09 / T10 / T13` 在相同 TiDB workspace 下通过 |
| 既有通过模板 | `T03 / T06 / T11 / T12` 不回退 |
| 规则类问题 | `R02 / R13 / R14` 继续通过 |
| 缺失外部数据源 | `T05 / T14 / T15` 继续走兜底，不编造外部指标 |
| correction 保护 | L2 / L3 模板不允许被 correction 改写核心骨架 |
| 追问 | 初问使用模板时，追问可继承上下文但必须重新做兼容性判断 |
| 图表 | 基于通过 SQL 的图表生成不回退 |
| diagnostics | 每次 ask 可看到模板命中、参数、决策和 fallback reason |
| 权限 | 普通用户保存 SQL 不会自动升级为 workspace 级业务口径 |
| 安全 | L3 参数化 SQL 必须经过参数校验和 dry-run |

建议测试矩阵：

1. **正例**：问题明确对应 `T09/T10/T13`，参数完整，应进入 L2/L3 并返回正确结果；
2. **近似负例**：问题与模板相似但口径不同，应降级或追问；
3. **参数缺失**：缺日期 / 渠道 / TOPN 等必填参数，应追问或降级；
4. **新增维度**：用户要求模板不支持的新分组，应降级为参考生成或追问；
5. **候选冲突**：两个模板相似度接近，应不自动进入 L3；
6. **权限边界**：普通用户保存 SQL 默认只能作为参考样例；
7. **版本边界**：deprecated 或过期模板不应命中；
8. **回归基线**：原先通过的模板、规则、追问、图表链路不回退。

## 9. 最终结论

本轮分析后的总体判断是：

1. **失败主因不是“系统完全不会命中模板”，而是“明确业务口径模板命中后仍然按普通参考样例处理”。**
2. **官方仓库在 general 路由、instructions 作为 prompt 约束、数据库方言、复杂 SQL 稳定性方面，确实存在和我们同方向的问题。**
3. **但我们本轮遇到的 runtime scope / deployHash / knowledge workspace 资产可见性问题，明显是当前分支特有，不是官方同一实现直接复现。**
4. **如果目标是把 `docs/业务需求` 中这批业务题稳定做准，最有效的方向不是把所有 `sql_pair` 硬执行，而是新增可选的业务模板锚定 / 参数化执行能力。**
5. **普通用户保存的 SQL 默认仍应作为参考样例；只有经过口径确认、参数槽识别和执行验证后，才升级为可执行模板。**
6. **前端需要做最小改动来表达保存意图和展示执行来源，但复杂模板管理器可以后置。**
