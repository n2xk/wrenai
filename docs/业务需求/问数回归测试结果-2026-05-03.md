## 结论

- 补槽链路整体可用：`tenant_plat_id`、`date_range`、`cohort_start_date/cohort_end_date`、`period_days`、`channel_id`、`metric_focus` 均能通过 UI 表单补入，并创建新的同 thread response。
- 多轮补槽可用但体验偏重：CR04 证明可以连续补槽，但建议后续把模板已知必填槽位一次性合并展示，减少用户多次提交。
- 外部依赖阻断行为符合当前策略：ROI 在补齐租户后明确阻断“投放金额”缺失，不编造外部数据。
- 仍需修复的功能问题：CR03 补齐 `date_range` 后进入 reference 模板 / core protection retry，但最终无 SQL 且 UI 长时间停在“正在生成 SQL”。这不应计为通过，需要后续代码修复或将该模板策略改为继续澄清 / 明确失败。

## 后续建议

1. 针对 CR03 增加自动化断言：补齐 `date_range` 后若路由为 `template_reference_sql` 且无 SQL，必须返回明确失败或追问，不允许 UI 持续 pending。
2. 优化补槽表单：尽量一次性展示模板所有缺失 required slots，特别是 cohort 类模板的日期范围 + period_days。
3. 若要支持 FULL ROI 用例，新增外部指标值补充表单或外部数据导入步骤；否则测试计划中继续把 ROI FULL 标为外部依赖阻断通过。
