# 问数回归测试计划（TiDB workspace / MCP Playwright）

## 1. 测试背景

本计划用于验证 `docs/业务需求` 目录下沉淀的业务需求，是否已经在系统内形成可用、可命中、可准确回答的问数能力，并能够在 **TiDB workspace** 中完成问答、追问和图表生成。

本轮测试要求：

- **全部通过 MCP Playwright 执行**
- **全部在 Chrome 中操作系统 UI**
- **全部在 TiDB workspace 中完成验证**
- 统一以 **TiDB SQL 能力链路** 作为主链路，不以 ES 作为当前执行链路

---

## 2. 测试目标

本轮目标分为两个层面：

### 2.1 覆盖性验证

验证 `docs/业务需求` 中的业务需求，是否已经被系统中的以下能力覆盖：

- 分析规则（instruction）
- SQL 模板（sql pair / sql template）
- suggested questions
- modeling AI assistant 生成的语义提示
- modeling AI assistant 生成的关联关系

### 2.2 准确性验证

验证系统对这些业务需求的以下输出是否准确：

- 问数结果
- 追问结果
- 图表生成结果
- 缺失外部数据源时的追问与兜底行为

## 6. 测试前置条件

在 UI 测试开始前，需确认以下前置条件成立：

1. 系统环境已重新初始化，可开始新一轮干净导入。
2. TiDB 测试库已准备好可供连接的数据。
3. 若 TiDB 测试库需要重建，应先完成：
   1. 执行 `python3 docs/业务需求/generate_local_tidb_schema.py`
   2. 导入 `docs/业务需求/local_tidb_schema.sql`
   3. 导入 `docs/业务需求/seed.sql`

## 7. 执行约束

本轮执行必须满足以下约束：

- 仅使用 **MCP Playwright** 完成 UI 操作与验证。
- 使用 **Chrome** 作为浏览器执行载体。
- 所有业务需求验证都必须在 **TiDB workspace** 内完成。
- 当前主链路仅验证 TiDB 能力，不将 legacy ES 作为独立 sql_pair 执行来源。
- 对于缺失外部数据源的问题，不允许系统直接编造投放金额、PV、UV、下载点击 UV 等值。

## 8. 测试执行流程

### Step 1：重新初始化系统环境

目标：确保本轮测试使用干净环境，避免旧 workspace、旧知识库、旧建模结果干扰。

检查点：

- 不复用历史测试 workspace
- 不复用历史知识库导入结果
- TiDB 数据源与测试数据状态明确可控

### Step 2：通过 MCP Playwright 拉起 Chrome 并创建 TiDB workspace

目标：在系统 UI 内创建一套本轮专用的 TiDB workspace。

操作路径：

- 系统设置
- 工作空间管理
- 新建 **TiDB workspace**

检查点：

- workspace 创建成功
- workspace 类型正确
- 后续建模、知识库、问答链路都在该 workspace 下进行

### Step 3：新增知识库

目标：为本轮规则与 SQL 模板导入提供承载容器。

检查点：

- 知识库创建成功
- 知识库归属到本轮 TiDB workspace

### Step 4：添加资产并连接 TiDB 数据源

目标：把当前回归数据接入 TiDB workspace。

检查点：

- TiDB 数据源可连接
- 连接参数正确
- 相关表可被系统识别

### Step 5：配置数据源并生成建议问题

目标：完成数据源可用性配置，并验证 suggested questions 可生成。

检查点：

- 数据源配置保存成功
- suggested questions 成功生成
- suggested questions 与业务主题具备相关性

### Step 6：用 modeling AI assistant 生成语义提示

目标：为后续问数补齐语义层提示信息。

检查点：

- 语义提示生成成功
- 生成结果与 TiDB 资产语义一致
- 能支撑业务术语问数

### Step 7：用 modeling AI assistant 生成关联关系

目标：补齐表之间或模型之间的关联关系，降低问数歧义。

检查点：

- 关联关系生成成功
- 关系方向、主外键或业务关联合理
- 能支持跨表统计问数

### Step 8：导入分析规则

导入来源：

- `docs/业务需求/knowledge-base/analysis-rules/`

目标：让系统先具备正确的业务口径和异常处理边界。

重点验证：

- `R13_缺失数据源处理` 已生效
- `R14_ES 数据使用限制` 已生效
- 首存、新客首存、TOPN、VIP、ROI 收入口径等规则可约束回答

### Step 9：导入 SQL 模板

导入来源：

- `docs/业务需求/knowledge-base/sql-templates/`

导入策略：

- 导入 11 个 `draft_sql` 模板作为主测试范围
- T05/T14/T15 不作为可执行 SQL pair 主链路导入；保留为缺失数据源场景验证项

目标：让系统具备可命中的 TiDB SQL 模板问数能力。

### Step 10：执行问数 / 追问 / 图表验证

使用 `docs/业务需求` 中的测试问题与标准答案进行验收，验证以下三类能力：

### 10.1 规则类问题验证

先验证系统是否被正确约束，例如：

- 首存定义
- 新客首存
- TOPN 口径
- VIP 分层口径
- 缺失数据源处理
- ES 数据使用限制

### 10.2 模板类问题验证

再验证系统是否能命中对应 SQL 模板并返回正确结果，例如：

- 综合日报
- 渠道与折扣映射
- cohort 收入
- TOPN 分层
- 日龄趋势
- 游戏类型分布
- 首存金额分桶

### 10.3 缺失外部数据源问题验证

验证系统在命中以下需求时是否先索取用户输入，而不是直接输出伪造结果：

- ROI
- 投放金额并表
- 流量指标并表
- 首存成本
- UV 下载率 / UV 注册率

---

## 9. 重点验收维度

### 9.1 覆盖性

通过标准：

- 分析规则可被系统命中并影响回答边界
- 11 个主模板问题都能在系统中形成有效问数链路
- 业务术语能在 TiDB workspace 中被理解

### 9.2 数值准确性

通过标准：

- 返回值与 `expected-results.md`、`csv/` 中的预期一致
- 关键总校验点一致
- 对照渠道 `990012` 不串入主渠道结果
- 失败充值、分母为 0 等边界场景处理正确

### 9.3 追问准确性

通过标准：

- 系统能继承上文筛选条件、日期范围、渠道范围、cohort 范围
- 追问不会丢失上下文
- 追问后的结果仍与预期答案一致

### 9.4 图表准确性

通过标准：

- 图表类型与问题意图匹配
- 图表维度、指标、排序与数据表一致
- 图表数值与问数结果一致
- 图表中的时间序列、分组、分层结果不发生错位

### 9.5 缺失数据源处理

通过标准：

- 系统明确指出缺失字段
- 系统索取的数据粒度正确
- 用户未补充外部数据前，系统不伪造 ROI / PV / UV / 下载点击 UV 等结果
- 用户补充后，系统可将外部值与内部 SQL 结果拼接输出
