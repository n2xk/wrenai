# 问数回归测试计划 v2（TiDB workspace / MCP Playwright）

## 1. 测试背景

本计划用于验证 `docs/业务需求` 目录下沉淀的业务需求，是否已经在系统内形成可用、可命中、可准确回答的问数能力，并能够在 **TiDB workspace** 中完成问答、追问、图表生成和缺失外部数据处理。

随着业务知识配置中心功能完成，当前知识库回归范围已经从：

```text
分析规则 + SQL 模板
```

扩展为：

```text
分析规则 + SQL 模板 + 业务词典 + 外部数据依赖 + 结构化知识字段
```

本轮测试要求：

- **全部通过 MCP Playwright 执行业务链路操作、导入、问数和结果采集**
- **全部在 Chrome 中操作系统 UI / 同源 API**
- **全部在 TiDB workspace 中完成验证**
- **完整回归必须先清理历史测试数据并重建本轮专用 TiDB workspace**
- **除用户明确声明“跳过 setup / 复用现有环境”外，不允许复用旧 workspace、旧知识库、旧问数线程或旧知识资产**
- 统一以 **TiDB SQL 能力链路** 作为唯一主链路，不以 ES 作为当前执行链路
- 数据库查询只作为可选旁证，不替代 MCP Playwright 验收结论

---

## 2. 测试目标

本轮目标分为四个层面。

### 2.1 覆盖性验证

验证 `docs/业务需求` 中的业务需求，是否已经被系统中的以下能力覆盖：

- 分析规则（instruction）
- SQL 模板（sql pair / sql template）
- 业务词典（business dictionary / business term）
- 外部数据依赖（external dependency）
- SQL 模板结构化字段（`business_signature`）
- 分析规则结构化字段（`related_business_terms` / `related_external_dependencies` / `runtime_usage`）
- suggested questions
- modeling AI assistant 生成的语义提示
- modeling AI assistant 生成的关联关系
- 参数泛化、语言泛化、数据边界和路由泛化用例

### 2.2 准确性验证

验证系统对这些业务需求的以下输出是否准确：

- 问数结果
- 追问结果
- 图表生成结果
- 业务词典同义词命中结果
- SQL 模板匹配与参数抽取结果
- 缺失外部数据源时的追问与兜底行为
- 普通 SQL、澄清、阻断、外部补充后继续计算等多路由行为

### 2.3 产品化验证

验证业务域特化语义是否由知识库配置驱动，而不是依赖代码硬编码：

- “首存 / 首充 / 首次存款”应由 `first_deposit` 业务词典定义驱动
- “续存 / 二存 / 六存 / 复存”应由 `retention_deposit` 业务词典定义驱动
- “TOP3 / 非TOP3”应由 `topn_segment` 业务词典定义驱动
- “ROI / 首存成本”应通过外部依赖 `ad_spend` 触发缺失数据追问
- “PV / UV / 下载点击UV”应通过外部依赖触发缺失数据追问

### 2.4 UI 体验观察

在 MCP Playwright 执行 UI 测试过程中，同步记录真实使用体验与产品可用性问题。重点观察：

- workspace 创建、知识库创建、数据源连接、建模、导入、问数、追问、图表生成等关键流程是否顺畅
- 表单字段、按钮文案、错误提示、加载状态、空状态是否清晰
- 批量导入、长任务等待、失败重试、成功反馈是否容易理解
- 业务词典、外部数据依赖、SQL 模板、分析规则四类资产的管理入口是否一致、易发现、易操作
- 问数结果、diagnostics、外部依赖阻断提示是否足够可解释

体验问题与优化建议需要单独沉淀到：

- `docs/业务需求/UI体验优化记录-YYYY-MM-DD.md`

## 6. 测试前置条件

在 UI 测试开始前，需确认以下前置条件成立。

### 6.1 完整回归环境清理硬性要求

完整回归默认必须从干净环境开始。除非用户明确要求“跳过 setup / 复用现有环境”，否则必须清理所有历史测试数据，并重新创建本轮专用 TiDB workspace。若由于审计、留痕或环境权限原因无法物理删除历史数据，必须使用全新隔离环境替代，并在最终报告中列出“未物理删除但已隔离”的对象。

必须清理的历史对象包括：

> 若无法物理删除，必须用新建对象隔离并记录原因。

| 类型 | 清理要求 |
| --- | --- |
| workspace | 不复用历史测试 workspace；完整回归必须新建 TiDB workspace |
| knowledge base / snapshot | 不复用旧知识库、旧 snapshot、旧 deploy hash |
| connector / data source | 不复用旧 connector；完整回归需重新连接 TiDB 数据源 |
| project / project_meta | 不复用旧项目、旧建模元数据 |
| model / relation / metric / view | 不复用旧 modeling 结果；需由本轮建模或导入产生 |
| instruction / sql_pair | 不复用旧分析规则、旧 SQL 模板记录；需由本轮导入产生 |
| business terms | 不复用旧业务词典记录；需由本轮导入产生 |
| external dependencies | 不复用旧外部数据依赖记录；需由本轮导入产生 |
| thread / thread_response / asking_task | 不复用旧问数线程、旧回答、旧任务结果 |
| suggested questions | 不复用旧建议问题；需由本轮数据源配置后重新生成 |

清理后必须记录验证证据：

- 旧 workspace / knowledge base 不再被本轮使用
- 本轮 workspace / knowledge base / connector / deploy hash 已明确记录
- 本轮知识库在导入前四类知识资产计数为 0，或能证明它是新建空知识库
- 本轮问数线程从本轮测试开始生成，不混入旧 thread / response

### 6.2 TiDB 测试库准备硬性要求

完整回归必须清空并重建 TiDB 测试库，或使用全新的隔离 TiDB 测试库。不能在带有未知历史数据的 TiDB 库上标记“全量干净回归通过”。

必须执行：

1. 执行 `python3 docs/业务需求/generate_local_tidb_schema.py`
2. 清空 / 重建目标 TiDB 测试 schema
3. 导入 `docs/业务需求/local_tidb_schema.sql`
4. 导入 `docs/业务需求/seed.sql`
5. 校验固定测试参数与总校验点：`tenant_plat_id=990001`、主渠道 `990011`、对照渠道 `990012`、主渠道总充值 `3248`、总有效投注 `7300`

若用户明确要求复用现有 TiDB 测试库，只能执行最小回归或非全量回归，并必须在最终报告中标记为“非全量干净回归”。

### 6.3 复用现有环境的例外条件

只有用户明确要求“跳过 setup / 复用现有环境 / 继续当前 workspace”时，才允许跳过完整清理。此时最终报告必须明确标记为：

```text
非全量干净回归：复用了现有 TiDB workspace / knowledge base
```

并记录：

- 复用的 `workspace_id`
- 复用的 `knowledge_base_id`
- 复用的 `kb_snapshot_id`
- 复用的 `deploy_hash`
- 复用前已有知识资产数量
- 复用前已有问数线程是否会影响本轮统计

### 6.4 基础服务要求

1. 本轮使用的数据源必须能支撑 TiDB workspace 中的语义建模、知识库导入与问答验证。
2. UI 与 AI service 均处于可用状态。
3. MCP Playwright 使用 Chrome 作为浏览器载体。

## 7. 执行约束

本轮执行必须满足以下约束：

- 仅使用 **MCP Playwright** 完成 UI 操作、同源 API 调用、问数、追问、图表生成和结果采集。
- 使用 **Chrome** 作为浏览器执行载体。
- 所有业务需求验证都必须在 **TiDB workspace** 内完成；不允许在 MySQL、PostgreSQL、SQLite、legacy ES workspace 或其他非 TiDB workspace 中替代执行。
- 当前主链路仅验证 TiDB 能力，不将 legacy ES 作为独立 sql_pair 执行来源。
- 完整回归必须先完成第 6.1 节的数据清理 / 隔离要求；未完成清理不得标记为“全量干净回归通过”。
- 若用户要求复用环境，最终报告必须按第 6.3 节标记为“非全量干净回归”。
- 对于缺失外部数据源的问题，不允许系统直接编造投放金额、PV、UV、下载点击 UV 等值。
- 对于阻塞模板，不以“执行 SQL 成功”为通过标准，而以“识别缺失依赖并正确索取外部数据”为通过标准。
- 对于业务词典，不以“页面存在”为唯一通过标准，必须通过问数 alias 命中验证其运行时生效。

## 8. 测试执行流程

### Step 1：重新初始化并清理系统环境

目标：确保完整回归使用干净环境，避免旧 workspace、旧知识库、旧建模结果、旧问数线程、旧知识资产干扰。

完整回归必须执行：

- 清理历史测试 workspace；若无法物理删除，必须新建隔离 TiDB workspace 并记录原因
- 清理历史 knowledge base / snapshot / deploy hash；若无法物理删除，必须新建隔离知识库并记录原因
- 清理历史 connector / project / modeling 结果；若无法物理删除，必须新建隔离对象并记录原因
- 清理历史 instruction / sql_pair / business term / external dependency；若无法物理删除，必须确认本轮知识库导入前为空
- 清理历史 thread / thread_response / asking_task；若无法物理删除，必须确认本轮统计不包含旧线程
- 清空并重建 TiDB 测试库 schema 与 seed，或使用全新隔离 TiDB 测试库

检查点：

- 本轮新建 TiDB workspace，且 workspace 类型为 TiDB
- 本轮新建知识库，导入前四类知识资产计数为 0，或有等价证据证明知识库为空
- 本轮新建或重新配置 TiDB connector
- 本轮问数线程从本轮测试开始生成
- 若用户明确要求复用环境，必须记录为“非全量干净回归”，并记录复用对象 ID 与复用前资产数量

### Step 2：通过 MCP Playwright 拉起 Chrome 并创建 TiDB workspace

目标：在系统 UI 内创建一套本轮专用的 TiDB workspace。

操作路径：

- 系统设置
- 工作空间管理
- 新建 **TiDB workspace**

检查点：

- workspace 创建成功
- workspace 类型明确为 TiDB
- 不创建、不切换、不复用任何非 TiDB workspace 做本轮验证
- 后续建模、知识库、问答链路都在该 TiDB workspace 下进行

### Step 3：新增知识库

目标：为本轮四类知识资产导入提供承载容器。

检查点：

- 知识库创建成功
- 知识库归属到本轮 TiDB workspace
- 知识库页面包含或可进入以下资产管理入口：
  - 分析规则
  - SQL 模板
  - 业务词典
  - 外部数据依赖

### Step 4：添加资产并连接 TiDB 数据源

目标：把当前回归数据接入 TiDB workspace。

检查点：

- TiDB 数据源可连接
- 连接参数正确
- 相关表可被系统识别

### Step 5：配置数据源并生成建议问题

目标：完成数据源可用性配置，并验证 suggested questions 可生成。

检查点：

- 数据源配置保存成功
- suggested questions 成功生成
- suggested questions 与业务主题具备相关性

### Step 6：用 modeling AI assistant 生成语义提示

目标：为后续问数补齐语义层提示信息。

检查点：

- 语义提示生成成功
- 生成结果与 TiDB 资产语义一致
- 能支撑业务术语问数

### Step 7：用 modeling AI assistant 生成关联关系

目标：补齐表之间或模型之间的关联关系，降低问数歧义。

## 9. 重点验收维度

### 9.1 覆盖性

通过标准：

- 分析规则可被系统命中并影响回答边界
- 11 个主模板问题都能在系统中形成有效问数链路
- 10 条业务词典都能导入，并至少通过关键 alias 用例证明运行时生效
- 4 条外部数据依赖都能导入，并至少通过 ROI / 流量指标用例证明运行时生效
- 结构化字段能进入导入记录、索引或 diagnostics

### 9.2 数值准确性

通过标准：

- 返回值与 `expected-results.md`、`csv/` 中的预期一致
- 关键总校验点一致
- 对照渠道 `990012` 不串入主渠道结果
- 失败充值、分母为 0 等边界场景处理正确

### 9.3 追问准确性

通过标准：

- 系统能继承上文筛选条件、日期范围、渠道范围、cohort 范围
- 追问不会丢失上下文
- 追问后的结果仍与预期答案一致

### 9.4 图表准确性

通过标准：

- 图表类型与问题意图匹配
- 图表维度、指标、排序与数据表一致
- 图表数值与问数结果一致
- 图表中的时间序列、分组、分层结果不发生错位

### 9.5 缺失数据源处理

通过标准：

- 系统明确指出缺失字段
- 系统索取的数据粒度正确
- 用户未补充外部数据前，系统不伪造 ROI / PV / UV / 下载点击 UV 等结果
- 用户补充后，系统可将外部值与内部 SQL 结果拼接输出（若当前产品已支持外部值回填）

### 9.6 diagnostics 可观测性

通过标准：

- 主模板问数可记录 `templateDecision`
- 可观测到 `sourceType = business_import`
- 可观测到 `templateLevel = L2`
- 可观测到 `templateMode = anchored_template`
- 外部依赖阻断回答可观测到缺失依赖集合或等价原因

### 9.7 问数结果产物化

通过标准：

- 图表型结果可固定到数据看板。
- 表格型结果可保存为数据表资产。
- 图表固定与数据表保存入口语义清晰，不互相混用。
- 数据看板 / 数据表列表 item 名称与详情页标题一致。
- 数据看板 / 数据表 item 支持重命名与删除。
- 保存后的数据表可以查看 SQL、预览数据，并进入 AI 操作链路。

### 9.8 导出能力

通过标准：

- 问数结果数据预览支持导出 CSV。
- 问数结果数据预览支持导出 Excel。
- 导出文件的列名、行数、数值与当前预览一致。
- 中文、日期、数字字段格式可读，不乱码。

### 9.9 反馈与治理闭环

通过标准：

- 问数结果正反馈、负反馈可持久化。
- 负反馈可记录原因与补充说明。
- 反馈管理页面可在用户可见 workspace 范围内查看反馈。
- 管理页面支持 workspace 筛选。
- 权限不足时不能查看反馈治理数据。

### 9.10 数据表 AI 操作

通过标准：

- 已保存数据表支持 Filter / Cleaning / Grouping / Enrichment smoke 验证。
- 结构化 Filter 能稳定生成确定性 SQL。
- AI rewrite 必须 dry-run 成功后保存。
- 失败时不保存新版本。
- History 能追踪版本、操作类型、生成模式和 SQL 变化。

### 9.11 业务需求泛化能力

通过标准：

- PX 参数泛化用例能证明时间、渠道、TOPN、周期、汇总粒度和输出形态变化后仍能准确问数。
- LING 语言泛化用例能证明业务词典 alias 与口语化表达参与召回，但模糊问题仍会澄清。
- EDGE 数据边界用例能证明失败订单、无数据、空分母、空桶、TOPN 不足和金额桶边界处理正确。
- ROUTE 路由泛化用例能证明系统能在业务模板、普通 SQL、澄清、外部依赖阻断和补充后继续计算之间正确切换。
- diagnostics 能记录模板决策、SQL 来源、降级原因、阻断原因或澄清原因，便于回归定位。
