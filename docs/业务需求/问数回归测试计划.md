# 问数回归测试计划 v2（TiDB workspace / MCP Playwright）

## 1. 测试背景

本计划用于验证 `docs/业务需求` 目录下沉淀的业务需求，是否已经在系统内形成可用、可命中、可准确回答的问数能力，并能够在 **TiDB workspace** 中完成问答、追问、图表生成和缺失外部数据处理。

随着业务知识配置中心功能完成，当前知识库回归范围已经从：

```text
分析规则 + SQL 模板
```

扩展为：

```text
分析规则 + SQL 模板 + 业务词典 + 外部数据依赖 + 问数策略 + 结构化知识字段
```

本轮测试要求：

- **全部通过 MCP Playwright 执行业务链路操作、导入、问数和结果采集**
- **全部在 Chrome 中操作系统 UI / 同源 API**
- **全部在 TiDB workspace 中完成验证**
- **完整回归必须先清理历史测试数据并重建本轮专用 TiDB workspace**
- **除用户明确声明“跳过 setup / 复用现有环境”外，不允许复用旧 workspace、旧知识库、旧问数线程或旧知识资产**
- 统一以 **TiDB SQL 能力链路** 作为唯一主链路，不以 ES 作为当前执行链路
- 数据库查询只作为可选旁证，不替代 MCP Playwright 验收结论

### 1.1 LLM Provider 限流控制与重试规则

问数、图表生成、推荐问题、语义规划等链路会触发 LLM provider 请求。为避免把 OpenRouter / 上游 provider 的临时 `429 rate limit` 误判为业务逻辑缺陷，完整回归必须遵守以下执行节奏和判定规则。

#### 执行节奏

- **LLM 相关用例串行执行**：同一时间只允许 1 个自然语言 ask / 图表生成 / 推荐问题 / 语义生成任务在跑。
- **等待当前任务稳定后再进入下一条**：必须等 UI 中 response 状态完成，且 SQL、答案、图表或阻断提示已经稳定展示后，再提交下一条。
- **普通问数间隔（默认加速节奏）**：每条自然语言 ask 完成后，等待 `1~3 秒` 再发下一条；本轮主要观察 provider `429`，其他 UI / SQL / 路由用例不需要额外放慢。
- **普通问数间隔（保守节奏）**：若已经出现 provider `429`、socket hang up 或上游延迟明显升高，恢复到 `5~8 秒`。
- **图表生成间隔**：每次图表生成完成后，等待 `1~3 秒` 再继续下一条 LLM 用例；若出现 429，恢复到 `8~12 秒`。
- **推荐问题 / 语义生成间隔**：每次推荐问题、语义描述、关联关系或类似 LLM 辅助生成完成后，等待 `1~3 秒`；若出现 429，恢复到至少 `10 秒`。
- **批次暂停（默认加速节奏）**：每连续执行 `8~10` 个 LLM 相关用例，暂停 `5~10 秒` 再继续。
- **批次暂停（保守节奏）**：若已经出现 provider 429，恢复为每连续执行 `8~10` 个 LLM 相关用例暂停 `60~90 秒`。
- **禁止并行压测式操作**：不得同时打开多个浏览器标签批量提交问数；不得用脚本无间隔批量触发 ask、chart、recommendation。
- **自适应退避**：一旦任意请求或任务结果出现 OpenRouter / provider `429`，立即暂停 `90~180 秒`，重试当前用例；重试后再继续时至少使用保守节奏跑满下一批次。

#### 429 重试规则

- 首次出现 OpenRouter / provider `429` 时，不直接判定业务用例失败，先标记为 `provider_transient_429`。
- 触发 `429` 后立即暂停 `90~180 秒`，再重试当前用例。
- 同一用例最多重试 `2` 次。
- 重试后通过：用例结果记为 **通过（带 provider 限流风险）**。
- 重试后仍失败，且错误仍为 provider `429`：用例结果记为 **环境 / provider 阻塞**，不得计为 SQL 生成、业务口径或 UI 功能失败。
- 若重试后不再是 `429`，而是 SQL 错误、路由错误、数据错误或 UI 操作错误，则按实际错误类型记录失败。

#### 结果记录要求

最终测试结果文档必须记录每个 `429` 事件：

| 字段 | 说明 |
| --- | --- |
| 用例编号 | 例如 T10 / OQ03 / ROUTE04 |
| 触发动作 | ask / chart / recommendation / semantic generation |
| provider | OpenRouter 返回的 provider 名称，如 DeepInfra / SiliconFlow / Novita |
| 错误摘要 | 保留简短错误文本，例如 temporarily rate-limited upstream |
| 等待时长 | 本次重试前暂停的秒数 |
| 重试次数 | 0 / 1 / 2 |
| 最终判定 | 通过（带 provider 限流风险）/ 环境阻塞 / 产品缺陷 |

## 2. 测试目标

本轮目标分为四个层面。

### 2.1 覆盖性验证

验证 `docs/业务需求` 中的业务需求，是否已经被系统中的以下能力覆盖：

- 分析规则（instruction）
- SQL 模板（sql pair / sql template）
- 业务词典（business dictionary / business term）
- 外部数据依赖（external dependency）
- 问数策略（ask policy rule）
- SQL 模板结构化字段（`business_signature`）
- 分析规则结构化字段（`related_business_terms` / `related_external_dependencies` / `runtime_usage`）
- suggested questions
- modeling AI assistant 生成的语义提示
- modeling AI assistant 生成的关联关系
- 参数泛化、语言泛化、数据边界和路由泛化用例

### 2.2 准确性验证

验证系统对这些业务需求的以下输出是否准确：

- 问数结果
- 追问结果
- 图表生成结果
- 业务词典同义词命中结果
- SQL 模板匹配与参数抽取结果
- 缺失外部数据源时的追问与兜底行为
- 普通 SQL、澄清、阻断、外部补充后继续计算等多路由行为

### 2.3 产品化验证

验证业务域特化语义是否由知识库配置驱动，而不是依赖代码硬编码：

- “首存 / 首充 / 首次存款”应由 `first_deposit` 业务词典定义驱动
- “续存 / 二存 / 六存 / 复存”应由 `retention_deposit` 业务词典定义驱动
- “TOP3 / 非TOP3”应由 `topn_segment` 业务词典定义驱动
- “ROI / 首存成本”应通过外部依赖 `ad_spend` 触发缺失数据追问
- “PV / UV / 下载点击UV”应通过外部依赖触发缺失数据追问
- “缺租户平台 / 玩家明细误套渠道日报 / 非 cohort 问题误套首存模板”等路由安全门槛应由知识库 **问数策略** 驱动，而不是依赖测试脚本或代码硬编码

### 2.4 UI 体验观察

在 MCP Playwright 执行 UI 测试过程中，同步记录真实使用体验与产品可用性问题。重点观察：

- workspace 创建、知识库创建、数据源连接、建模、导入、问数、追问、图表生成等关键流程是否顺畅
- 表单字段、按钮文案、错误提示、加载状态、空状态是否清晰
- 批量导入、长任务等待、失败重试、成功反馈是否容易理解
- 业务词典、外部数据依赖、SQL 模板、分析规则、问数策略五类资产的管理入口是否一致、易发现、易操作
- 问数结果、diagnostics、外部依赖阻断提示是否足够可解释

体验问题与优化建议需要单独沉淀到：

- `docs/业务需求/UI体验优化记录-YYYY-MM-DD.md`

## 6. 测试前置条件

在 UI 测试开始前，需确认以下前置条件成立。

### 6.1 完整回归环境清理硬性要求

> 若无法物理删除，必须用新建对象隔离并记录原因。

| 类型 | 清理要求 |
| --- | --- |
| workspace | 不复用历史测试 workspace；完整回归必须新建 TiDB workspace |
| knowledge base / snapshot | 不复用旧知识库、旧 snapshot、旧 deploy hash |
| connector / data source | 不复用旧 connector；完整回归需重新连接 TiDB 数据源 |
| project / project_meta | 不复用旧项目、旧建模元数据 |

- 追问不会丢失上下文
- 追问后的结果仍与预期答案一致

### 9.3.1 补充信息 / 澄清续跑准确性

通过标准：

- 原始缺槽问题必须展示补充信息入口或等价澄清提示，且不直接进入不可靠 SQL 生成。

- 若补充后仍失败，必须区分是“补槽值未进入 AI service / 前端显示未刷新 / SQL 生成失败 / 数据为空但 SQL 成功 / 外部依赖仍缺失”，并写入测试结果。
- 若用例本身只验证缺槽阻断，不要求最终 SQL，则最终报告必须明确写成“澄清阻断通过，不计入最终问数通过”。

### 9.4 图表准确性

通过标准：

- 图表类型与问题意图匹配
- 图表维度、指标、排序与数据表一致
- 图表数值与问数结果一致
- 图表中的时间序列、分组、分层结果不发生错位

### 9.5 缺失数据源处理

通过标准：

- 系统明确指出缺失字段
- 系统索取的数据粒度正确
- 用户未补充外部数据前，系统不伪造 ROI / PV / UV / 下载点击 UV 等结果
- 用户补充后，系统可将外部值与内部 SQL 结果拼接输出（若当前产品已支持外部值回填）

- 表格型结果可保存为数据表资产。

### 9.8 导出能力

- 问数结果数据预览支持导出 CSV。
- 问数结果数据预览支持导出 Excel。
- 导出文件的列名、行数、数值与当前预览一致。

### 9.9 反馈与治理闭环

通过标准：

- 负反馈可记录原因与补充说明。
- 反馈管理页面可在用户可见 workspace 范围内查看反馈。
- 管理页面支持 workspace 筛选。
- 权限不足时不能查看反馈治理数据。

### 9.10 数据表 AI 操作

通过标准：

- 已保存数据表支持 Filter / Cleaning / Grouping / Enrichment smoke 验证。
- 结构化 Filter 能稳定生成确定性 SQL。
- AI rewrite 必须 dry-run 成功后保存。
- 失败时不保存新版本。
- History 能追踪版本、操作类型、生成模式和 SQL 变化。

### 9.11 业务需求泛化能力

通过标准：
