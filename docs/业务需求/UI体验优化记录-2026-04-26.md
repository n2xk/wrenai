| ID | 严重度 | 场景 | 现象 | 影响 | 建议 | 证据 |
| --- | --- | --- | --- | --- | --- | --- |
| UX-001 | P1 | 数据源配置后生成建议问题 | 新 TiDB connector 连接成功并识别 25 张表后，`suggestedQuestions.questions=[]`，没有产生可直接点击的引导问题。 | 新用户在建完数据源后缺少“下一步问什么”的启动引导；也不利于回归自动化验证数据源配置质量。 | 数据源配置完成后，应基于表名、模型描述、业务词典和 SQL 模板生成 3~5 个可执行建议问题；若生成失败，显示明确原因与重试入口。 | setup/import 脚本采集到 suggested questions 为空。 |
| UX-002 | P1 | Modeling AI assistant 自动化状态判断 | 脚本最初因 runner 环境不支持某些等待写法误报 `setTimeout is not defined` / 失败，但后续查询任务状态发现语义描述与关联关系任务实际均已 `FINISHED`。 | 用户或测试脚本容易把“前端等待实现问题”误判为“AI assistant 失败”。 | 前端应暴露稳定的任务状态轮询 API/状态标签；测试侧应以任务状态和保存结果为准，不以浏览器等待异常作为最终失败依据。 | semantics task `3d5e9daf-3967-4521-a8cb-452612986140`、relationship task `9597578e-00ff-477a-b4eb-3a6c73e199cf` 均为 `FINISHED`。 |
| UX-003 | P2 | 本地 AI service 启动 | 使用 `python -m src.__main__` 在 development reload 模式下，非交互后台可能快速退出；未显式设置 `PG_CONN_STR` 时问数报 `None of the following authentication environment variables are set: ('PG_CONN_STR',)`。 | 本地回归容易因为启动方式和环境变量不一致产生非业务失败。 | 在项目 runbook 或 `just start` 中固化本地回归启动方式；启动时预检 `PG_CONN_STR` 并给出明确错误提示。 | 本轮稳定启动命令：`PG_CONN_STR=postgresql://postgres:postgres@127.0.0.1:9432/wrenai poetry run uvicorn src.__main__:app --host 127.0.0.1 --port 5555 ...`。 |
| UX-004 | P2 | 知识资产导入与回写 | SQL 模板、分析规则、业务词典、外部数据依赖可落库，但批量导入/结构化字段校验仍依赖自动化脚本和文档约定。 | 产品化配置时，业务人员难以确认每个字段是否被解析、是否生效、是否需要补充。 | 在知识库 UI 增加批量导入结果页：按资产类型显示成功/失败数、字段校验错误、是否参与 retrieval / anchored template / blocked_missing_source。 | 本轮导入计数：分析规则 14、业务词典 10、外部依赖 4、SQL pair 12。 |
| UX-005 | P2 | 追问诊断可解释性 | F_T04 修复前，追问在低 margin 时降级为 `trusted_reference` 并触发 SQL 解析错误；UI 上很难直接看出是“模板上下文连续性识别失败”。 | 运维/业务验收难定位模板是否被当作硬锚定、参考样本或完全生成。 | 问数结果诊断中展示 template decision 的关键字段：mode、templateId、fallbackReason、sqlSource、parameters、history continuity 命中情况。 | 修复后 F_T04 显示 `mode=anchored_template`、`fallbackReason=null`、`parameters.cohort_start_date=2026-04-02`。 |
| UX-006 | P2 | 图表生成诊断 | 多个图表最终 `FINISHED`，但 AI service 日志出现 Vega-Lite schema 校验错误（`'null' is not one of ['zero', 'center', 'normalize']`），随后走 deterministic chart fallback。 | 最终图表可用，但 UI 不容易看出经历过 fallback；如果 fallback 质量下降，业务用户难定位原因。 | 图表生成结果中展示 `fallbackUsed`、原始 schema 校验错误和最终 fallback 类型；也可在 chart diagnostics 中标记“自动修复/兜底”。 | T09/T12 等图表日志出现 Vega schema validation error 后 fallback，Playwright 结果仍为 `chart.status=FINISHED`。 |
| UX-007 | P3 | 缺失外部数据提示 | T05/T14/T15 能正确阻断并索取外部指标，但回答中的“需要按什么粒度补充”还偏通用。 | 用户可以知道缺什么，但不一定知道要按哪些键上传或录入外部数据。 | 外部依赖提示可进一步结构化输出：指标名、必需维度、时间粒度、示例表头、可上传入口。 | T15 回答提示缺 `访问UV、下载点击UV、访问PV`，并要求按每个日期、每个渠道补充。 |
