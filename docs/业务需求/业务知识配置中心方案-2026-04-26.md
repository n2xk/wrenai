# 业务知识配置中心方案（业务词典 / 外部数据依赖 / 结构化知识字段）

## 1. 背景

当前 `docs/业务需求/knowledge-base/` 已经沉淀了两类知识资产：

1. **分析规则**：`analysis-rules/`，用于描述业务口径、公式、缺失数据处理规则。
2. **SQL 模板**：`sql-templates/`，用于承载高置信 SQL 骨架，并通过 `question_variants` 展开为 `sql_pair`。

在 TiDB workspace 回归中，这两类资产已经能支撑当前业务问数测试通过。但目前仍有一类产品化风险：

- 一部分业务语义仍写在 runtime 代码里，例如：
  - `首存 / 首充 / 首次存款`
  - `续存 / 复存 / 2~6存 / 二存~六存`
  - `TOPN / TOP3 / 非TOP3`
  - `ROI / 首存成本 / 投放金额`
  - `PV / UV / 下载点击UV`
- 这些词汇不是测试硬编码，也不是固定答案，但属于当前业务域特化。
- 如果要做成通用产品能力，这些业务语义不应该继续写在代码里，而应该由知识库配置驱动。

因此，需要在现有知识库基础上补充两类资产：

1. **业务词典**：定义用户说法、同义词、业务概念、关联规则和模板。
2. **外部数据依赖**：定义当前系统没有直接数据源的指标，以及缺失时如何追问用户。

最终知识库应从：

```text
分析规则 + SQL 模板
```

扩展为：

```text
分析规则 + SQL 模板 + 业务词典 + 外部数据依赖
```

---

## 2. 目标与非目标

### 2.1 目标

1. 将业务词汇、业务特征、外部依赖从代码中迁移到知识库配置。
2. 让 runtime 只保留通用能力：匹配、参数抽取、模板执行、校验、追问、diagnostics。
3. 支持 workspace / knowledge base 级别的业务语义配置。
4. 支持 SQL 模板通过结构化 `business_signature` 描述自己的业务适用范围。
5. 支持外部数据依赖在问数时自动触发“向用户索取数据”，而不是编造结果。
6. 继续兼容当前 `analysis-rules/` 与 `sql-templates/` 的导入与回归流程。

### 2.2 非目标

1. 不在第一阶段实现外部数据源接入或上传数据入库。
2. 不把用户随手保存的 SQL 自动升级为高置信业务模板。
3. 不替代 MDL / semantic modeling；业务词典是补充业务语言和口径，不是表结构定义。
4. 不在 MVP 阶段做跨 workspace 的全局业务词库市场。

## 5. 新增资产一：业务词典

### 5.1 作用

业务词典解决的是：

> 用户怎么说，系统怎么理解。

例如：

| 业务概念 | 用户说法 / 同义词 | 系统概念 |
| --- | --- | --- |
| 首存 | 首存、首充、首次存款、第一次充值 | `first_deposit` |
| 续存 | 续存、复存、二存、三存、2存、2~6存 | `retention_deposit` |
| TOPN | TOP3、TOP5、前3、非TOP3 | `topn_segment` |
| 投充比 | 投充比、投注充值比 | `bet_deposit_ratio` |
| 杀率 | 杀率、平台杀率、输赢率 | `kill_rate` |

### 5.2 文件示例

建议文件：

```text
docs/业务需求/knowledge-base/business-dictionary/first_deposit.md
```

示例 front matter：

```yaml
---
kb_asset_type: business_term
import_target: business_dictionary
import_format_version: v1
id: first_deposit
name: 首存
category: metric
status: active
priority: high
aliases:
  - 首存
  - 首充
  - 首次存款
  - 第一次充值
  - first deposit
definition: 成功存款且 times = 1
canonical_expression: dwd_order_deposit.status = 2 AND dwd_order_deposit.times = 1
source_tables:
  - dwd_order_deposit
source_fields:
  - dwd_order_deposit.status
  - dwd_order_deposit.times
  - dwd_order_deposit.callback_time
related_rules:
  - R02
related_templates:
  - T03
  - T04
  - T08
  - T10
  - T13
features:
  - cohort
  - first_deposit
conflict_terms: []
source_documents:
  - 第一期数据报表需求V1.xlsx
---
```

### 5.3 字段说明

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `kb_asset_type` | 是 | 固定为 `business_term` |
| `import_target` | 是 | 固定为 `business_dictionary` |
| `id` | 是 | 稳定机器 ID，例如 `first_deposit` |
| `name` | 是 | 展示名称，例如 `首存` |
| `category` | 是 | `metric` / `dimension` / `segment` / `formula` / `event` / `business_process` |
| `aliases` | 是 | 用户可能使用的同义词 |
| `definition` | 是 | 人类可读定义 |
| `canonical_expression` | 否 | 可机器读取的表达式或 SQL 条件 |
| `source_tables` | 否 | 相关表 |
| `source_fields` | 否 | 相关字段 |
| `related_rules` | 否 | 关联分析规则 |
| `related_templates` | 否 | 关联 SQL 模板 |
| `features` | 否 | 模板匹配特征 |
| `conflict_terms` | 否 | 易混淆概念 |

## 6. 新增资产二：外部数据依赖

### 6.1 作用

外部数据依赖解决的是：

> 哪些指标当前系统没有数据源，遇到时应该怎么处理。

例如：

| 外部指标 | 用户说法 / 同义词 | 缺失处理 |
| --- | --- | --- |
| 投放金额 | 投放金额、买量成本、广告消耗、推广成本 | 向用户索取 |
| 访问PV | PV、访问PV、页面访问量 | 向用户索取 |
| 访问UV | UV、访问UV、访问人数 | 向用户索取 |
| 下载点击UV | 下载点击UV、下载UV、下载点击人数 | 向用户索取 |

### 6.2 文件示例

建议文件：

```text
docs/业务需求/knowledge-base/external-dependencies/ad_spend.md
```

示例 front matter：

```yaml
---
kb_asset_type: external_dependency
import_target: external_dependency
import_format_version: v1
id: ad_spend
name: 投放金额
status: active
source_status: missing
missing_behavior: ask_user
aliases:
  - 投放金额
  - 买量成本
  - 广告消耗
  - 推广成本
required_grain:
  - biz_date + channel_id
  - date_range + channel_id
  - cohort_period
required_by_terms:
  - roi
  - first_deposit_cost
required_by_templates:
  - T05
  - T14
related_rules:
  - R13
ask_user_prompt: 请提供当前问题对应统计粒度的投放金额。
validation:
  value_type: number
  min: 0
source_documents:
  - 第一期数据报表需求V1.xlsx
---
```

### 6.3 字段说明

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `kb_asset_type` | 是 | 固定为 `external_dependency` |
| `import_target` | 是 | 固定为 `external_dependency` |
| `id` | 是 | 稳定机器 ID，例如 `ad_spend` |
| `name` | 是 | 展示名称，例如 `投放金额` |
| `source_status` | 是 | `available` / `missing` / `partial` / `manual_input` |
| `missing_behavior` | 是 | `ask_user` / `block_answer` / `allow_partial_answer` |
| `aliases` | 是 | 用户可能使用的同义词 |
| `required_grain` | 否 | 用户补充数据时需要的粒度 |
| `required_by_terms` | 否 | 哪些业务概念依赖它 |
| `required_by_templates` | 否 | 哪些 SQL 模板依赖它 |
| `related_rules` | 否 | 关联分析规则，例如 `R13` |
| `ask_user_prompt` | 否 | 缺失时提示用户的话术 |
| `validation` | 否 | 用户补充值的校验规则 |

## 7. 现有 SQL 模板需要补的结构化字段

当前 SQL 模板已有：

- `parameters`
- `question_variants`
- `source_tables`
- `result_grain`
- `runtime_sync`

为了让 runtime 不再硬编码业务词，需要新增 `business_signature`。

### 7.1 SQL 模板新增字段示例

以 `T08 首存 cohort 续存` 为例：

```yaml
business_signature:
  concepts:
    - first_deposit
    - retention_deposit
  features:
    - cohort
    - retention
  metrics:
    - second_deposit_user_count
    - second_deposit_rate
    - second_deposit_avg_amount
  dimensions:
    - first_deposit_date
    - channel_id
  parameter_slots:
    - tenant_plat_id
    - channel_id
    - cohort_start_date
    - cohort_end_date
  external_dependencies: []
  positive_cues:
    - 首存cohort
    - 2~6存
    - 二存
    - 三存
    - 续存
  negative_cues:
    - ROI
    - 投放金额
  expected_grain: first_deposit_date + channel_id
```

以 `T15 流量指标并表` 为例：

```yaml
business_signature:
  concepts:
    - traffic_metrics
  features:
    - daily_summary
    - external_metrics
  metrics:
    - access_pv
    - access_uv
    - download_click_uv
    - uv_download_rate
    - uv_register_rate
  dimensions:
    - biz_date
    - channel_id
  parameter_slots:
    - tenant_plat_id
    - channel_id
    - start_date
    - end_date
  external_dependencies:
    - access_pv
    - access_uv
    - download_click_uv
  positive_cues:
    - PV
    - UV
    - 下载点击UV
    - UV下载率
    - UV注册率
  negative_cues: []
  expected_grain: biz_date + channel_id
```

### 7.2 字段说明

| 字段 | 说明 |
| --- | --- |
| `concepts` | 模板涉及的业务概念，引用业务词典 ID |
| `features` | 模板匹配特征，例如 `cohort` / `retention` / `topn_segment` |
| `metrics` | 输出或核心计算指标 |
| `dimensions` | 结果维度 |
| `parameter_slots` | 参数槽，与 SQL 中占位符对应 |
| `external_dependencies` | 模板依赖的外部指标 ID |
| `positive_cues` | 增强匹配的正向提示词 |
| `negative_cues` | 排除或降权的提示词 |
| `expected_grain` | 期望输出粒度 |

## 10. Runtime 使用流程

产品化后的问数流程建议如下：

1. **加载当前 scope**
   - workspace
   - knowledge base
   - snapshot
   - deploy hash
2. **读取知识配置**
   - 分析规则
   - SQL 模板
   - 业务词典
   - 外部数据依赖
3. **业务概念抽取**
   - 根据业务词典 `aliases` 识别用户问题中的概念
   - 生成 `detected_concepts`
4. **外部依赖检测**
   - 根据 `detected_concepts` 和模板 `external_dependencies` 判断是否缺数据
   - 如果缺失且 `missing_behavior = ask_user`，进入追问
5. **SQL 模板候选召回**
   - 继续使用语义检索召回 SQL pair
   - 用 `business_signature` 对候选重新打分
6. **模板 gating**
   - 判断来源、审批态、参数完整度、业务特征匹配、候选 margin、dry-run
7. **执行 / 降级 / 追问**
   - 通过：执行 anchored template
   - 缺参数：追问参数
   - 缺外部数据：追问外部数据
   - 低置信：降级到 reference / LLM 生成
8. **diagnostics 输出**
   - 命中的 concepts
   - 命中的 external dependencies
   - template decision
   - fallback reason

## 14. 验收标准

### 14.1 功能验收

1. 可以在 UI 中创建 / 编辑 / 删除业务词典。
2. 可以在 UI 中创建 / 编辑 / 删除外部数据依赖。
3. SQL 模板可以关联业务概念和外部依赖。
4. 分析规则可以关联业务概念和外部依赖。
5. 问数时 diagnostics 能看到：
   - detected concepts
   - matched aliases
   - required external dependencies
   - matched template business signature
6. 遇到外部依赖缺失时，系统按配置向用户索取数据。
7. 不修改代码，仅修改业务词典同义词，即可影响模板匹配。

### 14.2 回归验收

当前 TiDB workspace 回归必须继续通过：

- 主模板问数：`11/11`
- 追问：`2/2`
- 缺失外部数据源兜底：`3/3`
- 合计：`16/16`

### 14.3 产品化验收

1. 新增另一个业务域时，不需要改 core runtime 代码。
2. 业务词典禁用某个 alias 后，相关问法的匹配结果会变化。
3. 外部依赖从 `missing` 改为 `available` 后，不再触发“向用户索取”。
4. SQL 模板冲突时，`business_signature` 能参与候选区分。

## 16. 风险与处理

| 风险 | 说明 | 处理 |
| --- | --- | --- |
| 同义词过宽导致误命中 | 例如“首充”和普通充值混淆 | 增加 negative_cues / conflict_terms |
| 业务词典和 SQL 模板漂移 | 模板改了但 concepts 未更新 | 保存模板时校验关联字段 |
| 外部依赖过度阻塞 | 用户只是问内部指标却被要求补外部数据 | 依赖必须由 required_by_terms/templates 精确触发 |
| 多 workspace 语义不同 | 不同客户同词不同义 | 所有资产默认 workspace / knowledge base scoped |
| 配置过复杂 | 用户维护成本高 | MVP 只做 name/aliases/definition/relations |

## 17. 推荐实施顺序

1. **先补 docs schema 与样例**
   - 新增 `business-dictionary/`
   - 新增 `external-dependencies/`
   - 给 SQL 模板补 `business_signature`
   - 给分析规则补关联字段
2. **再补后端存储与 API**
   - 业务词典 CRUD
   - 外部依赖 CRUD
   - SQL 模板 metadata 扩展
3. **再补前端页面**
   - 业务词典 tab
   - 外部数据依赖 tab
   - SQL 模板关联字段
4. **再改 runtime**
   - 先读取配置
   - 旧硬编码作为 fallback
   - TiDB 回归通过后逐步移除硬编码
5. **最后清理特化代码**
   - 删除核心 runtime 中的业务词硬编码
   - 所有业务语义由知识库驱动

## 19. 一句话结论

当前 `analysis-rules + sql-templates` 已经足够支撑这轮 TiDB 回归；但要做成可复用产品功能，还需要补 `business-dictionary + external-dependencies` 两类知识资产，并给现有规则/模板补结构化关联字段。最终目标是让 runtime 不再知道“首存/续存/TOPN/PV/UV”这些具体业务词，而是从当前 workspace 的知识库配置中读取和执行。
