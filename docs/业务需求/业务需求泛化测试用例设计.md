# 业务需求泛化测试用例设计

## 1. 文档目的

本文件用于补充 `docs/业务需求/问数回归测试计划.md`、`docs/业务需求/第一期需求全覆盖补充用例.md` 和 `docs/业务需求/普通问数补充用例.md` 之外的泛化测试设计。

现有 T01~T15 更偏“核心业务模板能否命中并返回正确结果”，FC01~FC12 更偏“第一期 Excel 需求的全字段 / 全维度 / 全周期补齐”，OQ01~OQ10 更偏“普通 text-to-SQL 不被业务模板过度锚定”。本文件关注另一类问题：

> 同一个业务需求在不同参数、不同说法、不同边界数据、不同路由路径下，系统是否仍然稳定、准确、可解释。

因此，本文件不是替代 T / FC / OQ，而是作为完整业务回归的“泛化覆盖层”。

---

## 2. 用例分层

| 用例集 | 名称 | 主要目标 | 与现有用例关系 |
| --- | --- | --- | --- |
| PX | 参数泛化 | 验证时间、渠道、TOPN、周期、维度、输出形态变化后仍能正确问数 | 扩展 T / FC |
| LING | 语言泛化 | 验证业务词典 alias、同义表达、模糊表达、口语化表达 | 扩展业务词典与 OQ |
| EDGE | 数据边界 | 验证失败订单、无数据、分母为 0、空桶、TOPN 不足等边界 | 扩展数值准确性 |
| ROUTE | 路由泛化 | 验证模板命中、普通 SQL、澄清、外部依赖阻断、用户补充后继续计算等路径 | 扩展 diagnostics 与执行路径验证 |

## 3. 参数泛化用例（PX）

| ID | 覆盖点 | 推荐问题 | 路由预期 | 关键验收点 |
| --- | --- | --- | --- | --- |
| PX01 | 单日综合日报 | 统计租户平台990001下渠道990011在2026-04-02这一天的综合日报基础指标 | 可命中 T01 或等价模板 | 只返回 04-02；充值金额为 2080；不混入其他日期 |
| PX02 | 多渠道综合日报 | 对比租户平台990001下渠道990011和990012在2026-04-01到2026-04-07的综合日报基础指标 | 可命中 T01 扩展或普通 SQL | 结果按渠道分组；990012 不串入 990011 |
| PX03 | 渠道商 / 全渠道汇总 | 按渠道商汇总租户平台990001在2026-04-01到2026-04-07的存款人数、存款金额、投注人数和有效投注 | FC02 / 聚合模板 | 金额相加；人数按口径去重；汇总比例需重算 |
| PX04 | ROI 自定义周期 | 统计租户平台990001下渠道990011在2026-04-01到2026-04-03首存cohort的D1、D3、D7、D15、D30累计渠道收入 | T04/T05 长周期扩展 | 返回请求的周期集合；D7 后无新增行为时累计值保持 |
| PX05 | TOP5 / 非TOP5 | 统计租户平台990001下渠道990011在2026-04-15到2026-04-26的TOP5和非TOP5用户存款、有效投注、输赢、投充比和杀率 | T09/T20 扩展 | TOP5 + 非TOP5 合计等于 ALL；top_n=5 |
| PX06 | 首存用户 D30 趋势 | 输出租户平台990001渠道990011首存用户从D1到D30每日充值、提现、有效投注、输赢、杀率和投充比 | T10 长周期扩展 | D1~D30 行数完整；无分母时比例为空 |
| PX07 | 游戏类型自然周 / 月 | 按自然周统计租户平台990001渠道990011各游戏类型有效投注、下注次数、均注、输赢和投注占比 | T11 扩展或普通 SQL | 自然周边界正确；占比分母为周内总有效投注 |
| PX08 | 首存金额全桶位 | 按首存金额固定档位输出租户平台990001渠道990011在2026-04-01到2026-04-30的首存人数与占比，包含10、20、30、50、100、200、300、400、500、1000、2000、2000以上和其他 | T13/T22 扩展 | 所有桶位占比合计为 1；2000 与 2000以上不混淆 |
| PX09 | 续存汇总行 | 按日输出租户平台990001渠道990011在2026-04-01到2026-04-07首存及2到6存人数、率、均额，并输出汇总行 | T08 扩展 | 汇总比例按汇总分子/分母重算，不做每日比例简单平均 |
| PX10 | ROI 外部补充后实算 | 统计租户平台990001下渠道990011在2026-04-01到2026-04-03首存cohort从D1到D7的ROI，投放金额为04-01=1000、04-02=2000、04-03=500 | 先识别外部依赖，补充后继续计算 | ROI = cumulative_channel_revenue / ad_spend；不编造投放金额 |
| PX11 | 流量外部补充后实算 | 统计租户平台990001下渠道990011在2026-04-01到2026-04-06的渠道日报，并补充PV、UV、下载点击UV、UV下载率和UV注册率 | 先识别外部依赖，补充后继续计算 | UV下载率=下载点击UV/UV；UV注册率=注册人数/UV |
| PX12 | 泛化问题图表 | 对“TOP5和非TOP5用户存款、有效投注、输赢、投充比和杀率”生成分组对比图 | 图表生成 | 图表分组、指标、排序与 preview 一致 |

## 4. 语言泛化用例（LING）

| ID | 覆盖点 | 推荐问题 | 路由预期 | 关键验收点 |
| --- | --- | --- | --- | --- |
| LING01 | 首存同义词 | 统计渠道990011在2026-04-01到2026-04-03首充用户的二存到六存情况 | 命中 `first_deposit` / `retention_deposit` | “首充”按首存解释，结果等价 T08 |
| LING02 | 续存同义词 | 看一下渠道990011首存用户后续复存、二存、三存的转化 | 命中续存词典或澄清周期 | “复存 / 二存 / 三存”不被解释成普通充值总数 |
| LING03 | TOPN 口语表达 | 找出这个渠道投注流水最高的前5个大户，和其他用户对比投充比与杀率 | 命中 TOPN 口径 | “大户 / 前5个”映射 TOP5 |
| LING04 | 杀率同义词 | 看渠道990011在2026-04-01到2026-04-07的平台赢率和有效投注 | 命中 `kill_rate` | 平台赢率 = 输赢 / 有效投注 |
| LING05 | ROI 同义词 | 看这个渠道首存用户投放回收和回本情况 | 命中 ROI + 外部依赖 | 识别需要投放金额，不直接编造 ROI |
| LING06 | 投充比同义词 | 统计渠道990011的流水充值比 | 命中 `bet_deposit_ratio` | 流水充值比 = 有效投注 / 存款金额 |
| LING07 | 首存成本同义词 | 这个渠道新客首充成本是多少 | 命中 `first_deposit_cost` + `ad_spend` | 识别需要投放金额和首存人数 |
| LING08 | 流量指标口语表达 | 帮我把访问量、独立访客、下载点击人数一起放进渠道日报 | 命中 PV/UV/download_click_uv 外部依赖 | 明确索取外部流量指标 |
| LING09 | 模糊表达澄清 | 帮我看看这个渠道最近效果怎么样 | 不应强行命中模板 | 先澄清渠道、时间、关注指标，或给出可选分析方向 |
| LING10 | 明确不用模板 | 不用业务报表模板，直接查原始充值订单，看看渠道990011每天成功充值金额 | 普通 text-to-SQL 优先 | 不应为了模板覆盖而强行 anchored_template |

## 5. 数据边界用例（EDGE）

| ID | 覆盖点 | 推荐问题 | 路由预期 | 关键验收点 |
| --- | --- | --- | --- | --- |
| EDGE01 | 失败充值排除 | 查询渠道990011在2026-04-01到2026-04-07成功充值金额，并说明失败充值是否计入 | 普通 SQL 或 T01 | 失败充值 66 不计入成功充值 |
| EDGE02 | 渠道隔离 | 查询渠道990011充值金额，同时对照渠道990012是否被混入 | 普通 SQL 或 T01 | 990012 数据不能串入 990011 |
| EDGE03 | 分母为 0 | 查询无充值但有投注或无投注日期的投充比/杀率 | 规则约束 | 分母为 0 时返回 NULL/空值/无法计算，不报错不编造 |
| EDGE04 | 无数据日期 | 查询渠道990011在2026-04-20单日的综合日报基础指标 | 普通 SQL 或模板 | 无数据时明确为空或 0，口径一致 |
| EDGE05 | TOPN 人数不足 | 对一个活跃用户不足5人的时间段查询TOP5和非TOP5 | TOPN 规则 | TOPN 不足时说明实际人数，不出现负数非TOPN |
| EDGE06 | 2000边界 | 查询首存金额2000元和2000元以上的用户数 | T13/T22 | 2000 元归固定 2000 桶，>2000 归 2000以上 |
| EDGE07 | 空金额桶 | 查询全首存金额桶位并展示无数据桶 | T13/T22 | 空桶按产品约定返回 0 或不展示，报告需说明策略 |
| EDGE08 | 空游戏类型 | 查询全游戏类型流水分布，包含当前无数据的游戏类型 | T11/T21 | 无数据类型处理策略明确；总有效投注可对账 |
| EDGE09 | 登录去重 | 查询每天登录去重玩家数 | 普通 SQL | 同一玩家多次登录只计 1 人 |
| EDGE10 | 负优惠 / 加扣款 | 查询综合日报中的优惠、彩金、加扣款合计 | T01 | 负数加扣款按符号参与计算，不被取绝对值 |
| EDGE11 | 汇总比例重算 | 查询首存率/续存率的汇总行 | T08/FC07 | 汇总比例按汇总后分子分母重算，不是每日比例平均 |
| EDGE12 | 占比合计 | 查询游戏类型投注占比或首存金额桶占比 | T11/T13 | 占比合计接近 1，允许四舍五入误差 |

## 6. 路由泛化用例（ROUTE）

| ID | 覆盖点 | 推荐问题 | 路由预期 | 关键验收点 |
| --- | --- | --- | --- | --- |
| ROUTE01 | 明确模板命中 | 统计租户平台990001下渠道990011在2026-04-01到2026-04-07的综合日报 | anchored_template | diagnostics 可见 `templateDecision.mode = anchored_template` 或等价字段 |
| ROUTE02 | 普通 SQL 路径 | 直接查原始提现订单，统计租户平台990001下渠道990011在2026-04-01到2026-04-07每天成功提现金额 | 普通 text-to-SQL | 不要求命中 SQL 模板，SQL 来源可为 LLM/semantic |
| ROUTE03 | 参数不足澄清 | 看这个渠道最近表现怎么样 | 澄清 | 不应默认编造渠道、日期或指标 |
| ROUTE04A | ROI缺平台优先澄清 | 统计渠道990011首存cohort的ROI | 澄清 | 缺租户平台时先补业务槽位，不直接进入外部投放金额阻断 |
| ROUTE04 | 外部依赖阻断 | 统计租户平台990001下渠道990011在2026-04-01到2026-04-03首存cohort的ROI | 阻断 / 索取外部数据 | 业务槽位齐全后要求补充投放金额，不编造 ROI |
| ROUTE05 | 外部补充后继续 | 在 ROUTE04 后补充投放金额并继续计算 | 继续计算 | 能复用上文 cohort 条件并输出 ROI |
| ROUTE06 | negative cue 防误命中 | 不要用综合日报模板，直接查充值订单表，统计租户平台990001下渠道990011在2026-04-01到2026-04-07的成功充值笔数 | 普通 text-to-SQL | 不被 T01 错误锚定 |
| ROUTE07 | 追问继承条件 | 对 T09 结果追问“那只看非TOP3呢？” | 追问 | 继承日期、渠道、tenant_plat_id，仅切换用户类型 |
| ROUTE08 | 追问改 TOPN | 对 TOP3 结果追问“改成TOP5看看” | 追问 + 参数变更 | top_n 从 3 改为 5，其他条件不丢失 |
| ROUTE09 | 保存参考样例后复用 | 保存一次普通问数结果为“参考样例”，再用相近问题提问 | reference | 可参考 SQL，但不应自动升级为业务口径 |
| ROUTE10 | 保存业务口径后命中 | 管理员保存高置信结果为“业务口径”，再用等价业务问题提问 | anchored_template | 能观察到业务模板命中；非管理员不能标记业务口径 |
| ROUTE11 | diagnostics 完整性 | 分别检查模板命中、普通 SQL、阻断、失败问数的调用诊断 | 可观测性 | 问题、thread ID、SQL 来源、降级原因不为空 |
| ROUTE12 | 图表 / 数据表分流 | 对图表型结果固定到看板，对表格型结果保存为数据表 | 产品路由 | 图表不保存成数据表；表格不固定到看板 |

## 7. 执行建议

### 7.1 完整泛化回归

当需要声明“业务需求泛化能力通过”时，建议至少执行：

1. PX01~PX12 全量。
2. LING01~LING10 全量。
3. EDGE01~EDGE12 全量。
4. ROUTE01~ROUTE12 全量。

通过条件：

- PX 用例中参数变化不导致明显错表、错口径、错过滤条件。
- LING 用例中业务词典 alias 能被召回；模糊表达不被强行模板化。
- EDGE 用例中边界数据不报错、不编造、不串渠道。
- ROUTE 用例中 diagnostics 能解释系统为什么选择模板、普通 SQL、澄清或阻断。

### 7.2 最小泛化 smoke

当只做最小回归时，建议抽样：

| 用例集 | 最小 smoke |
| --- | --- |
| PX | PX02、PX05、PX10 |
| LING | LING01、LING05、LING10 |
| EDGE | EDGE01、EDGE03、EDGE06 |
| ROUTE | ROUTE01、ROUTE03、ROUTE04A、ROUTE04、ROUTE11 |

### 7.3 与测试报告的关系

最终报告建议单独列出以下结论，避免把“模板覆盖率高”误判为“泛化能力强”：

- 核心模板回归：T01~T15 是否通过。
- Excel 全覆盖补充：FC01~FC12 是否通过或被标记为产品缺口。
- 普通问数泛化：OQ01~OQ10 是否通过，且不被过度模板锚定。
- 业务需求泛化：PX / LING / EDGE / ROUTE 是否通过。
