# 问数策略配置建议

日期：2026-05-01

## 1. 结论

`docs/业务需求` 下此前没有单独的“问数策略配置清单”文档；相关设计分散在：

- `docs/业务需求/问数语义计划与知识治理方案-2026-04-30.md`
- `docs/业务需求/knowledge-base/import-format.md`
- `docs/业务需求/knowledge-base/**`

当前系统已经具备问数策略能力：

- UI 已在知识库工作台提供“问数策略”入口；
- UI server 会把当前 workspace / knowledge base 下的 active policy rules 注入 AI service；
- AI service `ask_policy.py` 支持 request-level policy 与 `WREN_ASK_POLICY_FILE` 文件策略；
- 当前策略字段包括：
  - `query_contains_any`
  - `template_ids`
  - `forbidden_templates`
  - `required_slots`
  - `reason_code`

因此，本文件不是新增一套独立机制，而是把当前业务需求沉淀为可录入、可回归、可扩展的默认问数策略。

## 2. 推荐录入位置

优先录入：知识库详情页的 **问数策略** tab。

原因：

1. 问数策略应随知识库生效，而不是全系统无差别生效。
2. 作用范围默认就是当前知识库，用户不需要再配置 scope。
3. 系统设置可保留为高级治理入口，但不应作为业务人员配置问数策略的主入口。

## 3. 默认问数策略清单

### 3.1 必填租户平台策略

| 字段 | 建议值 |
| --- | --- |
| 策略 ID | `require_tenant_plat_for_core_metrics` |
| 名称 | 核心经营指标必须明确租户平台 |
| 触发词 | 充值、存款、首存、首充、续存、提现、投注、杀率、投充比、ROI、渠道、玩家、cohort、留存、日龄、首存成本、获客成本 |
| 必填槽位 | `tenant_plat_id` |
| 禁用模板 | 空 |
| reason_code | `missing_required_tenant_plat_id` |
| 用户提示 | “请补充要查询的租户平台，例如：租户平台 990001。” |
| 验收用例 | LING01 原始问题缺少租户平台时，不进入 SQL 生成，先触发澄清 |

  - 存款
  - 首存
  - 首充
  - 续存
  - 提现
  - 投注
  - 杀率
  - 投充比
  - ROI
  - 渠道
  - 玩家
  - cohort
  - 留存
  - 日龄
  - 首存成本
  - 获客成本

### 3.5 外部投放金额缺失阻断策略

这类策略不建议强行塞进 `ask_policy_rule`，因为当前 `ask_policy_rule` 字段只能表达模板禁用和必填槽位，不能表达“缺少外部数据源”。应继续由外部数据依赖资产承载：

- `docs/业务需求/knowledge-base/external-dependencies/ad_spend.md`
- 关联业务词：`roi`、`first_deposit_cost`
- 关联模板：`T05`、`T14`
- 缺失行为：`ask_user`

治理要求：

| 场景 | 行为 |
| --- | --- |
| 问题要求 ROI、投放金额、首存成本、获客成本、CPA | 如果缺少投放金额，必须阻断并追问 |
| 问题只查站内充值、投注、提现、登录、注册 | 不应触发投放金额追问 |
| 用户没有提供投放数据 | 不允许 LLM 编造投放金额 |

### 3.6 外部流量指标缺失阻断策略

这类策略同样由外部数据依赖资产承载：

- `access_pv`
- `access_uv`
- `download_click_uv`

治理要求：

| 场景 | 行为 |
| --- | --- |
| 问题要求 PV、UV、下载点击 UV、UV 下载率、UV 注册率 | 缺少外部流量指标时阻断并追问 |
| 问题只查站内充值、投注、首存、续存 | 不应触发流量指标追问 |
| 用户没有提供流量数据 | 不允许 LLM 编造 PV / UV / 下载点击 UV |

### 3.7 TOPN / 大户口语与活跃用户边界

这类规则不建议新增为 `ask_policy_rule`，因为当前问数策略字段只能表达“必填槽位 / 禁用模板”，不能表达默认排序指标和人数不足 N 的计算边界。应继续由业务字典、分析规则和 SQL 模板共同承载：

- 业务字典：`docs/业务需求/knowledge-base/business-dictionary/topn_segment.md`
- 分析规则：`docs/业务需求/knowledge-base/analysis-rules/R05_TOPN 口径.md`
- 关联模板：`T06`、`T09`、`T12`

治理要求：

| 场景 | 行为 |
| --- | --- |
| 用户说“大户 / 头部用户 / 高流水用户 / 投注流水最高的前 N 个用户” | 默认识别为 TOPN 用户分层 |
| 未明确 TOPN 排序依据，但出现“投注流水最高 / 高流水 / 大户” | 默认按统计区间内累计有效投注额排序，不再追问排序依据 |
| 用户说“其他用户 / 其余用户 / 非TOPN” | 与 TOPN 形成 `TOPN` / `NON_TOPN` 对比 |
| 活跃用户不足 N 人 | TOPN 使用实际活跃用户数，非 TOPN 为空集合或 0，并在回答中说明 |
| 用户明确指定其他排序指标 | 以用户显式指标为准；如模板不支持，应降级为普通问数或追问确认 |

## 5. 分析规则 / 业务字典 / 外部数据依赖是否需要优化

需要优化，但不建议把三类能力合并成一种配置。

推荐边界：

| 资产类型 | 应承担什么 | 不应承担什么 |
| --- | --- | --- |
| 分析规则 | 业务口径、公式解释、适用报表、计算注意事项 | 不直接承担模板禁用、跨轮槽位状态、外部数据补齐 |
| 业务字典 | 业务词别名、规范表达式、适用/不适用场景、必填槽位、关联规则/模板 | 不写复杂 SQL、不承载外部数据生命周期 |
| 外部数据依赖 | 缺失指标定义、触发/不触发场景、补充粒度、校验、追问话术 | 不自动假设已能联邦 join，不编造缺失值 |
| 问数策略 | 必填槽位、模板禁用、路由 guard、可测试 reason_code | 不替代业务口径文档，也不替代外部数据依赖 |

### 5.1 分析规则优化建议

现状检查：

- R01-R14 已经有 `related_business_terms`、`related_external_dependencies`、`runtime_usage` 字段。
- 分析规则目前更适合继续作为 instruction retrieval / template matching 的业务口径来源。

建议优化：

1. 保持分析规则只表达业务口径，不把模板禁用逻辑写进分析规则正文。
2. R02 / R03 / R10 等首存、首充、续存规则应明确关联：
   - `first_deposit`
   - `new_customer_first_deposit`
   - `retention_deposit`
   - `T03` / `T04` / `T08` / `T10` / `T13`
3. R07 / R09 / R13 应明确关联外部依赖：
   - `ad_spend`
   - `access_pv`
   - `access_uv`
   - `download_click_uv`
4. R13 “缺失数据源处理”应成为外部依赖的统一兜底规则：
   - 命中外部依赖但数据缺失时：追问；
   - 未命中外部依赖时：不应泛化追问；
   - 不允许 LLM 编造外部数据。
5. 如果后续需要在分析规则中声明槽位，建议只作为“提示字段”而非 runtime 硬门槛；硬门槛仍交给问数策略 / 业务字典 / SQL 模板。

### 5.2 业务字典优化建议

现状检查：

- `first_deposit.md` 已经升级到 v2，包含适用场景、不适用场景和 `required_slots`。
- 其他业务词多数仍缺少 v2 治理字段。

建议为所有核心业务词补齐：

| 字段 | 用途 |
| --- | --- |
| `applicable_scenarios` | 防止业务词被过度泛化 |
| `not_applicable_scenarios` | 作为 negative cues 来源，降低误锚定 |
| `required_slots` | 让问数策略和模板 guard 能抽取必填槽位 |
| `supported_grains` | 为 P1 SemanticPlan 的 grain guard 做准备 |
| `related_rules` | 连接分析规则 |
| `related_templates` | 连接 SQL 模板 |
| `conflict_terms` | 标记容易混淆的业务词 |

优先补齐对象：

| 业务词 | 必补原因 | 建议 required_slots |
| --- | --- | --- |
| `first_deposit` | 已补大部分字段，建议补 `supported_grains` | `tenant_plat_id` |
| `new_customer_first_deposit` | 容易和普通首存混淆 | `tenant_plat_id` |
| `retention_deposit` | 续存/留存/cohort 容易误路由 | `tenant_plat_id` |
| `topn_segment` | TOP3 / 非 TOP3 是分层口径 | `tenant_plat_id` |
| `bet_deposit_ratio` | 依赖充值与有效投注口径 | `tenant_plat_id` |
| `kill_rate` | 依赖输赢与有效投注口径 | `tenant_plat_id` |
| `roi` | 依赖外部投放金额 | `tenant_plat_id` |
| `first_deposit_cost` | 依赖外部投放金额和首存人数 | `tenant_plat_id` |
| `traffic_metrics` | 依赖外部 PV / UV / 下载点击 UV | 视问题需要 `channel_id`、`date_range` |

### 5.3 外部数据依赖优化建议

现状检查：

- `ad_spend.md` 已经升级到 v2，包含触发/不触发场景、生命周期和输入方式。
- `access_pv.md`、`access_uv.md`、`download_click_uv.md` 仍是 v1，缺少触发/不触发场景、生命周期和输入方式。

建议将四个外部依赖统一到 v2：

| 字段 | 建议 |
| --- | --- |
| `trigger_when` | 明确哪些问题会触发外部依赖 |
| `not_trigger_when` | 明确哪些站内指标问题不触发外部依赖 |
| `lifecycle` | 统一为 `per_question` |
| `input_modes` | 至少支持 `single_value`、`csv_upload` |
| `required_grain` | 保留当前粒度描述 |
| `required_grain_schema` | 后续用于结构化校验 |
| `value_schema` | 后续用于多列 CSV / 表格校验 |
| `validation` | 保留数值非负等基础校验 |
| `join_contract` | 只作为目标设计，不视为当前自动 join 能力 |

特别注意：

- 当前外部依赖仍是“缺失阻断 + 追问”，不是自动联邦 join。
- 即使文档写了 `join_contract`，也不能把它当成已实现的数据拼接能力。
- 对 ROI、PV、UV、下载点击 UV，宁可追问，也不要生成看似完整但基于编造外部值的 SQL 或解释。

## 6. 落地优先级

| 优先级 | 动作 | 原因 |
| --- | --- | --- |
| P0 | 在知识库“问数策略”中录入 3.1-3.4 | 直接覆盖 LING01 / OQ06 / OQ08 的路由风险 |
| P0 | 将策略 reason_code 纳入回归记录 | 便于判断是模板、策略还是外部依赖生效 |
| P1 | 把 `access_pv` / `access_uv` / `download_click_uv` 升级为 v2 | 减少外部依赖误触发或漏触发 |
| P1 | 给业务字典补齐 v2 场景字段 | 让模板 negative cues 和 SemanticPlan 有稳定来源 |
| P2 | 给外部依赖补 `required_grain_schema` / `value_schema` / `join_contract` | 为后续结构化补数和联邦 join 做准备 |
| P2 | 增加策略变更影响用例列表 | 让策略修改前能知道会影响哪些回归 case |

## 7. 回归测试要求

录入或修改策略后，至少覆盖：

| 用例 | 期望 |
| --- | --- |
| LING01 缺 `tenant_plat_id` | 返回澄清，不进入 SQL 生成 |
| LING01 补充 `tenant_plat_id=990001` | 能继续问数并正确识别“首充/首存” |
| OQ06 | 不使用 `T01` 渠道日报模板 |
| OQ08 | 不使用 `T03` / `T08` 首存 cohort 类模板 |
| ROI / 首存成本问题且未提供投放金额 | 触发外部投放金额追问 |
| PV / UV / 下载点击 UV 问题且未提供外部流量数据 | 触发对应外部数据追问 |
| 普通充值 / 投注 / 提现问题 | 不误触发外部投放或流量指标追问 |
| LING03-SLOT 大户 / 前5个 / 投注流水最高 | 命中 TOPN 口径，补齐租户、渠道、时间后能生成 SQL |
| EDGE05-SLOT 活跃用户不足 5 人 | 使用默认活跃用户和 TOPN 排序口径，不继续追问排序依据 |

最终仍需按 `docs/业务需求/问数回归测试计划.md` 跑全量回归，不能只跑新增策略命中的少量用例。
