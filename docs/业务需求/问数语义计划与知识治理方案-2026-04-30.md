# 问数语义计划与知识治理方案

日期：2026-04-30

## 1. 背景

当前问数回归中暴露了几类典型问题：

1. **SQL 模板误锚定**
   - OQ06 被错误锚定到渠道日报模板。
   - OQ08 被错误锚定到首存 cohort 模板。
   - 本质问题不是某几个问题的关键词不准，而是系统缺少通用的“模板采纳安全门槛”。

2. **关键业务参数缺失但没有先澄清**
   - LING01 中“统计渠道990011...”缺少 `tenant_plat_id`。
   - 系统没有先追问，而是进入 SQL 生成并失败。
   - 加上“租户平台990001”后，同义词“首充”可以正常识别，说明核心问题是必填槽位缺失。

3. **业务知识结构还不够支撑运行时决策**
   - 业务词典目前偏“同义词 + 定义 + 关联字段”。
   - 外部数据依赖目前偏“缺失说明 + 追问话术”。
   - SQL 模板目前偏“可参考 SQL”，但系统还不能稳定判断什么时候可硬套、什么时候只能参考、什么时候必须不用。

本方案目标是：在不让用户理解复杂技术概念的前提下，把问数系统升级为更稳定、可解释、可扩展的产品能力。

## 2. 代码现状与复用基线

本方案不是从零建设一套全新的问数系统。当前代码中已经存在多项可复用能力，P0 应优先复用和连通这些能力，而不是重建。

### 2.1 已存在能力

| 能力 | 代码现状 | P0 处理方式 |
| --- | --- | --- |
| 意图分类 | `wren-ai-service/src/pipelines/generation/intent_classification.py` 已有 `TEXT_TO_SQL` / `GENERAL` / `MISLEADING_QUERY` / `USER_GUIDE` 四分类 | 不重建，作为后续 SemanticPlan 的 intent 来源之一 |
| 模板评分与分级 | `fixed_order_ask_runtime.py` 已有 `_score_sql_sample_for_query()`、`build_template_decision()`，并支持 `reference` / `trusted_reference` / `anchored_template` / `executable_template` | 先诊断 OQ06 / OQ08 的实际 decision trace，再决定补通用 guard、negative scenarios 或阈值 |
| 模板阈值配置 | `fixed_order_ask_runtime.py` 当前直接通过 `os.getenv("WREN_TEMPLATE_*")` 读取阈值，不在 `config.py` 统一配置 | P0 继续按现状使用环境变量；如需产品化配置中心，作为后续配置治理任务 |
| 模板参数抽取 | `fixed_order_ask_runtime.py` 已有 `_extract_template_parameters_from_query()` 和 `missing_parameters` | 可复用，但它依赖模板 placeholders；P0 仍需新增独立 slot guard 覆盖未命中模板的普通问数 |

| diagnostics 数据 | `ThinkingTrace` / `AskTemplateDecision` 已包含 `decision_reason`、`fallback_reason`、`missing_parameters`、`template_mode` 等字段 | P0 先展示已有字段；标准 `decision` contract 可作为 P1 规范化目标 |
| SQL 模板元数据 | `sql_pair` 已有 `templateMode`、`parameterSchema`、`businessSignature` 等字段 | P0 重点补 UI 暴露与导入文档，而非新增 DB 字段 |
| 分析规则关联字段 | repository / service / REST API / adaptor 已支持 `relatedBusinessTerms`、`relatedExternalDependencies` | 如 GraphQL controller 仍被使用，补齐该路径透传；不是全链路缺失 |

| 能力 | 当前缺口 | 建议阶段 |
| --- | --- | --- |

## 3. 核心原则

### 3.1 用户界面简单，系统内部结构化

用户维护的是：

- 业务词是什么意思；
- 有哪些同义词；
- 用哪些字段计算；
- 问这类问题需要哪些条件；
- 缺外部数据时怎么处理；
- SQL 模板适合 / 不适合什么场景。

系统内部再转成：

- 语义计划；
- 槽位策略；
- 模板采纳策略；
- 外部数据契约；
- diagnostics 诊断信息。

### 3.2 SQL 模板是候选知识，不是默认答案

SQL 模板不应该只因为文本相似度高就被采纳。  
系统必须先判断：

- 意图是否匹配；
- 业务主体是否匹配；
- 指标是否匹配；
- 维度 / 粒度是否匹配；
- 必填参数是否齐全；
- 是否触发不适用场景。

只有全部满足时，才允许使用硬模板。

### 3.3 缺关键参数时必须先澄清

对于 `tenant_plat_id` 这类业务关键参数，不能依赖 LLM 自觉补齐。  
系统应在 SQL 生成前执行确定性检查：

1. 用户问题里是否明确给出；
2. 对话上下文是否可以继承；
3. workspace / 知识库是否有唯一默认值；
4. 如果都没有，则必须追问，不进入 SQL 生成。

### 3.4 路由结果必须可解释

diagnostics 需要解释：

- 为什么采用 SQL 模板；
- 为什么没有采用 SQL 模板；
- 为什么降级为普通 SQL；
- 为什么需要追问；
- 为什么因为外部数据缺失而阻断。

```text
SQL 生成 / 模板实例化
  ↓

  ↓

```

## 5. 运行时决策契约

为了避免策略散落在 prompt、retrieval 排序和 SQL 生成逻辑中，问数 runtime 应先产出统一的决策对象，再进入 SQL 生成或澄清流程。

### 5.1 标准路由枚举

本方案统一区分“模板类型”和“本次问数路由结果”。

#### 模板类型 `template_type`

| 类型 | 含义 |
| --- | --- |
| `executable_template` | 可直接实例化的固定业务模板 |
| `reference_example` | 只能作为参考样例，不允许直接套用 |

> 注意：`template_type` 是文档侧概念，当前代码真实字段是 `templateMode`。缺外部数据不建议作为 `templateMode` 的新枚举表达，而应通过 `businessSignature.externalDependencies` + 外部依赖检测得到 `blocked_missing_external_data` 路由。

#### 问数路由 `route`

| 路由 | 含义 |
| --- | --- |
| `template_answer` | 使用可执行模板实例化 SQL 并回答 |
| `template_reference_sql` | 参考模板生成 SQL，但不直接套模板 |
| `normal_text_to_sql` | 不采用模板，走普通 text-to-SQL |
| `clarification_required` | 缺少关键信息，先追问 |
| `blocked_missing_external_data` | 缺少外部数据，先阻断并要求补充 |
| `safe_empty_result` | 查询合法但结果为空，安全返回空结果解释 |
| `failed` | 生成、校验或执行失败 |

> 说明：历史 UI 文案中的 `anchored_template` 可作为 `template_answer` 的展示别名，但内部 route 建议统一为 `template_answer`。

### 5.2 决策对象结构

Runtime 每次问数应生成类似结构，供 SQL 生成、diagnostics、测试断言和 UI 展示使用：

```yaml
decision:
  route: normal_text_to_sql
  sql_source: llm_generated
  selected_template_id: null
  selected_template_type: null
  candidate_templates:
    - id: T08
      template_type: executable_template
      decision: rejected
      reason_codes:
        - subject_mismatch
        - grain_mismatch
      diagnostics_summary: 当前问题不是首存 cohort 续存分析，因此未采用 T08。
  missing_slots: []
  resolved_slots:
    tenant_plat_id:
      value: "990001"
      source: explicit_user_input
  external_dependencies: []
  diagnostics_summary: 未采用业务模板，已按普通问数生成 SQL。
```

#### P0 route 与现有字段映射

P0 不要求立刻新增统一 `route` API 字段，可以先从现有 `AskTemplateDecision` 和阻断逻辑推导：

| 目标 route | P0 可用判断 |
| --- | --- |
| `template_answer` | `template_decision.mode in [anchored_template, executable_template]` 且 `sql_source in [anchored_template, rendered_template]` |
| `template_reference_sql` | `template_decision.mode = trusted_reference`，或有候选模板但 `sql_source = generated` |
| `normal_text_to_sql` | `decision_reason = no_sql_pair_candidates`，或模板因低置信 / 冲突降级为 reference |
| `clarification_required` | P0 slot guard 返回缺必填槽位阻断提示 |
| `blocked_missing_external_data` | `detect_missing_external_source_requirement()` 命中 |
| `failed` | SQL 生成、dry-run、修正或执行失败 |

P1 再将这些推导结果规范化为显式 `decision.route`。

> P0 实现边界：`clarification_required` 是 diagnostics / route 层语义，不要求新增 AI-service API status。实际响应可复用当前外部数据缺失的交互形态：`status = finished`、`type = GENERAL`、`general_type = DATA_ASSISTANCE`、`content = 追问提示`。P1 再设计真正的 `needs_clarification` 状态、`clarification_session_id` 和 `slot_values` resume 流程。

### 5.3 标准拒绝 / 降级原因

模板候选被拒绝、降级或阻断时，必须使用标准 reason code，避免 diagnostics 只能展示自然语言：

| reason code | 含义 |
| --- | --- |
| `intent_mismatch` | 查询意图不匹配 |
| `subject_mismatch` | 业务主体不匹配，例如玩家查询误命中渠道模板 |
| `metric_mismatch` | 指标不匹配 |
| `dimension_mismatch` | 维度不匹配 |
| `grain_mismatch` | 结果粒度不匹配 |
| `missing_required_slot` | 缺少必填业务参数 |
| `external_dependency_missing` | 缺少外部数据 |
| `negative_scenario_matched` | 命中模板不适用场景 |
| `low_template_coverage` | 模板无法覆盖用户问题中的关键指标 / 条件 |
| `dry_run_failed` | 模板实例化 SQL 校验失败 |
| `llm_plan_unavailable` | 语义计划抽取失败，禁止高置信硬套模板 |

### 5.4 决策优先级

建议按照以下顺序执行：

1. 解析用户问题中的显式参数和业务词。
2. 生成或兜底生成语义计划。P0 可先使用 deterministic slot / cue guard；完整 SemanticPlan 放到 P1。
3. 执行业务槽位检查。
4. 如果缺少必填槽位且不能从上下文 / 默认值唯一补齐，则返回 `clarification_required`。
5. 检查外部数据依赖。
6. 如果缺少外部数据，则返回 `blocked_missing_external_data`。
7. 对 SQL 模板候选执行采纳策略。
8. P0 只有通过 required slots、positive / negative cues、features、externalDependencies 等当前代码可消费的最小 guard，才允许 `template_answer`。
9. 部分匹配但不能硬套时，降级为 `template_reference_sql`。
10. 无可靠模板时，使用 `normal_text_to_sql`。

> P1 引入结构化 SemanticPlan 后，再把 subject / metric / dimension / grain 提升为可程序化硬门槛。P0 文档中的 `expected_grain` 只能作为诊断和未来策略字段，不应在没有结构化计划的情况下直接阻断模板。

## 6. 知识资产设计

最终建议形成 5 类结构化知识。

### 6.1 业务词典 `business_dictionary`

#### 用户可见字段

P0 阶段只暴露用户可理解字段：

- 名称
- 同义词
- 定义
- 计算口径
- 关联表
- 关联字段
- 关联分析规则
- 关联 SQL 模板
- 适用场景
- 不适用场景

#### 系统内部字段

由系统或 AI assistant 自动生成，不默认暴露：

```yaml
semantic_type: metric | dimension | segment | formula | filter | entity | time_expression
business_subject: player | channel | game | cohort | traffic
intent_tags: []
supported_grains: []
required_slots: []
optional_slots: []
positive_examples: []
negative_examples: []
route_policy:
  can_trigger_template: true
  can_trigger_clarification: true
  can_trigger_external_dependency: false
```

#### 示例：首存

用户看到：

```text
业务词：首存
同义词：首充、首次存款、第一次充值
定义：成功存款且 times = 1
计算口径：status = 2 AND times = 1
关联字段：dwd_order_deposit.status、times、callback_time
```

系统内部可推断：

```yaml
semantic_type: metric
business_subject: player
intent_tags:
  - first_deposit
  - cohort
required_slots:
  - tenant_plat_id
supported_grains:
  - player_id + date_range
  - channel_id + first_deposit_date
```

### 6.2 业务槽位策略 `business_slot_policy`

这是建议新增的知识类型。

> P0 阶段不建议立即新增完整 DB / UI 资产。先以内置 `tenant_plat_id` 策略 + 业务词典 / SQL 模板中的 `required_slots` 字段落地；P1/P2 再沉淀为可配置资产。

#### 用户可见字段

- 参数名称
- 参数说明
- 适用业务场景
- 缺失时处理方式
- 默认追问话术
- 是否允许从上下文继承
- 是否允许使用 workspace / 知识库默认值

#### 系统内部字段

```yaml
id: tenant_plat_id
name: 租户平台
semantic_type: required_filter
required_when:
  - business_subject in [player, channel, recharge, bet, cohort]
  - source_tables contains dwd_order_deposit
  - source_tables contains dwd_bet_order
resolution_order:
  - explicit_user_input
  - conversation_context
  - workspace_default
  - knowledge_base_default
  - clarify
clarify_prompt: 请问要查询哪个租户平台？例如 990001。
```

#### 用途

用于解决 LING01 这类问题：

- 用户问“统计渠道990011的首充情况”；
- 系统识别到“首充”和“渠道”；
- 系统发现缺少 `tenant_plat_id`；
- 如果上下文和默认值都不能补齐，则先追问；
- 不进入 SQL 生成。

---

### 6.3 外部数据依赖 `external_dependency`

#### 用户可见字段

- 外部指标名称
- 同义词
- 当前状态：系统内已有 / 系统内暂无 / 可连接外部源 / 用户手动补充
- 缺失时处理方式
- 需要的统计粒度
- 补充提示语
- 校验规则

#### 系统内部字段

```yaml
semantic_type: external_metric
source_status: missing | available | connector_available | user_supplied
required_grain_schema:
  dimensions:
    - biz_date
    - channel_id
  time_grain: day
  required_keys:
    - biz_date
    - channel_id
value_schema:
  field: ad_spend
  type: number
  unit: currency
  min: 0
resolution_policy:
  order:
    - connected_source
    - uploaded_file
    - manual_input
    - clarify
missing_behavior: ask_user
input_modes:
  - single_value
  - table_input
  - csv_upload
join_contract:
  join_keys:
    - biz_date
    - channel_id
  join_type: left
trigger_when:
  any_metric:
    - roi
    - first_deposit_cost
not_trigger_when:
  - only_asks_recharge_amount
  - only_asks_player_count
```

> 当前实现边界：`join_contract` 是目标设计字段，用于描述未来外部数据补齐后的关联键和关联方式。现有代码更偏向“缺失说明 + 追问话术 + requiredGrain / externalDependencies 检测”：UI 侧会把外部数据依赖转换为 instruction metadata，AI-service runtime 通过 `detect_missing_external_source_requirement()` 判断是否阻断并提示补充数据；尚未根据 `join_contract` 自动生成联邦 join 或自动拼接外部数据。因此 P0 不应把 `join_contract` 当作已完整执行的能力。

#### 用途

用于处理 T05 / T14 / T15：

- ROI 依赖投放金额；
- 投放金额当前 TiDB 数据源没有；
- 系统不能编造；
- 必须提示用户补充对应粒度的数据；
- 用户补充后再继续计算。

#### 外部数据生命周期

外部数据补充后必须明确生命周期，避免跨 workspace / 跨问题污染：

| 生命周期 | 建议用途 |
| --- | --- |
| `per_question` | 默认值，仅当前问数有效 |
| `thread_session` | 当前对话内复用，需要 UI 明确提示 |
| `workspace_asset` | 保存为数据表或外部数据资产，需要权限与审计 |

P0 建议只支持 `per_question`，即：

- 用户补充的数据只用于当前问数；
- 不自动写入知识库；
- 不自动复用到其他对话；
- diagnostics 记录本次使用了用户补充数据；
- 后续如需复用，应通过“保存为数据表 / 外部数据资产”形成显式动作。

### 6.4 SQL 模板 `sql_template`

#### 用户可见字段

- 模板名称
- 适用场景
- 不适用场景
- 必填参数
- 结果粒度
- 关联业务词
- 关联分析规则
- 外部数据依赖
- 示例问题
- SQL 内容

#### 系统内部字段

```yaml
template_type: executable_template | reference_example
business_signature:
  template_id: T01
  concepts: []
  features: []
  metrics: []
  dimensions: []
  parameter_slots: []
  external_dependencies: []
  positive_cues: []
  negative_cues: []
  expected_grain: biz_date + channel_id
adoption_policy:
  require_intent_match: true
  require_grain_match: true
  require_required_slots: true
  allow_reference_when_partial_match: true
```

#### 模板分级

| 文档侧类型 | 当前代码字段 | 用途 | 是否可直接套用 |
| --- | --- | --- | --- |
| `executable_template` | `templateMode = executable_template` | 固定报表、固定分析场景 | 是 |
| `reference_example` | `templateMode = reference` 或 `trusted_reference` | 相似业务参考 | 否，只能参考 |

> 当前 UI 的“业务口径”不是 `executable_template`。现有代码默认把“业务口径”保存为 `assetKind = sql_template`、`templateLevel = L2`、`templateMode = anchored_template`。`executable_template` 应作为更严格的高级/导入型模板使用：只有参数 schema 完整、场景高度固定、dry-run 可通过、缺参处理明确时才允许启用。

缺外部数据不是当前 `templateMode` 的合法值，建议表达为：

```yaml
template_status: blocked_missing_source
business_signature:
  externalDependencies:
    - ad_spend
```

runtime 通过外部依赖检测将本次问数路由为 `blocked_missing_external_data`。

#### P0 最小模板 Profile

P0 不需要一次性完成全部高级字段，但每个可执行模板至少应具备：

```yaml
template_type: executable_template
required_slots:
  - tenant_plat_id
expected_grain: biz_date + channel_id
positive_scenarios:
  - 渠道日基础汇总
negative_scenarios:
  - 单玩家充值明细
  - 非渠道粒度汇总
```

这些字段可以先写入 Markdown front matter，通过导入器或 runtime 读取；不一定要立刻全部暴露在 UI 中。

P0 实现时需映射到当前代码字段：

| Markdown / 文档字段 | 当前代码字段 | 当前消费情况 |
| --- | --- | --- |
| `template_type` | `templateMode` | 已支持 `reference` / `trusted_reference` / `anchored_template` / `executable_template` |
| `required_slots` | `parameterSchema.required` | AI-service 已读取 `parameter_schema.required` 判断模板必填参数 |
| `expected_grain` | `businessSignature.expectedGrain` | UI 可保存；AI-service 当前尚未结构化消费，P0 不应依赖它做硬门槛 |
| `positive_scenarios` | `businessSignature.positiveCues` | AI-service 当前已用于模板加分 |
| `negative_scenarios` | `businessSignature.negativeCues` | AI-service 当前已用于模板降分 |
| `external_dependencies` | `businessSignature.externalDependencies` | AI-service 当前已用于外部依赖检测 |

## 7. 导入格式 v2 与兼容策略

当前 `docs/业务需求/knowledge-base/import-format.md` 使用 `import_format_version: v1`。本方案新增字段后，建议升级为 `v2`，但 runtime 和导入器必须兼容 v1。

### 7.1 v2 新增字段

SQL 模板建议新增：

```yaml
import_format_version: v2
template_type: executable_template
required_slots:
  - tenant_plat_id
expected_grain: biz_date + channel_id
positive_scenarios: []
negative_scenarios: []
business_signature:
  metrics: []
  dimensions: []
  parameter_slots: []
```

导入器必须把 v2 文档字段映射到当前已存在的 SQL pair / API 字段，避免文档字段和代码字段各自演进：

| v2 文档字段 | 当前 SQL pair / API 字段 | 备注 |
| --- | --- | --- |
| `template_type` | `templateMode` | `executable_template` 映射为 `executable_template`；`reference_example` 映射为 `reference` 或 `trusted_reference` |
| `required_slots` | `parameterSchema.required` | 当前 AI-service 已读取 `parameter_schema.required` 判断模板参数缺失 |
| `expected_grain` | `businessSignature.expectedGrain` | P0 只保存和展示，不作为硬门槛 |
| `positive_scenarios` | `businessSignature.positiveCues` | 当前 scoring 已消费 |
| `negative_scenarios` | `businessSignature.negativeCues` | 当前 scoring 已消费 |
| `external_dependencies` | `businessSignature.externalDependencies` | 当前外部依赖检测已消费 |
| `template_status` | `status` 或 `businessSignature.templateStatus` | 不可写入 `templateMode`；缺外部数据应转为 runtime route |

业务词典建议新增：

```yaml
import_format_version: v2
applicable_scenarios: []
not_applicable_scenarios: []
required_slots: []
```

外部数据依赖建议新增：

```yaml
import_format_version: v2
trigger_when: []
not_trigger_when: []
lifecycle: per_question
input_modes:
  - single_value
```

### 7.2 向后兼容规则

- v1 文档仍可导入。
- 缺少 `template_type` 时，默认映射为 `templateMode = reference`，不允许高置信硬套。
- 缺少 `required_slots` 时，runtime 仍应执行内置 `tenant_plat_id` 策略。
- 缺少 `expected_grain` 时，P0 不作为硬门槛；P1 引入 SemanticPlan 后再启用结构化 grain guard。
- 导入器应忽略未知字段，避免后续字段扩展导致导入失败。
- `runtime_sync` 继续作为运行态审计字段，不参与导入决策。

### 7.3 现有资产迁移规则

已有知识库资产不应在升级时被自动提升为硬模板：

- 现有 `reference` SQL pair 继续保持 `templateMode = reference`。
- 现有 `admin_marked` / `business_import` 且 `assetKind = sql_template` 的业务模板，可以默认视为 `anchored_template`。
- 不自动升级任何旧模板为 `executable_template`。
- 没有 `parameterSchema.required` 的旧模板，不能作为参数化硬模板直接套用。
- 缺少 `businessSignature.expectedGrain` 时视为未知，不作为 P0 阻断条件，也不作为高置信硬套依据。
- 如需提升为 `executable_template`，必须补齐 `parameterSchema.required`、明确适用/不适用场景，并通过 dry-run / 回归验证。

### 8.1 P0a：Runtime-only 最小修复

目标：不先大改 UI / DB 资产，优先解决 OQ06 / OQ08 / LING01。

P0a 只做五件事：

1. **诊断 OQ06 / OQ08 的实际模板决策**
   - 复现问题并保存 `template_decision`。
   - 查看 `decision_reason`、`fallback_reason`、`score`、`margin`、`template_mode`、`business_signature`。
   - 不能预设根因一定是阈值或 negative cues；需根据 trace 判断是检索分数、阈值、subject/grain 缺失、negative scenario 缺失还是上下文干扰。

2. **基于现有模板决策补最小通用 guard**
   - 优先在已有 `build_template_decision()` / scoring 基础上补 `required_slots`、`positiveCues`、`negativeCues`、`features`、`externalDependencies` 的最小通用判断。
   - `expectedGrain`、subject、metric、dimension 的结构化硬门槛需要可靠 SemanticPlan 支撑，放到 P1；P0 只能作为 diagnostics / 轻量降权依据。
   - 必要时再补模板 `negative_scenarios` 或调整 `WREN_TEMPLATE_*` 阈值。
   - 避免只针对 OQ06 / OQ08 写特例关键词。
   - 新 guard 应支持环境变量或配置开关快速关闭，避免回归发现“该答的不答”时无法回滚。
   - 不建议 P0 直接全局调高阈值；优先使用局部 guard、negative cues 和测试约束。

3. **内置 `tenant_plat_id` slot guard**
   - 可复用 `_extract_template_parameters_from_query()` 等已有提取结果，但不能只依赖它。
   - 原因：`_extract_template_parameters_from_query()` 只有在 placeholders 已知时才抽取；如果用户问题没有命中模板、模板未声明 `tenant_plat_id` placeholder，普通 text-to-SQL 路径仍会漏掉平台槽位。
   - P0 必须新增独立 deterministic `tenant_plat_id` extractor / slot guard，覆盖普通 text-to-SQL、未选中模板、模板降级为 reference 的场景。
   - 缺失时返回阻断式追问提示，不进入 SQL 生成。
   - P0 暂不做完整多轮 clarification resume。

4. **展示现有 diagnostics**
   - P0 先展示 `AskTemplateDecision` 已有字段：`mode`、`template_id`、`template_title`、`decision_reason`、`fallback_reason`、`missing_parameters`、`sql_source`。
   - 标准 `decision` contract 作为 P1 规范化目标，不强行要求 P0 一次完成。

5. **补最小自动测试**
   - AI-service 单元测试覆盖 OQ06 禁止误用 T01、OQ08 禁止误用 T03/T08、LING01 缺 `tenant_plat_id` 不生成 SQL。
   - `verify_tidb_regression.py` 当前是 SQL 结果校验脚本，不应直接假设 1 小时即可变成问数路由测试器；如要扩展，需要新增 ask API 调用与 route assertion 能力。
   - P0 试行通过后，再跑完整 `docs/业务需求/问数回归测试计划.md`，不能只跑新增的三条单测。

  - enhance 时保留 deterministic slot guard、missing slot、clarification request、resolved slots 和 candidate templates，避免 LLM 覆盖已确定的安全门槛。
- P2 Policy as Code：
  - 新增 `wren-ai-service/src/core/ask_policy.py`；
  - 支持 YAML / JSON policy 文件、policy version、forbidden templates、required slots 和独立单元测试；
  - 通过 `WREN_ASK_POLICY_FILE=/path/to/policy.yaml` 加载策略；
  - 运行时将 `policy_version`、`policy_reason_codes`、`policy_violations` 写入 `semantic_plan.decision` 和 `template_decision`；
  - 当策略命中 forbidden template 时，当前模板降级为 reference + generated，避免继续硬套。
- P3 产物化 smoke：
  - 新增 `docs/业务需求/verify_artifactization_smoke.py`；
  - 独立验证数据看板 item、数据表 spreadsheet、反馈、CSV/Excel 导出工具和相关 API；
  - 支持 `--dry-run`、Cookie / Authorization、workspace / knowledge base runtime scope 参数；
  - 与 ask route regression 分离，避免把“问数路由是否正确”和“结果产物化是否可用”混在一个脚本中。

本轮后仍属于产品化增强而非主链路阻塞的能力：

- P1 跨进程 / 跨服务持久 clarification session（当前仍以内存 TTL session 为主，足够覆盖本地回归，但服务重启后不会恢复）；
- P2 策略变更审计 UI、差异对比和一键回滚（当前 policy-as-code 已可版本化和单测，但尚未做管理页面）；
- P3 把产物化 smoke 纳入 CI 并绑定真实登录态 / 权限矩阵（当前 runner 支持 dry-run 和只读 HTTP 检查）。

## 10. 上下文继承、澄清状态、失败兜底与安全边界

### 10.1 上下文继承边界

`tenant_plat_id` 等高风险业务槽位可以从上下文继承，但必须满足：

- 仅限同一 workspace；
- 仅限同一 knowledge base；
- 仅限同一 thread；
- 如果上下文出现多个候选值，必须追问；
- 如果用户切换 workspace / KB，不允许继承；
- diagnostics 应展示继承来源；
- UI 应在回答中提示“已沿用上一轮租户平台 990001”。

负例测试必须覆盖：

- 不同 workspace 不继承 `tenant_plat_id`。
- 不同 knowledge base 不继承 `tenant_plat_id`。
- 新对话不继承旧对话的 `tenant_plat_id`。
- 用户显式输入新平台时，覆盖历史平台。
- 历史问题里有多个平台候选时，必须追问，不能猜一个。
- 与当前问题业务无关的历史问答，不得作为槽位来源。

### 10.2 多轮澄清状态机边界

完整多轮 clarification 不进入 P0，应作为 P1 独立专题设计。

P1 需要新增或明确：

```yaml
clarification_state:
  status: needs_clarification | clarification_provided | resuming | cancelled
  clarification_session_id: string
  original_question: string
  pending_slots:
    - tenant_plat_id
  resolved_slots:
    tenant_plat_id: "990001"
  expires_at: datetime
```

P0 的行为仅为：

- 缺关键槽位时返回追问提示；
- 不进入 SQL 生成；
- 不自动恢复原任务；
- 用户补充后需要重新发起或继续当前问题。
- 技术响应可复用当前 GENERAL / DATA_ASSISTANCE 结果形态；不新增 `needs_clarification` API status。

### 10.3 LLM 失败兜底

语义计划抽取不应完全依赖 LLM。建议：

- 关键槽位先用 deterministic extractor 识别，例如租户平台、渠道、玩家 ID、日期范围；
- LLM 语义计划失败时，不允许进入 `template_answer`；
- 可以降级为 `normal_text_to_sql` 或 `clarification_required`；
- LLM / provider 429 时，应提供可重试提示，不应误判为业务模板不匹配；
- diagnostics 记录 `llm_plan_unavailable`。

### 10.4 SQL 与外部数据安全边界

- 模板参数必须参数化绑定或经过严格转义。
- 用户补充外部数据必须做类型校验和范围校验。
- 外部数据 join key 不能由用户自由输入后直接拼 SQL。
- 用户不得补充原始 SQL 片段作为外部数据。
- SQL dry-run 失败时不能继续执行。
- 保存为数据表 / 外部数据资产时必须记录 workspace、创建人和来源问题。

### 10.5 MDL 对齐原则

业务词典、SQL 模板和外部依赖最终都应对齐当前知识库的 MDL / deployed snapshot：

- 业务词典中的 source tables / fields 应优先映射到 MDL model / column。
- 如果字段不在当前 KB snapshot 中，AI assistant 应提示不可用。
- SQL 模板采纳必须基于当前 deploy hash / kb snapshot。
- KB 切换后，不能复用旧 snapshot 的模板决策。

## 11. Policy as Code 与可测试性

策略引擎不应只存在于 prompt 中。新增的模板采纳 guard、slot guard、外部依赖 guard 应尽量以纯函数或声明式配置实现，并可独立单测。

### 11.1 P0 测试要求

P0 至少需要：

- `build_template_decision` 或其新增 guard 的单元测试；
- `tenant_plat_id` slot guard 单元测试；
- 外部依赖阻断与 slot 阻断的行为一致性测试；
- OQ06 / OQ08 / LING01 的最小 route assertion。

优先扩展现有 AI-service 测试，而不是先把 Markdown 回归脚本改成完整问数测试平台：

| 测试文件 | P0 适合新增的断言 |
| --- | --- |
| `wren-ai-service/tests/pytest/core/test_fixed_order_ask_runtime.py` | 模板 scoring / guard、`tenant_plat_id` extractor、OQ06 / OQ08 不误锚定 |
| `wren-ai-service/tests/pytest/services/test_deepagents_skill_execution_mode.py` | 问数执行模式下缺槽位、阻断提示、diagnostics 透传 |
| `docs/业务需求/verify_tidb_regression.py` | 继续承担 TiDB SQL 模板结果校验；若要测 route，需新增 ask API 调用与 route assertion，不建议作为 P0 主路径 |

### 11.2 P2 策略版本化方向

当前已提供最小 `Policy as Code` 骨架：`wren-ai-service/src/core/ask_policy.py` 可以加载 YAML / JSON 策略文件，并把 policy version / reason codes / violations 写回 diagnostics。策略文件路径通过 `WREN_ASK_POLICY_FILE` 指定。

策略文件示例：

```yaml
policy_id: semantic_governance
version: semantic_governance_v1
rules:
  - id: forbid_cohort_template_for_plain_recharge
    reason_code: policy_plain_recharge_template_guard
    when:
      query_contains_any:
        - 普通充值
    forbidden_templates:
      - T08
  - id: require_tenant_for_channel_metrics
    reason_code: policy_missing_tenant_for_channel_metric
    when:
      query_contains_any:
        - 渠道
    required_slots:
      - tenant_plat_id
```

这类策略已能被 pytest 独立加载和验证，避免每次只能依赖 Playwright 或人工回归。后续产品化重点是把策略编辑、变更记录、差异对比、影响用例和回滚能力暴露给高级管理员。

## 12. UI 产品设计

### 12.1 默认简单模式

业务用户默认只看到：

- 基础信息；
- 同义词；
- 定义；
- 计算口径；
- 关联字段；
- 适用场景；
- 缺失时怎么处理。

不显示：

- semantic type；
- route policy；
- join contract；
- trigger expression；
- gating rule。

### 12.2 AI 推荐区

显示为：

```text
AI 推荐的运行时设置

- 系统识别该业务词属于：指标
- 可能需要参数：租户平台、日期范围
- 适用模板：首存 cohort、首存金额分桶
- 不建议用于：渠道日报、单玩家充值明细

[采用建议] [编辑]
```

### 12.3 高级模式

仅管理员 / 高级用户可展开。

用于：

- 修改模板采纳策略；
- 配置外部数据 join key；
- 配置强制澄清规则；
- 配置 forbidden templates；
- 维护正反例。

## 13. 推荐落地优先级

### 第一优先级

1. P0a runtime-only 修复，先覆盖 `tenant_plat_id`。
2. 基于实际 trace 修复 OQ06 / OQ08 模板误锚定，不预设根因为阈值或 negative cues。
3. diagnostics 先展示现有 `AskTemplateDecision` 字段。
4. OQ06 / OQ08 / LING01 回归通过。

### 第二优先级

1. 业务词典增加适用 / 不适用场景。
2. 外部数据依赖增加触发 / 不触发场景。
3. SQL 模板增加模板类型与结果粒度。
4. 测试用例增加 `expected_route` / `forbidden_templates`。

### 第三优先级

1. AI assistant 自动生成结构化知识建议。
2. 高级模式配置策略。
3. 策略版本管理。
4. 自动化评估与回归报告。

## 14. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 字段过多，用户难理解 | 默认简单模式，高级字段隐藏 |
| AI 推荐不准 | 用户确认后生效，不自动覆盖 |
| 规则过强导致该答的不答 | diagnostics 展示阻断原因，可人工调策略 |
| 新 guard 引发大面积路由变化 | 通过环境变量 / 配置开关灰度启用，可快速回退旧门控 |
| 全局阈值调整影响不可控 | P0 不直接全局调高阈值，优先局部 guard / negative cues / tests |
| 模板过弱导致模板价值下降 | 模板分级：硬模板 / 参考模板并存 |
| 槽位策略过于通用导致频繁追问 | 支持上下文继承和默认值 |
| 外部数据依赖触发过度 | 增加 not_trigger_when |
| 测试成本上升 | 用例结构化后自动验证 |
| 上下文继承用错租户 / KB | 限制同 workspace / KB / thread，并在 diagnostics 展示来源 |
| LLM 抽取失败导致误硬套模板 | LLM 失败时禁止 `template_answer`，降级或追问 |
| 外部数据污染其他对话 | P0 默认 `per_question` 生命周期 |
| SQL 注入或非法拼接 | 参数化、类型校验、dry-run 阻断 |
| P0 同时修改 AI 服务和 UI 导致风险扩大 | P0a 先做 runtime + tests，P0c 再做 UI 简单模式 |
| 多轮澄清状态失控 | P0 不做 resume 状态机，P1 单独设计 clarification session |
| Policy 逻辑不可测 | 新增 guard 写成纯函数并补单元测试 |
| 小范围试行通过但全量回归失败 | P0 试行后必须执行 `docs/业务需求/问数回归测试计划.md` 全量回归并记录 route / SQL 来源 / fallback |

## 15. 最终结论

完整方案不是让用户手工维护复杂技术配置，而是：

```text
用户维护简单业务知识
  ↓
AI assistant 生成结构化建议
  ↓
管理员确认或微调
  ↓
系统内部用语义计划和策略引擎稳定执行
```

短期 P0 解决当前回归问题：

- OQ06 / OQ08 不再误用模板；
- LING01 缺 `tenant_plat_id` 先追问；
- 模板类问题仍能稳定命中；
- 缺外部数据时不编造。

中期 P1 / P2 降低维护成本并增强治理能力。

长期 P3 让新增业务需求、泛化用例和未来更多问数场景可以持续扩展，而不是依赖不断堆特例规则。
