# 数据表 AI 操作回归用例

日期：2026-04-28

## 目标

覆盖 Spreadsheet 固化第二阶段的真实业务回归风险：

1. AI Filter / Cleaning / Grouping / Enrichment 不改变原业务口径。
2. 结构化 Filter 优先走确定性规则，减少 LLM 不稳定性。
3. 非结构化操作可安全 fallback 到 AI SQL rewrite，并通过 dry-run 后才保存版本。
4. History 中可追溯 `previousSql`、`generatedSql`、`operationType`、`generationMode`。

## 测试前置条件

- 只在 TiDB workspace 中执行。
- 使用 `docs/业务需求/问数回归测试计划.md` 已导入的 SQL 模板、分析规则、业务词典、外部数据依赖。
- 至少已有一条问数结果可保存为数据表。
- 数据表详情页可看到：Show SQL、Update columns、History、资源设置、AI Assistant。

## Golden Spreadsheet

建议从以下问题生成问数结果并保存为数据表：

```text
查询租户平台990001下渠道990011在2026-04-01到2026-04-07每天登录过的去重玩家数
```

保存后记录：

| 字段 | 记录 |
|---|---|
| Spreadsheet ID | 手工填写 |
| 原始 SQL | History v1 / Show SQL |
| 原始行数 | Preview rowCount |
| 原始核心过滤 | tenant/platform/channel/date |

## 回归用例

### F-01 结构化等值筛选

操作：Filter

```text
只保留 biz_date 为 2026-04-05 的记录
```

期望：

- `generationMode = structured`。
- SQL 外层增加 `WHERE "biz_date" = '2026-04-05'`。
- 原 SQL 中租户、平台、渠道等核心过滤不丢失。
- dry-run 通过并保存为新版本。
- 预览数据仅包含 `2026-04-05`。

### F-02 结构化日期范围 + 多条件

操作：Filter

```text
biz_date 从 2026-04-01 到 2026-04-07 且 channel_id 为 990011 且 tenant_id 为 990001
```

期望：

- `generationMode = structured`。
- SQL 外层包含 `BETWEEN` 和多个 `AND` 条件。
- 数值字段不被错误加字符串引号。
- dry-run 通过并保存为新版本。

### F-03 结构化 IN / 不等于 / 空值 / 包含

操作：Filter

```text
channel_id 为 990011、990012 且 status 不等于 test 且 deleted_at 为空 且 user_name 包含 vip
```

期望：

- `generationMode = structured`。
- SQL 外层包含 `IN (...)`、`<>`、`IS NULL`、`LIKE '%vip%'`。
- 不支持字段导致 dry-run 失败时，不保存版本，并提示错误。

### F-04 非结构化异常筛选 fallback

操作：Filter

```text
帮我找出这段时间登录行为异常波动的日期
```

期望：

- 结构化解析失败，fallback 到 AI rewrite。
- `generationMode = ai`。
- 生成 SQL dry-run 通过后才保存。
- 若 AI SQL 不可执行，则不保存版本。

### C-01 Cleaning 不改变业务过滤

操作：Cleaning

```text
把空渠道名称统一标记为未知渠道，并确保日期字段格式一致
```

期望：

- `generationMode = ai`。
- 原始租户、平台、渠道、日期范围过滤不丢失。
- 输出列中能看到清洗后的渠道名称或等价字段。
- 统计行数不因清洗操作异常减少。

### G-01 Grouping 调整粒度

操作：Grouping

```text
按渠道汇总 2026-04-01 到 2026-04-07 的去重登录玩家数
```

期望：

- `generationMode = ai`。
- SQL 改为渠道粒度聚合。
- 指标仍为去重玩家数，不变成登录次数。
- 日期和租户/平台过滤保留。

### E-01 Enrichment 补充派生字段

操作：Enrichment

```text
增加一列，标记登录日期是工作日还是周末
```

期望：

- `generationMode = ai`。
- 输出列新增工作日/周末标签或等价字段。
- 原始指标和过滤条件保留。
- dry-run 通过后保存新版本。

## 验收标准

- F-01 / F-02 必须稳定通过，不依赖 LLM。
- F-03 对存在字段应通过；对不存在字段必须安全失败且不保存版本。
- F-04 / C-01 / G-01 / E-01 至少完成 dry-run 成功路径和失败不保存路径验证。
- 每次操作后 History 版本递增，payload 可追踪：
  - `operationType`
  - `instruction`
  - `generationMode`
  - `previousVersion`
  - `previousSql`
  - `queryId`（AI rewrite 场景）

## 体验记录

回归时同步记录：

- AI 操作按钮是否容易理解。
- AI 操作弹窗说明是否足够明确。
- 失败提示是否能说明 SQL 生成失败、dry-run 失败或权限失败。
- History 是否足够支撑撤销/追溯。
- 数据表列表是否位于右侧主工作区，不再占用左侧历史导航。
