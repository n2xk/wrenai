> `qa_orders 表一共有多少条记录？总金额是多少？`

结果：

- API：`POST /api/v1/stream/ask`

- 自然语言 ask 没有完成最终 SQL 执行和回答返回。
- 失败点在 LLM provider 限流，不是 connector、metadata、资产导入或 SQL 执行链路。
- 建议后续使用 BYOK / 切换可用 provider / 等限流恢复后重跑自然语言 ask 和图表生成。

## 6. 本轮发现与修复

### 6.1 DuckDB 激活知识库运行时失败

初次执行 DuckDB 资产导入时，连接测试与表探测均通过，但激活知识库运行时失败：

```text
Unsupported data source: DUCKDB
```

根因：

- `ConnectionMetadataService.listTables()` 对 DuckDB 有专门路径，会先 `prepareDuckDB` 再由 Wren Engine `listTables()`。
- `ConnectionMetadataService.getVersion()` 原先没有 DuckDB 分支，直接走 Ibis `getVersion()`。
- Ibis adaptor 不支持 `DUCKDB` 的 version endpoint，因此激活 runtime 时失败。

修复：

- `wren-ui/src/server/services/metadataService.ts`
  - `getVersion()` 对 DuckDB 先 prepare runtime，并返回 `DuckDB runtime`。
- `wren-ui/src/server/services/tests/metadataService.test.ts`
  - 增加 DuckDB version 分支回归测试。

验证：

```bash
cd wren-ui
yarn test src/server/services/tests/metadataService.test.ts
yarn check-types
```

结果：均通过。

### 6.2 Trino 历史坏 catalog 干扰

初次 Trino 连接测试失败：

```text
TrinoExternalError ... FATAL: database "wren_source_alpha_smoke" does not exist
```

原因：Trino 容器中存在历史 smoke catalog 文件，指向已不存在的 PostgreSQL database，导致 Ibis/Trino 枚举时被坏 catalog 干扰。

本轮处理：

1. 新增 `docker/.trino/catalog/external_qa_pg.properties` 指向本轮 PostgreSQL QA 数据源。
2. 补齐历史 smoke catalog 依赖的 `wren_source_alpha_smoke` / `wren_source_beta_smoke` 测试库，避免 Trino catalog 枚举报错。
3. 重启 `wrenai-local-trino-1` 后重测通过。

建议：后续测试前应清理或隔离 Trino catalog 目录，避免历史 catalog 影响 provider 测试。

### 6.3 ClickHouse Decimal 类型识别

ClickHouse `qa_orders.amount Decimal(12,2)` 在表探测中返回：

```text
amount: UNKNOWN
```

但模型导入、部署和数据预览均通过，预览中 `amount` 以 string 返回。

建议：后续将 ClickHouse Decimal 类型映射补充为 `DECIMAL` 或统一数值类型，降低语义建模和问数生成风险。
