# 真实外部数据源连接验证测试方案

> 日期：2026-04-29  
> 适用范围：Wren UI 新版「数据连接器 / 知识库添加资产」链路  
> 目标：验证新 connector provider 已覆盖旧 `/setup` connection 能力，并能在真实外部数据源上完成连接测试、表探测、资产导入与基础问数。

## 1. 背景

当前系统已将旧 `/setup` 数据源接入流程迁移到：

- 系统设置 → 数据连接器
- 知识库 → 添加资产向导 → 新建 / 选择数据库连接器

新版 connector 需要覆盖旧 connection 支持的 12 类数据源：

1. PostgreSQL
2. MySQL / TiDB
3. BigQuery
4. DuckDB
5. Oracle
6. SQL Server / MSSQL
7. ClickHouse
8. Athena
9. Snowflake
10. Redshift
11. Trino
12. Databricks

本测试文档用于补齐“真实外部数据源连接验证”，避免只通过单元测试和 mock API 证明功能正确。

## 2. 测试目标

### 2.1 必须验证

每个 provider 至少验证：

1. 可以在「系统设置 → 数据连接器」中创建连接器。
2. 表单字段与该 provider 的真实连接参数匹配。
3. 「连接测试」可以成功返回表数量与样例表。
4. 可以在「知识库 → 添加资产」中选择该 database connector。
5. 可以探测表列表。
6. 可以导入至少 1 张表为知识库资产。
7. 导入后可以在建模 / 资产列表中看到该资产。
8. 对导入资产执行一个基础问数，能生成 SQL 并返回数据。

### 2.2 建议验证

如果数据源具备测试数据，进一步验证：

1. 字段类型识别是否正确。
2. schema / catalog / database 过滤是否正确。
3. 密钥编辑后是否可继续测试连接。
4. 删除 connector 后知识库资产向导不会继续展示不可用连接。
5. 旧 connection bridge 迁移后的 connector 配置是否可读、可编辑。

## 3. 测试前置条件

### 3.1 服务要求

本地服务需要启动：

```bash
# UI：3002
cd wren-ui
PORT=3002 PG_URL=postgres://postgres:postgres@127.0.0.1:9432/wrenai TZ=UTC yarn dev

# AI service：5555
cd wren-ai-service
poetry run python -m src.__main__
```

Docker 依赖需要包含：

- PostgreSQL / Wren metadata DB
- Wren Engine
- Ibis Server
- 需要的向量库 / pgvector 依赖

### 3.2 真实数据源 Docker 启动要求

本计划中的待测数据源默认全部通过 Docker / Docker Compose 启动，不直接依赖开发者本机已安装的数据库服务。每个 provider 应准备独立的 compose service 或 compose profile，并在测试记录中写明：

- compose 文件或 profile 名称
- 容器名称
- 暴露端口
- 初始化脚本 / seed 数据位置
- 测试完成后的清理命令

为避免本地资源占用、端口冲突和跨数据源残留状态干扰，**同一时间 Docker 中最多只保留 3 个待测数据源存在**。这里的“待测数据源”指 provider 数据源容器或数据源容器组，不包含 Wren metadata DB、Wren Engine、Ibis Server、pgvector 等系统运行依赖。

执行规则：

1. 按批次启动数据源，每批最多 3 个 provider。
2. 当前批次测试完成并记录结果后，先执行对应 `docker compose down` / `docker rm` / volume 清理策略，再启动下一批。
3. 若某个 provider 需要多个容器组成测试环境，也按 1 个 provider 数据源组计数。
4. 如确实存在无法 Docker 化的云托管 provider 验证，应在测试记录中明确标记为“云托管例外”，并说明为什么不能使用 Docker；但默认测试方案仍以 Docker 启动为准。

### 3.3 测试账号与权限

测试账号需要具备：

- workspace 管理权限
- connector 创建 / 编辑 / 删除权限
- knowledge base 资产导入权限
- 问数权限

建议使用非系统样例空间，例如：

- Workspace：`TiDB workspace` 或专门的 `External datasource QA workspace`
- Knowledge Base：每轮测试新建或使用独立测试知识库

### 3.4 安全要求

真实数据源测试必须遵守：

1. 不提交真实密码、token、private key、service account JSON 到仓库。
2. 测试凭证只放在本地 `.env`、密码管理器或安全测试环境中。
3. 测试数据源优先使用只读账号。
4. 测试 SQL 不执行写入、删除、DDL 变更，DuckDB 初始化 SQL 例外但仅限测试库。
5. 测试记录中只写脱敏连接信息。

### 3.5 云数仓 / Emulator 测试原则

BigQuery、Snowflake、Redshift、Athena、Databricks 这类云数仓或云服务 provider 不能简单等同于“Docker 启动一个本地数据库”：

1. **真实云验收是最终准入标准**：需要使用真实云账号、真实数据集 / warehouse / cluster / workgroup / SQL Warehouse、只读凭证完成连接测试、表探测、资产导入、问数和预览。
2. **本地 emulator smoke 只做早期冒烟**：可以用于验证表单字段、payload 转换、Ibis / SDK 基础调用路径和错误提示，但不能替代真实云验收。
3. **协议兼容 smoke 不算 provider 验收**：例如用 PostgreSQL 冒充 Redshift，只能证明 host/port/password 这类基础字段能传递，不能证明 Redshift SQL 方言、系统表、schema/catalog 元数据和权限模型正确。
4. **Emulator 需要产品支持 endpoint override**：若 emulator 不是官方服务 endpoint，前端表单、connector config、UI → Ibis payload、Ibis server 连接层都需要支持高级 endpoint / host override 字段；否则 UI 里填了 emulator 参数也无法真正连到本地 emulator。
5. **默认隐藏 emulator 字段**：endpoint override 属于测试 / 高级诊断能力，不应在普通用户的新建连接器表单中直接暴露；建议放在 feature flag 或“高级设置 / 测试模式”后面。

当前建议分层如下：

| Provider | 真实云验收 | 本地 emulator smoke | 本地 emulator 结论 |
| --- | --- | --- | --- |
| BigQuery | 必须 | 可选，候选：第三方 BigQuery emulator | 需要 BigQuery API endpoint / client options override；当前 UI 与 Ibis 连接层未提供。 |
| Snowflake | 必须 | 可选，候选：LocalStack Snowflake emulator | 需要 Snowflake host / account URL override；当前 UI 与 Ibis 连接层未提供。 |
| Redshift | 必须 | 不建议作为等价 emulator | PostgreSQL 兼容 smoke 不能证明 Redshift；当前本地 Ibis server 也未暴露 redshift connector endpoint。 |
| Athena | 必须 | 可选，候选：LocalStack Athena + S3 + Glue | 需要 AWS endpoint_url / S3 / Glue / Athena endpoint override；当前 UI 与 Ibis 连接层未提供。 |
| Databricks | 必须 | 暂不建议 | 未找到可替代 SQL Warehouse 的等价 Docker emulator；需要真实 Databricks workspace / SQL Warehouse。 |

## 5. 通用测试流程

对每个 provider 重复执行以下流程。

### 5.1 创建连接器

1. 打开 `http://127.0.0.1:3002`。
2. 登录测试账号。
3. 进入目标 workspace。
4. 打开：系统设置 → 数据连接器。
5. 点击「添加连接器」。
6. 类型选择「数据库」。
7. Provider 选择当前测试 provider。
8. 填写连接参数。
9. 点击「连接测试」。
10. 记录返回信息：
    - 是否成功
    - 表数量
    - sampleTables
    - version
    - 错误信息，如失败
11. 保存连接器。

验收标准：

- 连接测试成功。
- 保存后 connector 出现在列表中。
- provider、名称、作用域显示正确。

### 5.2 知识库添加资产

1. 进入：知识库。
2. 选择目标知识库。
3. 点击「添加资产」。
4. 数据源选择已保存的 database connector。
5. 确认列表中不会出现 REST API / Python 工具等非 database connector。
6. 等待表列表加载。
7. 选择 1-3 张测试表。
8. 导入资产。
9. 确认资产列表出现导入表。

验收标准：

- 只展示 database connector。
- 表列表可加载。
- 导入资产成功。
- 不出现 provider 不支持、metadata 获取失败、schema 过滤错误等异常。

### 5.3 建模与问数

1. 进入知识库建模区域。
2. 检查导入表、字段、类型是否合理。
3. 如测试数据简单，可手动补充字段描述。
4. 发起一个基础问题，例如：
   - “这张表一共有多少条记录？”
   - “按日期统计订单数。”
   - “统计金额总和和平均金额。”
5. 观察问数结果：
   - SQL 是否使用正确表名 / schema / catalog。
   - SQL 是否执行成功。
   - 结果是否有数据。
   - 是否可以查看 SQL。
   - 是否可以导出 CSV / Excel，如当前功能要求覆盖。

验收标准：

- 能生成 SQL。
- SQL 可以执行。
- 返回结果与数据源直接查询结果一致或可解释。
- 无 socket hang up、metadata not found、table not found 等异常。

## 6. Provider 专项检查点

### 6.1 DuckDB

重点验证：

- `initSql` 是否在连接测试前执行。
- `extensions` 是否会被转换为 `INSTALL extension;`。
- `configurations` 是否会作为 DuckDB session props 设置。
- 表探测是否基于最新 runtime，而不是旧 DuckDB session 残留。

建议测试配置：

```json
{
  "initSql": "CREATE TABLE orders AS SELECT 1 AS id, 'paid' AS status, 100 AS amount;",
  "extensions": [],
  "configurations": {}
}
```

预期：

- 连接测试发现 `orders` 表。
- 问“orders 表有多少条记录”返回 1。

### 6.2 Oracle

重点验证：

- Host / Port / Database 模式。
- DSN 模式。
- DSN 作为 secret 保存后编辑不会泄露。

至少通过一种连接方式。

### 6.3 MSSQL

重点验证：

- `trustServerCertificate=true` 时是否可连接自签名证书测试库。
- 表名 schema 是否正确，例如 `dbo.orders`。

### 6.4 ClickHouse

重点验证：

- SSL 开关是否影响连接 URL。
- database 过滤是否正确。
- 返回字段类型是否合理。

### 6.5 Athena

重点验证：

- AWS Access Key 模式。
- OIDC Web Identity Token 模式，如测试环境支持。
- Instance Profile 模式，如服务运行环境支持。
- `s3StagingDir` 是否可写。
- schema/database 过滤是否正确。

### 6.6 Databricks

重点验证：

- PAT token 模式。
- Service principal 模式，如测试环境支持。
- SQL Warehouse `serverHostname` 与 `httpPath` 是否正确。
- catalog/schema 表名是否被正确识别。

### 6.7 Trino

重点验证：

- `schemas` 格式必须为 `catalog.schema`。
- 多 schema 用英文逗号分隔。
- 表探测是否只返回配置 schema 下的表。

示例：

```text
hive.default,tpch.sf1
```

## 7. 云数仓 / Emulator 补充测试流程

### 7.1 代码能力检查

在启动 emulator 之前，先检查当前代码是否具备 endpoint override 能力，否则不应直接进入 UI 回归：

```bash
python3 - <<'PY'
from pathlib import Path
files = [
    'wren-ui/src/server/adaptors/ibisAdaptorTypes.ts',
    'wren-ui/src/features/settings/connectors/connectorsPayloadUtils.ts',
    'wren-ui/src/server/dataSource.ts',
    'wren-engine/ibis-server/app/model/__init__.py',
    'wren-engine/ibis-server/app/model/data_source.py',
]
needles = [
    'apiEndpoint',
    'endpointUrl',
    'endpoint_url',
    'clientOptions',
    'client_options',
    'api_endpoint',
    'aws_endpoint',
    'snowflakeHost',
    'hostOverride',
    'serviceEndpoint',
]
for f in files:
    txt = Path(f).read_text(errors='ignore')
    hits = [n for n in needles if n in txt]
    print(f'{f}: endpoint override hits={hits or "none"}')
PY
```

验收判断：

- 如果 BigQuery / Snowflake / Athena 仍没有 endpoint override 字段，则本地 emulator 只能记录为“代码能力未满足”，不能继续声明完成 provider smoke。
- 如果 Redshift / Athena / Databricks 在 Ibis server `DataSource` enum 中不存在，则 UI 侧即使能保存 connector，metadata / query 也会在 Ibis endpoint 层失败，应按 Blocker 记录。

### 7.2 Ibis endpoint 支持检查

当前 UI metadata 默认调用 Ibis v2 endpoint。可用下面命令确认某个 provider 是否被当前 Ibis server 暴露：

```bash
for ds in athena redshift databricks; do
  echo "--- $ds"
  curl -sS -o "/tmp/wren_${ds}.json" -w "%{http_code}\n" \
    -X POST "http://127.0.0.1:8000/v2/connector/${ds}/metadata/tables" \
    -H 'Content-Type: application/json' \
    --data '{"connectionInfo":{}}'
  cat "/tmp/wren_${ds}.json"
  echo
done
```

验收判断：

- 返回 `422` 且错误中提示 path `data_source` enum 不包含该 provider：当前 Ibis server 未支持该 provider，不能进入 UI 级真实 provider 测试。
- 返回 provider 字段校验错误或真实连接错误：说明 endpoint 已暴露，可以继续准备凭证 / emulator / seed 数据。

### 7.3 Emulator smoke 候选

| Provider | 候选方案 | 必要产品能力 | Smoke 覆盖范围 | 不覆盖内容 |
| --- | --- | --- | --- | --- |
| BigQuery | 第三方 BigQuery emulator | BigQuery API endpoint / client options override；测试 service account 或 emulator 兼容认证 | 表单、payload、Ibis BigQuery client 调用、metadata 基础路径 | 真实 BigQuery 权限、slot、标准 SQL 兼容性、生产网络认证 |
| Snowflake | LocalStack Snowflake emulator | Snowflake host/account URL override；测试账号映射 | 表单、payload、Snowflake driver 基础连接、metadata 基础路径 | 真实 Snowflake warehouse、role、权限、性能、完整 SQL 方言 |
| Athena | LocalStack Athena + S3 + Glue | AWS endpoint_url override；S3/Glue/Athena endpoint 一致配置；staging dir | 表单、payload、AWS SDK / Athena 基础路径 | 真实 AWS IAM、Glue Catalog 权限、S3 策略、Athena 执行引擎差异 |
| Redshift | 不建议用 emulator 判定通过；可选 PostgreSQL 协议 smoke | Ibis redshift endpoint 支持；或单独 PostgreSQL 协议 smoke 标记 | 只验证 host/port/user/password 基础字段传递 | Redshift 系统表、方言、IAM、Serverless、性能 |
| Databricks | 无等价本地 emulator | 真实 SQL Warehouse hostname/httpPath/token | 不做本地 smoke | Databricks provider 全部关键能力 |

### 7.4 2026-04-30 本轮补充检查结果

本轮按上面的“代码能力检查”和“Ibis endpoint 支持检查”执行，结论如下：

| 检查项 | 结果 | 证据 |
| --- | --- | --- |
| BigQuery / Snowflake / Athena emulator 所需 endpoint override | 未满足 | `wren-ui/src/server/adaptors/ibisAdaptorTypes.ts`、`wren-ui/src/features/settings/connectors/connectorsPayloadUtils.ts`、`wren-ui/src/server/dataSource.ts`、`wren-engine/ibis-server/app/model/__init__.py`、`wren-engine/ibis-server/app/model/data_source.py` 均未检出 `apiEndpoint` / `endpointUrl` / `endpoint_url` / `client_options` 等字段。 |
| 当前本地 Ibis server DataSource enum | 不包含 Athena / Redshift / Databricks | `wren-engine/ibis-server/app/model/data_source.py` 当前枚举为 `bigquery/canner/clickhouse/mssql/mysql/oracle/postgres/snowflake/trino/local_file/s3_file/minio_file/gcs_file`。 |
| Athena endpoint smoke | 未支持 | `POST /v2/connector/athena/metadata/tables` 返回 `422`，path `data_source` enum 不包含 `athena`。 |
| Redshift endpoint smoke | 未支持 | `POST /v2/connector/redshift/metadata/tables` 返回 `422`，path `data_source` enum 不包含 `redshift`。 |
| Databricks endpoint smoke | 未支持 | `POST /v2/connector/databricks/metadata/tables` 返回 `422`，path `data_source` enum 不包含 `databricks`。 |
| BigQuery / Snowflake endpoint | endpoint 存在，但不能做 emulator smoke | `POST /v2/connector/bigquery|snowflake/metadata/tables` 不返回 enum 422；但当前连接层只能走官方 endpoint 与真实凭证，无法指向本地 emulator。 |
| 相关单元测试 | 通过 | `cd wren-ui && yarn test src/server/utils/tests/connectorDatabaseProvider.test.ts src/server/adaptors/tests/ibisAdaptor.test.ts --runInBand`，2 个 test suites / 35 tests 通过。 |

因此，本轮不应把 BigQuery / Snowflake / Redshift / Athena / Databricks 标记为“Docker 已验证通过”。正确状态是：

- BigQuery / Snowflake：等待 endpoint override 能力后可做 emulator smoke；最终仍要真实云验收。
- Redshift / Athena / Databricks：先补齐 Ibis server provider endpoint / connector 支持，再做 UI 级测试；Databricks 直接进入真实云验收，不做本地 emulator。

## 8. 失败分级与处理建议

| 级别 | 定义 | 示例 | 处理 |
| --- | --- | --- | --- |
| Blocker | provider 无法保存或连接测试必失败 | 必填字段缺失、后端不识别 provider | 必须修复后重测 |
| Critical | 能保存但不能导入资产 / 表探测失败 | listTables 报错、schema 过滤错误 | 必须修复后重测 |
| Major | 能导入但问数执行失败 | SQL table not found、字段名错误 | 根据 provider 优先级修复 |
| Minor | UI 文案 / 样式 / 错误提示问题 | 报错不友好、字段提示不清楚 | 记录体验优化 |

## 10. 最终验收标准

本轮真实外部数据源连接验证通过的最低标准：

1. P0 provider 全部通过：PostgreSQL、MySQL/TiDB、DuckDB。
2. P1 provider 至少完成真实连接测试和表探测：BigQuery、Snowflake、Redshift、Trino、ClickHouse、MSSQL。
3. P2 provider 至少完成一套可用凭证路径验证：Oracle、Athena、Databricks。
4. 知识库添加资产向导只展示 database connector。
5. `/setup/*` 访问会兼容跳转到新版知识库工作台，不再进入旧 setup 流程。
6. 失败项都有明确错误信息、日志和 owner。
7. 云数仓 provider 的本地 emulator smoke 只能作为补充证据，不能替代真实云验收。
8. 若 UI provider 已展示但 Ibis server endpoint 不支持，应按 Blocker 处理，不得标记为有条件通过。

## 11. 建议执行顺序

建议按风险和可获得环境分批执行。每批最多启动并保留 3 个 Docker 待测数据源，批次结束后先清理本批数据源，再进入下一批：

1. 第一批 P0：PostgreSQL、MySQL/TiDB、DuckDB。
2. 第二批 P1：ClickHouse、MSSQL、Trino。
3. 第三批 Ibis 支持补齐检查：确认 Redshift、Athena、Databricks 是否已在 Ibis server endpoint 暴露；未暴露则先修复，不进入 UI 回归。
4. 第四批 emulator smoke：BigQuery、Snowflake、Athena（仅当 endpoint override 能力已补齐）。
5. 第五批真实云验收：BigQuery、Snowflake、Redshift、Athena、Databricks。
6. 第六批 P2 Docker 数据库：Oracle。

每批完成后先修 blocker / critical，再进入下一批，避免多个 provider 同时失败导致定位困难。
