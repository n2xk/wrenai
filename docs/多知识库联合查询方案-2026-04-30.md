# 多知识库联合查询方案

> 日期：2026-04-30  
> 状态：方案草案  
> 目标：把当前“多知识库知识检索增强”升级为真正的“跨知识库数据表联合查询”。  
> 适用范围：Wren UI / Apollo API / Wren AI Service / Wren Engine / Trino federation runtime。

## 1. 背景与结论

用户对“多知识库查询”的直觉理解是：

> 选择多个知识库后，系统能把多个知识库中的数据表放进同一个可执行查询上下文，并支持跨知识库 JOIN / UNION / 聚合。

这和当前代码里的能力不完全一致。

当前系统已经具备“多知识库选择”的 UI 和部分后端链路，但实际语义更接近：

> 第一个知识库作为主执行范围，其他知识库只参与 SQL 模板、分析规则、历史问题等知识检索增强。

因此，当前能力不能直接作为“真正多知识库联合查询”的验收通过标准。真正的多知识库联合查询需要新增一个 **Composite / Federated Query Runtime**，在一次 ask 中合并多个知识库的资产、语义层、关系与业务知识，并由一个可执行的联邦查询 runtime 执行 SQL。

这说明当前多知识库主要影响知识检索，不影响 SQL 生成所需的完整 DB schema 检索，也不生成一个跨知识库可执行语义层。

这部分可以作为真正多知识库联合查询的基础，但它当前只作用于 **单个知识库内部**，不是跨知识库组合。

## 3. 产品定义

### 3.1 新定义

**多知识库联合查询** 指：

> 用户在一次对话中选择多个知识库后，系统创建一个联合查询上下文，把这些知识库的可执行资产、语义层、关系、SQL 模板、分析规则、业务词典和外部数据依赖组合起来，用统一的联邦 runtime 生成并执行 SQL。

必须支持：

1. 跨知识库表 JOIN。
2. 跨知识库表 UNION / 聚合。
3. 多知识库业务口径共同生效。
4. 多知识库 SQL 模板 / 分析规则 / 业务词典参与生成。
5. 表名冲突时可解释、可消歧。
6. 查询结果可继续追问、生成图表、固定到数据看板、保存到数据表、导出 CSV / Excel。

### 3.2 当前旧能力命名建议

为了避免误导，当前已存在的能力建议命名为：

- “引用多个知识库”
- “多知识库业务知识增强”
- “从多个知识库检索业务知识”

不要把当前能力直接命名为“多知识库联合查询”。

## 4. 设计原则

1. **执行上下文必须真实合并**  
   多知识库联合查询不能只合并 retrieval scope，必须生成一个可执行的 composite runtime。

2. **权限先行**  
   用户必须对所有被选知识库都有读权限，且第一阶段限定在同一个 workspace 内。

3. **命名空间显式化**  
   每个知识库 / connector / table 必须有稳定 namespace，避免同名表随机命中。

4. **跨知识库关系需要结构化配置**  
   不允许 LLM 凭字段名相似性随意 JOIN。可以允许“候选关系推荐”，但生产查询应优先使用已确认关系。

5. **可缓存但必须可失效**  
   联合 runtime 可按知识库组合和 snapshot/deploy hash 缓存；任一成员知识库重新部署、connector 更新或关系更新后必须失效。

6. **先支持数据库型知识库**  
   第一阶段只支持可被 Trino federation 的 database connector。REST JSON / Python Tool / 文件类能力后续单独设计。

## 5. 目标架构

### 5.1 核心概念：Composite Query Runtime

新增一个逻辑对象：`CompositeQueryRuntime`。

它表示一次多知识库联合查询可执行上下文，包含：

| 字段 | 含义 |
| --- | --- |
| `id` | composite runtime id |
| `workspaceId` | 所属 workspace |
| `knowledgeBaseIds` | 参与联合查询的知识库 id，有序保存 |
| `memberDeployHashes` | 每个知识库当前使用的 deploy hash |
| `runtimeProjectId` | 内部 Trino runtime project id |
| `deployHash` | composite runtime 自身的 deploy hash |
| `manifestHash` | 合并后 MDL 的 hash |
| `status` | preparing / ready / failed / stale |
| `error` | 准备失败原因 |

第一阶段建议把 composite runtime 做成 **可缓存的短生命周期运行时**，而不是用户显式管理的知识库。

### 5.2 Runtime 解析流程

当用户选择多个知识库并发起 ask：

1. UI 发送：

```json
{
  "question": "按客户地区统计 GMV",
  "knowledgeBaseIds": ["kb-orders", "kb-customers"],
  "queryMode": "federated_knowledge_bases"
}
```

2. UI API 校验：
   - 所有知识库属于同一个 workspace。
   - 用户对所有知识库都有 read 权限。
   - 每个知识库都有最新可执行 snapshot / deploy。

3. `CompositeQueryRuntimeService.resolve()`：
   - 加载每个知识库的 latest executable snapshot。
   - 加载每个知识库的 MDL / models / relations / views。
   - 加载每个知识库绑定的 connector 和 Trino catalog。
   - 生成或复用 composite Trino runtime project。
   - 生成合并后的 MDL。
   - 部署 composite runtime。

4. 创建 ask task：

```json
{
  "runtime_scope_id": "composite-deploy-hash",
  "runtime_identity": {
    "workspaceId": "ws-1",
    "knowledgeBaseId": null,
    "deployHash": "composite-deploy-hash"
  },
  "retrieval_scope_ids": [
    "composite-deploy-hash",
    "kb-orders-deploy-hash",
    "kb-customers-deploy-hash"
  ],
  "knowledgeBaseIds": ["kb-orders", "kb-customers"]
}
```

5. AI service：
   - `db_schema_retrieval` 使用 `composite-deploy-hash`。
   - SQL templates / instructions / business terms 使用成员知识库 deploy hash + composite deploy hash。
   - SQL 生成只基于 composite schema 中的 namespace 表名。

6. SQL 执行：
   - 通过 composite Trino runtime project 执行。

## 6. MDL 合并与命名空间

### 6.1 模型命名

为避免不同知识库存在同名表，合并后的模型名必须稳定且可读。

建议内部模型名：

```text
<kb_slug>__<model_name>
```

示例：

```text
orders_kb__orders
customer_kb__customers
```

同时保留 display metadata：

```json
{
  "originKnowledgeBaseId": "kb-orders",
  "originKnowledgeBaseName": "订单知识库",
  "originModelName": "orders",
  "displayName": "订单知识库.orders"
}
```

### 6.2 物理表定位

合并后的模型需要指向 Trino catalog/schema/table：

```json
{
  "catalog": "kb_xxx_connector_yyy",
  "schema": "public",
  "table": "orders"
}
```

SQL 生成阶段应优先使用 MDL model name，最终由 Wren Engine / model substitution 转换到 Trino 可执行 SQL。

### 6.3 同名字段与语义冲突

不同知识库中的字段描述、业务词典、分析规则可能冲突。处理策略：

1. 成员知识库内规则只默认作用于该知识库来源的模型。
2. 跨知识库规则必须显式配置适用范围。
3. 如果用户问题无法判断使用哪个知识库的口径，应返回澄清问题，而不是随机选择。

## 7. 跨知识库关系设计

### 7.1 新增关系类型

新增结构化“跨知识库关系”：

| 字段 | 示例 |
| --- | --- |
| `workspaceId` | `ws-1` |
| `sourceKnowledgeBaseId` | `kb-orders` |
| `sourceModelId` | `orders` |
| `sourceColumnName` | `customer_id` |
| `targetKnowledgeBaseId` | `kb-customers` |
| `targetModelId` | `customers` |
| `targetColumnName` | `customer_id` |
| `relationshipType` | `many_to_one` |
| `joinType` | `left` / `inner` |
| `description` | `订单.customer_id 关联客户.customer_id` |
| `confidence` | `manual_confirmed` / `ai_suggested` |
| `enabled` | true |

### 7.2 页面入口

建议放在：

- 知识库工作台 → 关系 → 跨知识库关系
- 或 系统设置 → 业务知识配置中心 → 跨知识库关系

第一阶段只允许手动确认关系；AI 推荐关系作为辅助，不自动生效。

### 7.3 生成逻辑

合并 MDL 时：

1. 导入成员知识库内部已有关系。
2. 导入启用状态的跨知识库关系。
3. 把关系中的 model id 映射到合并后的 namespaced model。
4. 写入 composite manifest。

## 8. API 与服务改造

### 8.1 新增服务

建议新增：

```text
wren-ui/src/server/services/compositeQueryRuntimeService.ts
```

职责：

1. 校验多知识库选择。
2. 计算 composite runtime cache key。
3. 创建 / 复用 composite runtime project。
4. 调用 Trino catalog ensure。
5. 合并 MDL。
6. 触发 deploy。
7. 返回 composite runtime identity。

### 8.2 复用现有 federation 能力

可复用：

- `FederatedRuntimeProjectService` 的 catalog 生成思路。
- `buildTrinoCatalogProperties()`。
- `generateTrinoCatalogName()`，但需要扩展为支持 composite member source。
- `extractFederatedSchemaBindings()`。
- `deployService.deploy()`。
- `mdlService.makeCurrentModelMDLByRuntimeIdentity()` 的 MDL 生成能力。

需要新增 / 调整：

1. `FederatedRuntimeProjectService` 当前按单个 `knowledgeBaseId` 工作，需要抽出 reusable helper。
2. composite runtime 不应写回某一个知识库的 `runtimeProjectId`。
3. composite deploy log 需要能被 runtimeIdentity 找到，建议新增 `runtimeKind = composite_knowledge_bases` 或单独表。

### 8.3 Ask task 创建逻辑

当前：

- `knowledgeBaseIds[0]` 决定 runtime。
- 多个 knowledgeBaseIds 只变成 retrieval scopes。

目标：

- `knowledgeBaseIds.length === 1`：保持现有单知识库逻辑。
- `knowledgeBaseIds.length > 1 && queryMode !== 'federated_knowledge_bases'`：保持旧“多知识库知识增强”兼容模式。
- `knowledgeBaseIds.length > 1 && queryMode === 'federated_knowledge_bases'`：走 composite runtime。

建议新增请求字段：

```ts
type KnowledgeQueryMode =
  | 'single_knowledge_base'
  | 'knowledge_retrieval_enhancement'
  | 'federated_knowledge_bases';
```

## 9. AI Service 改造

### 9.1 Schema retrieval 必须使用 composite runtime

当前测试明确验证 `db_schema_retrieval` 只使用 primary runtime。

真正多知识库联合查询下，必须改为：

- `runtime_scope_id = composite deploy hash`
- `db_schema_retrieval` 检索 composite schema。
- `sql_pairs_retrieval` / `instructions_retrieval` / `historical_question` 检索成员知识库 + composite scope。

### 9.2 SQL 生成提示词需要增加约束

必须加入：

1. 使用 namespaced model。
2. 不要引用未出现在 composite schema 中的表。
3. 跨知识库 JOIN 必须基于明确关系。
4. 表名冲突时优先返回澄清问题。
5. 如果问题只涉及一个知识库，可只查询该知识库模型，不强制跨库 JOIN。

### 9.3 思考步骤展示

建议新增 thinking steps：

- `ask.federated_runtime_resolved`
  - 显示参与知识库数量、主数据源 / catalogs。
- `ask.cross_kb_models_selected`
  - 显示命中的知识库和模型。
- `ask.cross_kb_relations_applied`
  - 显示使用的跨知识库关系数量。
- `ask.cross_kb_ambiguity_detected`
  - 需要澄清时展示。

## 10. UI 改造

### 10.1 新对话 composer

当用户选择多个知识库时，显示模式选择：

1. **联合查询**：跨知识库表生成 SQL，可 JOIN / 聚合。
2. **仅引用知识**：当前旧模式，只引用其他知识库的业务知识。

默认建议：

- 如果选择多个知识库，默认进入“联合查询”。
- 但如果任一知识库不支持 federation，提示用户只能“仅引用知识”。

### 10.2 查询结果页

展示：

- 本次使用的知识库列表。
- 本次使用的 composite runtime / deploy hash。
- SQL 中涉及的知识库 / 表。
- 如果固定到看板 / 保存数据表，需要保存 composite runtime identity。

### 10.3 错误提示

典型错误：

| 场景 | 提示 |
| --- | --- |
| 选中的知识库没有可执行快照 | `知识库「X」尚未部署，无法参与联合查询` |
| connector 不支持 Trino federation | `知识库「X」的数据源暂不支持联合查询，可切换为“仅引用知识”模式` |
| 表名冲突无法消歧 | `多个知识库中都存在 orders，请指定要查询的知识库或配置业务关系` |
| 缺少跨知识库关系 | `需要关联订单与客户，但尚未配置跨知识库关系` |
| 权限不足 | `你没有访问知识库「X」的权限` |

## 11. 数据模型建议

### 11.1 `composite_query_runtime`

```sql
CREATE TABLE composite_query_runtime (
  id uuid PRIMARY KEY,
  workspace_id uuid NOT NULL,
  cache_key text NOT NULL UNIQUE,
  runtime_project_id bigint NOT NULL,
  deploy_hash text,
  manifest_hash text,
  status text NOT NULL,
  error text,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

### 11.2 `composite_query_runtime_member`

```sql
CREATE TABLE composite_query_runtime_member (
  id bigserial PRIMARY KEY,
  runtime_id uuid NOT NULL REFERENCES composite_query_runtime(id),
  knowledge_base_id uuid NOT NULL,
  kb_snapshot_id uuid,
  deploy_hash text NOT NULL,
  ordinal int NOT NULL
);
```

### 11.3 `cross_knowledge_relation`

```sql
CREATE TABLE cross_knowledge_relation (
  id uuid PRIMARY KEY,
  workspace_id uuid NOT NULL,
  source_knowledge_base_id uuid NOT NULL,
  source_model_id text NOT NULL,
  source_column_name text NOT NULL,
  target_knowledge_base_id uuid NOT NULL,
  target_model_id text NOT NULL,
  target_column_name text NOT NULL,
  relationship_type text NOT NULL,
  join_type text NOT NULL DEFAULT 'left',
  description text,
  confidence text NOT NULL DEFAULT 'manual_confirmed',
  enabled boolean NOT NULL DEFAULT true,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

## 12. 分阶段实施计划

### Phase 0：边界校正

目标：避免当前能力继续被误称为真正多知识库联合查询。

工作：

1. UI 文案区分：
   - “联合查询”
   - “仅引用知识”
2. 测试计划注明当前旧能力边界。
3. 增加文档说明：当前 `knowledgeBaseIds` 不等于跨知识库 JOIN。

验收：

- 用户不会把“引用多个知识库”误认为可跨表 JOIN。

### Phase 1：Composite Runtime 基础设施

工作：

1. 新增 `CompositeQueryRuntimeService`。
2. 新增 composite runtime 表。
3. 实现多知识库 latest deploy / snapshot 加载。
4. 实现 runtime cache key。
5. 基于 Trino 创建 composite runtime project。
6. 合并基础 MDL models。

验收：

- 两个知识库各有一张表时，可以生成一个 composite runtime。
- composite schema retrieval 可以同时看到两张表。

### Phase 2：跨知识库关系与 SQL 生成

工作：

1. 新增跨知识库关系表和 UI。
2. 合并 MDL relations。
3. AI prompt / pipeline 支持跨 KB relation。
4. 表名冲突消歧。
5. thinking steps 展示。

验收：

- 可基于关系生成跨知识库 JOIN SQL。
- 缺少关系时能澄清或失败明确。

### Phase 3：结果资产化

工作：

1. 图表生成支持 composite runtime。
2. 固定到看板保存 composite runtime identity。
3. 保存到数据表保存 composite runtime identity。
4. 导出 CSV / Excel 支持 composite query result。
5. 看板缓存刷新使用 composite runtime。

验收：

- 联合查询结果可以图表化、固定、保存、导出，并可刷新。

### Phase 4：完整回归与体验优化

工作：

1. MCP Playwright 全链路回归。
2. 权限 / 错误提示 / loading / stale runtime 体验优化。
3. 大知识库组合性能压测。

### 13.1 测试数据准备

#### 知识库 B：客户知识库 `KB Customers`

| customer_id | customer_name | region |
| --- | --- | --- |

#### 跨知识库关系

## 14. 验收标准

真正多知识库联合查询可以上线的最低标准：

1. 两个知识库的两张表可以跨知识库 JOIN。
2. SQL 生成使用明确关系，不随机关联。
3. 多知识库业务规则可以影响生成结果。
4. 表名冲突有稳定 namespace 或澄清机制。
5. 查询结果可查看 SQL、生成图表、固定看板、保存数据表、导出 CSV / Excel。
6. 追问、刷新、历史对话恢复后仍保持联合查询上下文。
7. 权限不足、unsupported connector、缺少关系、runtime stale 都有明确提示。
8. 单知识库问数和旧“仅引用知识”模式不回归。

## 15. 主要风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| Composite runtime 创建慢 | 首次提问等待时间长 | cache runtime；创建时显示 preparing；后台预热常用组合。 |
| 表名 / 业务口径冲突 | SQL 错误或结果误导 | 强制 namespace；冲突时澄清；规则限定适用范围。 |
| 跨知识库关系缺失 | LLM 随机 JOIN | 没有确认关系时禁止自动 JOIN，先推荐关系或提示配置。 |
| Connector federation 支持不足 | 部分知识库不能联合查询 | 第一阶段只支持 Trino 可 federation provider，其他降级为仅引用知识。 |
| 权限与审计复杂 | 数据泄露风险 | 所有成员知识库逐一权限校验；审计记录完整 knowledgeBaseIds。 |
| 看板缓存失效复杂 | 固定资产刷新失败 | 固定资产保存 composite runtime key 和 member deploy hashes，任一变化后重建。 |

## 16. Trino 联邦数据源覆盖现状

当前系统的 database connector provider 覆盖范围和 Trino 自动联邦覆盖范围并不一致。实现真正多知识库联合查询时，不能默认认为“UI 支持的数据源都可以参与联邦查询”。

### 16.1 代码现状

`wren-ui/src/server/utils/connectorDatabaseProvider.ts` 中，数据库连接器支持 12 类 provider：

```ts
export const DATABASE_CONNECTOR_PROVIDERS = [
  'postgres',
  'mysql',
  'bigquery',
  'duckdb',
  'oracle',
  'mssql',
  'clickhouse',
  'athena',
  'snowflake',
  'redshift',
  'trino',
  'databricks',
] as const;
```

但当前自动 Trino federation 只启用了 5 类：

```ts
const AUTO_FEDERATABLE_PROVIDERS = new Set<DatabaseConnectorProvider>([
  'postgres',
  'mysql',
  'bigquery',
  'snowflake',
  'redshift',
]);
```

`buildTrinoCatalogProperties()` 当前只实现：

- PostgreSQL → `connector.name=postgresql`
- MySQL / TiDB → `connector.name=mysql`
- BigQuery → `connector.name=bigquery`
- Snowflake → `connector.name=snowflake`
- Redshift → `connector.name=redshift`

并且存在认证能力限制：

- Snowflake：当前联邦不支持仅 privateKey 认证。
- Redshift：当前联邦不支持 IAM 鉴权。
- Trino：代码中明确 `自动联邦暂未实现 Trino-to-Trino catalog 映射`。

### 16.2 本地 Trino 插件可用性

本地 `wrenai-local-trino-1` 容器的 `/usr/lib/trino/plugin` 中可以看到以下相关插件：

- `postgresql`
- `mysql`
- `bigquery`
- `snowflake`
- `redshift`
- `oracle`
- `sqlserver`
- `clickhouse`
- `duckdb`

这说明 Oracle / MSSQL / ClickHouse / DuckDB 不是 Trino 插件完全缺失，而是当前 Wren UI 的 catalog properties 构建逻辑尚未接入。

### 16.3 覆盖矩阵

| 数据源 | UI Connector 支持 | Trino 插件存在 | 当前自动联邦接入 | 说明 |
| --- | --- | --- | --- | --- |
| PostgreSQL | ✅ | ✅ | ✅ | 已接入。 |
| MySQL / TiDB | ✅ | ✅ | ✅ | TiDB 走 MySQL connector。 |
| BigQuery | ✅ | ✅ | ✅ | 已接入，但依赖 Trino BigQuery 配置。 |
| Snowflake | ✅ | ✅ | ⚠️ 部分 | 只支持 password；privateKey 不支持。 |
| Redshift | ✅ | ✅ | ⚠️ 部分 | 只支持 password；IAM 不支持。 |
| DuckDB | ✅ | ✅ | ❌ | Trino 插件存在，但当前未生成 catalog。 |
| Oracle | ✅ | ✅ | ❌ | Trino 插件存在，但当前未生成 catalog。 |
| MSSQL / SQL Server | ✅ | ✅ | ❌ | Trino 插件存在，但当前未生成 catalog。 |

### 16.4 对多知识库联合查询的影响

真正多知识库联合查询依赖 composite Trino runtime。如果某个知识库的数据源不能被自动联邦，则该知识库不能参与“联合查询”模式，只能参与“仅引用知识”模式。

因此，UI 在用户选择多个知识库时需要做能力判断：

1. 所有成员知识库都可 federation：允许“联合查询”。
2. 部分知识库不可 federation：
   - 禁用“联合查询”；或
   - 显示不可参与的知识库和原因；或
   - 允许用户切换为“仅引用知识”。
3. 同一知识库内有多个 connector 时，只有全部参与查询的 connector 可 federation，才能生成完整 composite runtime。

### 16.5 补齐优先级

建议按下面顺序补齐 Trino federation provider：

#### P0：本地 Docker 可验证且 Trino 插件已存在

1. MSSQL / SQL Server
2. Oracle
3. ClickHouse

理由：

- Trino 插件已经存在。
- 当前 UI connector 字段已经存在。
- 可以通过 Docker provider 测试方案验证。
- 对企业数据源覆盖价值高。

#### P1：需要处理 runtime 生命周期或文件语义

4. DuckDB

理由：

- Trino 插件存在。
- 但当前 DuckDB connector 的核心能力依赖 `initSql`、extensions、configurations 和 runtime prepare。
- 需要明确 Trino DuckDB catalog 如何承载 initSql / 文件路径 / session 生命周期。

#### P2：补齐已有 provider 的认证模式

5. Snowflake privateKey
6. Redshift IAM

理由：

- provider 已经接入。
- 但认证模式未覆盖完整。
- 需要确认 Trino connector 对 privateKey / IAM 的配置支持与密钥保存方式。

#### P3：单独设计，不建议简单补 catalog

7. Athena
8. Databricks
9. Trino-to-Trino

理由：

- Athena 更接近 AWS Athena / Glue / S3 组合能力，需要单独 endpoint、staging、catalog 权限设计。
- Databricks SQL Warehouse 没有当前直接 Trino connector；可能应通过 Delta Lake / Iceberg / Hive Metastore 进入 federation，而不是直接连 SQL Warehouse。
- Trino-to-Trino federation 可能造成 catalog/schema 转发、权限和查询下推复杂度，需要单独设计。

### 16.6 补齐后的测试要求

每补齐一个 provider，都必须新增以下测试：

1. `buildTrinoCatalogProperties()` 单元测试：验证 catalog properties 字段正确。
2. `canAutoFederateConnector()` 单元测试：验证该 provider 在完整配置下返回 true，在缺少密钥 / 不支持认证模式下返回 false。
3. Docker 级连接验证：启动对应数据源容器，Trino catalog 能 `SHOW TABLES`。
4. 知识库级 federation 验证：知识库导入该 provider 表后，能够生成 federated runtime project。
5. 多知识库联合查询 E2E：该 provider 作为成员知识库之一参与跨知识库 JOIN。
6. 降级体验测试：不支持的认证模式或 provider 在 UI 中明确提示，不允许误进入“联合查询”。

### 16.7 文档与 UI 文案要求

在多知识库联合查询正式上线前，UI 和文档必须明确：

- “支持数据源连接”不等于“支持联邦联合查询”。
- “仅引用知识”可以使用不可 federation 的知识库参与业务知识检索。
- “联合查询”只支持可生成 Trino composite runtime 的知识库。
- 不支持原因要展示到具体 provider / connector / authentication mode，而不是笼统提示失败。
