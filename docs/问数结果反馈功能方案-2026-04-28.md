# 问数结果反馈功能方案：Was this result helpful?

日期：2026-04-28

## 1. 背景

当前问数结果卡片底部已经展示：

```text
Was this result helpful?   👍   👎
```

但现有实现只是在前端本地状态中切换选中态，刷新页面后反馈会丢失，也不会进入后端分析、质量治理或回归测试闭环。

现有相关代码：

- `wren-ui/src/components/pages/home/promptThread/AnswerResult.tsx`
  - 渲染 `Was this result helpful?`
  - 点击后只更新本地 `helpfulFeedback` state。
- `wren-ui/src/features/home/thread/threadWorkbenchMessages.ts`
  - 存放 `helpfulPrompt`、`helpfulPositive`、`helpfulNegative` 文案。

本方案目标是把该入口升级为可持久化、可分析、可治理的问数结果反馈系统。

## 2. 目标

### 2.1 用户体验目标

用户可以对每条问数结果反馈：

- 有帮助
- 没有帮助

负反馈时可以进一步选择原因和填写补充说明。

### 2.2 产品分析目标

系统可以统计：

- 哪些问题回答质量高。
- 哪些知识库、SQL 模板、分析规则容易产生负反馈。
- 哪类失败原因最多。
- 哪些真实用户问题应该沉淀为回归测试用例。

### 2.3 治理闭环目标

负反馈可以进入治理流程：

- 创建待修复样本。
- 转成回归测试用例。
- 辅助优化 SQL 模板。
- 辅助优化分析规则。
- 辅助发现业务词典或外部数据依赖缺口。

## 3. 产品交互设计

### 3.1 默认状态

建议中文化展示为：

```text
这个结果有帮助吗？   有帮助   没有帮助
```

如果继续保留英文，也可以沿用：

```text
Was this result helpful?   Helpful   Not helpful
```

### 3.2 点击“有帮助”

交互：

1. 按钮高亮。
2. 直接提交反馈。
3. 展示轻提示：

```text
感谢反馈
```

不建议在 MVP 中对正反馈弹窗，避免打断用户。

### 3.3 点击“没有帮助”

建议弹出轻量 Popover 或 Drawer：

```text
Provide additional feedback

* What went wrong?

Answer
[ ] SQL generation failed
[ ] Incorrect data retrieved
[ ] Incorrect AI summary
[ ] Failed to adhere instructions
[ ] Failed to adhere summary instructions
[ ] Failed to adhere SQL-Pairs
[ ] Other

How was your experience?
________________

[提交反馈]
```

说明：

- 问题类型使用上图这套选项，便于和现有线上反馈口径对齐。
- 如果界面需要中文化，可以在展示层翻译，但后端 `reasonCodes` 仍使用稳定英文枚举。
- 补充说明不是必填，但建议保留，用于收集复现上下文和用户期望。

提交后：

- 负反馈按钮高亮。
- 展示：

```text
已记录反馈
```

### 3.4 修改和撤销

用户应可以修改反馈：

- 再次点击当前选中项：撤销反馈。
- 从“有帮助”切到“没有帮助”：覆盖原反馈。
- 从“没有帮助”切到“有帮助”：覆盖原反馈。

一条 response 对同一个用户只允许存在一条反馈。

## 4. 数据模型设计

新增表：`thread_response_feedback`

```ts
thread_response_feedback {
  id
  thread_response_id
  thread_id

  project_id
  workspace_id
  knowledge_base_id
  kb_snapshot_id
  deploy_hash
  actor_user_id

  rating              // positive / negative
  reason_codes        // jsonb string[]
  comment             // text
  source              // result_footer / regression_test / api
  metadata            // jsonb

  created_at
  updated_at
}
```

### 4.1 唯一约束

建议：

```sql
unique(thread_response_id, actor_user_id)
```

含义：

- 同一用户对同一条问数结果只保留一条反馈。
- 后续再次提交时走 upsert。

如果后续支持匿名用户，可以扩展为：

```sql
unique(thread_response_id, actor_user_id, client_session_id)
```

### 4.2 runtime scope 字段

需要保存：

- `project_id`
- `workspace_id`
- `knowledge_base_id`
- `kb_snapshot_id`
- `deploy_hash`

原因：

- 支持 workspace / knowledge base 隔离。
- 支持后续按知识库统计质量。
- 支持回归测试和问题治理时定位环境。

```ts

  FAILED_TO_ADHERE_SQL_PAIRS = 'failed_to_adhere_sql_pairs',
  OTHER = 'other',
}
```

展示文案建议先对齐截图英文：

```ts
{
  sql_generation_failed: 'SQL generation failed',
  incorrect_data_retrieved: 'Incorrect data retrieved',
  incorrect_ai_summary: 'Incorrect AI summary',
  failed_to_adhere_instructions: 'Failed to adhere instructions',
  failed_to_adhere_summary_instructions: 'Failed to adhere summary instructions',
  failed_to_adhere_sql_pairs: 'Failed to adhere SQL-Pairs',
  other: 'Other',
}
```

如果后续需要中文化，可在 i18n 层映射为：

```ts
{
  sql_generation_failed: 'SQL 生成失败',
  incorrect_data_retrieved: '查询出的数据不正确',
  incorrect_ai_summary: 'AI 总结不正确',
  failed_to_adhere_instructions: '没有遵循分析规则/业务指令',
  failed_to_adhere_summary_instructions: '没有遵循总结指令',

```

## 6. metadata 设计

反馈保存时建议同步保存当时回答的关键快照，避免后续分析时再拼复杂上下文。

```ts
metadata: {
  question,
  sql,
  responseKind,
  answerStatus,
  chartStatus,
  askTaskType,
  templateDecision: {
    mode,
    templateId,
    templateTitle,
    templateLevel,
    templateMode,
    sqlSource,
    fallbackReason,
    score,
  },
  traceId,
  sourceResponseId,
  hasChart,
  hasPreview,
}
```

关键点：

- `templateDecision` 可用于分析 SQL 模板/分析规则是否可靠。
- `traceId` 可用于追踪 AI service 侧日志。
- `sql` 和 `question` 快照可用于转回归测试。

## 7. API 设计

建议使用 REST，与当前 `thread-responses` 相关 API 保持一致。

### 7.1 获取当前反馈

```http
GET /api/v1/thread-responses/:id/feedback
```

返回：

```ts
{
  feedback: {
    id: number;
    rating: 'positive' | 'negative';
    reasonCodes: string[];
    comment?: string | null;
    createdAt: string;
    updatedAt: string;
  } | null;
}
```

### 7.2 创建或更新反馈

```http
PUT /api/v1/thread-responses/:id/feedback
```

body：

```ts
{
  rating: 'positive' | 'negative';
  reasonCodes?: string[];
  comment?: string;
}
```

行为：

- 如果不存在反馈，则创建。
- 如果已存在反馈，则覆盖。
- 正反馈允许不传 `reasonCodes`。
- 负反馈建议至少传一个 `reasonCode`，但 MVP 可先允许为空，避免用户被强阻断。

### 7.3 撤销反馈

```http
DELETE /api/v1/thread-responses/:id/feedback
```

返回：

```ts
{
  success: true
}
```

## 8. 服务端实现方案

### 8.1 新增文件

建议新增：

```text
wren-ui/src/pages/api/v1/thread-responses/[id]/feedback.ts
wren-ui/src/server/repositories/threadResponseFeedbackRepository.ts
wren-ui/src/server/services/threadResponseFeedbackService.ts
wren-ui/src/server/controllers/threadResponseFeedbackController.ts
```

如果想先做轻量 MVP，也可以先把 controller 逻辑放在现有 thread response 相关 controller 附近，但长期建议独立。

### 8.2 权限与 scope 校验

保存反馈前必须校验 response 属于当前 runtime scope。

建议复用现有逻辑：

```ts
ctx.askingService.assertResponseScope(responseId, runtimeIdentity)
```

然后再读取：

```ts
ctx.askingService.getResponseScoped(responseId, runtimeIdentity)
```

避免跨 workspace / knowledge base 写入反馈。

### 8.3 Upsert 流程

伪代码：

```ts
await askingService.assertResponseScope(responseId, runtimeIdentity);

const response = await askingService.getResponseScoped(
  responseId,
  runtimeIdentity,
);

const feedback = await feedbackRepository.upsert({
  threadResponseId: response.id,
  threadId: response.threadId,
  projectId: response.projectId ?? null,
  workspaceId: response.workspaceId ?? null,
  knowledgeBaseId: response.knowledgeBaseId ?? null,
  kbSnapshotId: response.kbSnapshotId ?? null,
  deployHash: response.deployHash ?? null,
  actorUserId,
  rating,
  reasonCodes,
  comment,
  source: 'result_footer',
  metadata: buildFeedbackMetadata(response),
});
```

## 9. 前端实现方案

### 9.1 替换本地 state

当前 `AnswerResult.tsx` 中有：

```ts
const [helpfulFeedback, setHelpfulFeedback] = useState<
  'positive' | 'negative' | null
>(null);
```

应改为从服务端状态驱动：

```ts
const {
  feedback,
  submitting,
  submitPositiveFeedback,
  openNegativeFeedback,
  deleteFeedback,
} = useThreadResponseFeedback(threadResponse.id);
```

### 9.2 新增 hook

新增：

```text
wren-ui/src/hooks/useThreadResponseFeedback.ts
```

职责：

- 拉取当前 response 的反馈。
- 提交正反馈。
- 提交负反馈。
- 撤销反馈。
- 乐观更新 UI。
- 失败时回滚并提示错误。

### 9.3 新增 REST 工具

新增：

```text
wren-ui/src/utils/threadResponseFeedbackRest.ts
```

方法：

```ts
getThreadResponseFeedback(selector, responseId)
upsertThreadResponseFeedback(selector, responseId, payload)
deleteThreadResponseFeedback(selector, responseId)
```

所有请求都需要携带 runtime scope query，与当前 home/thread API 保持一致。

### 9.4 负反馈组件

新增：

```text
wren-ui/src/components/pages/home/promptThread/ResponseFeedbackPopover.tsx
```

职责：

- 展示负反馈原因。
- 支持多选。
- 支持补充说明。
- 提交时调用 hook。

## 10. 埋点和分析

即使反馈会落库，仍建议保留前端 telemetry。

事件：

```ts
response_feedback_submitted
response_feedback_deleted
```

属性：

```ts
{
  responseId,
  threadId,
  workspaceId,
  knowledgeBaseId,
  rating,
  reasonCodes,
  responseKind,
  askTaskType,
  templateId,
  templateMode,
  sqlSource,
}
```

注意：telemetry 只是辅助分析，不能替代业务落库。

## 11. 和 SQL 模板 / 分析规则的联动

### 11.1 正反馈

如果回答命中了 SQL 模板或分析规则，用户点“有帮助”，可以记录为正向信号：

- 模板使用成功次数 +1。
- 分析规则使用成功次数 +1。

但不建议因为一次正反馈就自动升级为高置信模板。

### 11.2 负反馈

如果回答命中了 SQL 模板或分析规则，用户点“没有帮助”，并选择：

- SQL generation failed
- Incorrect data retrieved
- Failed to adhere instructions
- Failed to adhere SQL-Pairs

则应记录为治理信号：

- 模板待复核。
- 分析规则待复核。
- 该问题可加入回归测试候选池。

另外：

- `Incorrect AI summary` 主要进入回答总结/解释质量治理。
- `Failed to adhere summary instructions` 主要进入总结指令治理。
- `Other` 需要依赖补充说明做人工归因，不建议直接自动归入模板或分析规则问题。

## 12. 回归测试沉淀

后续可以增加“加入回归测试”动作，将负反馈转换为测试样本：

```ts
{
  question,
  actualSql,
  expectedSql,
  feedbackReasonCodes,
  expectedResultNote,
  workspaceId,
  knowledgeBaseId,
  sourceFeedbackId,
}
```

这可以与 `docs/业务需求/问数回归测试计划.md` 的真实问数用例形成闭环。

## 13. 是否接入 AI service

MVP 不建议直接接 AI service 自动修复。

原因：

- 负反馈首先应该稳定记录和归因。
- 自动修复 SQL 可能引入不可控变化。
- 需要先积累样本，建立人工/回归验证流程。

后续可以扩展：

- 负反馈后推荐“重新生成”。
- 根据反馈原因生成修复建议。
- 将反馈样本输入评测集。

## 14. 测试方案

### 14.1 Repository / Service 单元测试

覆盖：

- 创建正反馈。
- 创建负反馈。
- 正反馈更新为负反馈。
- 负反馈更新为正反馈。
- 撤销反馈。
- 同一用户同一 response 只能有一条反馈。

### 14.2 API 测试

新增：

```text
wren-ui/src/tests/api/thread_response_feedback_api.test.ts
```

覆盖：

- `GET` 无反馈返回 null。
- `PUT` 正反馈成功。
- `PUT` 负反馈带原因成功。
- `PUT` 非法 rating 返回 400。
- `DELETE` 成功。
- 跨 scope response 不允许写入。

### 14.3 前端测试

覆盖：

- 初始无反馈。
- 点击“有帮助”后高亮。
- 点击“没有帮助”弹出原因选择。
- 负反馈提交后高亮。
- 再次点击撤销。
- 提交失败回滚 UI。

### 14.4 E2E 测试

用 MCP Playwright 验证：

1. 打开一条已有问数结果。
2. 点击“有帮助”。
3. 刷新页面，状态仍然保持。
4. 改成“没有帮助”。
5. 选择原因并提交。
6. 再次刷新页面，负反馈仍然存在。

## 15. 分阶段实施

### Phase 1：MVP

目标：让入口真正可用。

范围：

1. 新增反馈表。
2. 新增 repository/service/API。
3. 前端接入 API。
4. 正反馈直接提交。
5. 负反馈支持原因和备注。
6. 刷新后状态保持。
7. 单测/API 测试/E2E 验证。

### Phase 2：反馈管理和统计

范围：

1. 新增反馈列表页。
2. 按 workspace / knowledge base / 模板 / 分析规则聚合。
3. 支持查看原线程。
4. 支持一键加入回归测试候选。

### Phase 3：质量治理闭环

范围：

1. SQL 模板正负反馈统计。
2. 分析规则正负反馈统计。
3. 低质量问题自动进入治理队列。
4. 与回归测试计划联动。

## 16. 验收标准

- 用户点击正反馈后刷新页面仍保持。
- 用户点击负反馈后可以选择原因并保存。
- 同一用户修改反馈时不会产生重复记录。
- 跨 workspace / knowledge base 不能写入反馈。
- 反馈记录包含 response、thread、runtime scope 和必要 metadata。
- 单测、API 测试、前端测试通过。
- MCP Playwright 能完成端到端验证。

## 17. 结论

`Was this result helpful?` 不应该只是一个前端选中态或埋点按钮，而应该成为问数质量反馈入口。

建议先完成 Phase 1：反馈持久化、负反馈原因、刷新保持和基础测试；后续再扩展反馈管理、SQL 模板/分析规则质量统计和回归测试沉淀。
