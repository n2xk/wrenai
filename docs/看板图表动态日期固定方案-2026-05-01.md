# 看板图表动态日期固定方案

日期：2026-05-01  
状态：方案设计  
范围：`wren-ui`

## 1. 背景

当前问数生成图表后，用户可以把图表固定到看板。现有体验里还有一个“调整图表”按钮，但它实际调整的是图表展示样式，例如图表类型、颜色、轴字段等，并不能调整问数条件或日期维度。

真实业务场景中，用户更常见的需求是：

> 我问了 04-03 到 04-07 的数据，并生成了图表。  
> 当我把这个图表固定到看板后，希望它以后定时刷新时，日期范围可以随着现实时间自动向后滚动，而不是永远固定查询 04-03 到 04-07。

因此，本方案选择“方案 B”：**固定到看板时保存一份可解释、可编译的查询控制配置，而不是只保存一次性生成出来的静态 SQL。**

## 2. 目标

### 2.1 产品目标

1. 固定到看板时，用户可以明确选择日期范围的行为：
   - 固定当前日期范围；
   - 随现实时间滚动。
2. 对于有日期条件的图表，看板定时刷新时可以自动更新日期范围。
3. 对于没有日期条件的图表，保持当前快速固定体验，不增加额外操作成本。
4. 暂时隐藏“调整图表样式”功能，避免用户误以为它可以调整查询维度或日期逻辑。

### 2.2 技术目标

1. dashboard item 继续保留原始 SQL，保证可回溯。
2. dashboard item 新增可选的 `queryControls` 配置，用于描述日期滚动规则。
3. 看板预览和看板缓存刷新统一走同一套 SQL 编译逻辑。
4. 第一阶段只支持安全、可识别的日期范围，不做全 SQL 语义重写。

## 3. 当前代码分析

### 3.1 固定到看板入口

当前图表固定入口主要在：

- `wren-ui/src/components/pages/home/promptThread/ChartAnswer.tsx`
- `wren-ui/src/components/pages/home/promptThread/ChartAnswerPinPopover.tsx`
- `wren-ui/src/components/pages/home/promptThread/ChartAnswerPinModal.tsx`

现有流程：

```text
点击固定到看板
→ 打开 ChartAnswerPinPopover
→ 选择已有看板，或点击“新建看板并固定”
→ ChartAnswer.tsx 调用 createDashboardItem
```

`ChartAnswer.tsx` 中当前提交参数只包含：

```ts
createDashboardItem(responseRuntimeSelector, {
  itemType,
  responseId: threadResponse.id,
  dashboardId,
});
```

也就是说，当前固定动作没有携带任何“动态日期”“查询控制”“维度调整”信息。

### 3.2 现有 popover 的职责

`ChartAnswerPinPopover.tsx` 当前是轻量弹层，主要负责：

1. 展示可固定的看板列表；
2. 标识默认看板；
3. 提供“新建看板并固定”入口。

这个 popover 很适合作为**目标看板选择器**，但不适合承载复杂配置，例如日期模式、滚动窗口、预览范围等。

### 3.3 新建看板弹窗

`ChartAnswerPinModal.tsx` 当前只做一件事：

```text
输入新看板名称
→ 新建看板
→ 立即固定当前图表
```

后续可以复用或扩展这个 modal，让“新建看板并固定”也支持动态日期配置。

### 3.4 现有图表样式调整入口

图表样式调整按钮在：

- `wren-ui/src/components/chart/index.tsx`
- `wren-ui/src/components/pages/home/promptThread/ChartAnswer.tsx`

`ChartAnswer.tsx` 当前向 `<Chart />` 传入：

```tsx
onEdit={onEdit}
```

`wren-ui/src/components/chart/index.tsx` 中通过 `onEdit` 判断是否展示“调整图表”按钮：

```ts
const shouldShowEditAction = Boolean(onEdit) && !hideEditAction;
```

第一阶段可以通过不传 `onEdit`，或显式传 `hideEditAction`，暂时隐藏“调整图表”按钮。

### 3.5 dashboard item 持久化

dashboard item 的 detail 定义在：

- `wren-ui/src/server/repositories/dashboardItemRepository.ts`

当前 `DashboardItemDetail` 主要包含：

```ts
interface DashboardItemDetail {
  sql: string;
  sqlMode?: 'wren' | 'dialect';
  chartSchema?: Record<string, any>;
  renderHints?: Record<string, any>;
  runtimeIdentity?: {...} | null;
  sourceResponseId?: number | null;
  sourceThreadId?: number | null;
  sourceQuestion?: string | null;
}
```

`detail` 本身是 JSONB 字段，因此可以兼容式新增 `queryControls`，不需要第一阶段新增独立数据表。

### 3.6 看板执行和刷新

当前看板执行 SQL 的位置主要有：

- `wren-ui/src/server/api/dashboardRestShared.ts`
- `wren-ui/src/server/backgrounds/dashboardCacheBackgroundTracker.ts`

两处当前都是直接执行：

```ts
item.detail.sql
```

这意味着如果原始 SQL 中写死了日期：

```sql
WHERE order_date BETWEEN '2026-04-03' AND '2026-04-07'
```

看板刷新时仍然会查询同一段日期，不会随现实时间变化。

## 4. 推荐交互方案

### 4.1 总体原则

**结合现有固定到看板 popover，但不要把复杂配置塞进 popover。**

推荐拆分职责：

| 组件 | 职责 |
| --- | --- |
| `ChartAnswerPinPopover` | 选择目标看板；进入新建看板流程 |
| 新增或扩展的配置 modal/drawer | 配置固定后的日期行为 |
| `createDashboardItem` | 保存 dashboard item 和 `queryControls` |

### 4.2 已有看板固定流程

推荐流程：

```text
点击“固定到看板”
→ 弹出当前 ChartAnswerPinPopover
→ 用户选择目标看板
→ 判断当前 SQL 是否存在可识别日期范围
   → 如果没有：沿用当前逻辑，立即固定
   → 如果有：打开“固定到看板配置”弹窗
→ 用户选择日期行为
→ 确认固定
```

配置弹窗示意：

```text
固定到看板

目标看板：经营日报

数据时间范围

( ) 固定当前日期范围
    始终查询 2026-04-03 至 2026-04-07

(●) 随时间自动滚动
    保持 5 天窗口，随看板刷新自动向后移动

滚动基准：
[ 到昨天 ]  [ 到今天 ]

预览：
如果今天是 2026-05-01，本次刷新会查询 2026-04-26 至 2026-04-30

[取消] [固定到看板]
```

### 4.3 新建看板并固定流程

当前“新建看板并固定”只输入看板名称。后续可以升级为一个更完整的 modal：

```text
新建看板并固定

看板名称：[ 本周经营复盘 ]

数据时间范围
( ) 固定当前日期范围
(●) 随时间自动滚动

[取消] [新建并固定]
```

推荐第一阶段保持用户路径简单：

1. 点击 popover 中的“新建看板并固定”；
2. 打开新建看板 modal；
3. 如果当前 SQL 有日期范围，在同一个 modal 中展示动态日期配置；
4. 点击“新建并固定”时同时创建看板和 dashboard item。

### 4.4 无日期范围的兼容体验

如果当前 SQL 没有识别到日期范围，则不打开额外配置弹窗：

```text
点击固定到看板
→ 选择看板
→ 立即固定成功
```

这样避免对普通图表造成额外打扰。

## 5. 数据模型设计

### 5.1 前端提交参数

扩展 `CreateDashboardItemInput`：

```ts
export type CreateDashboardItemInput = {
  dashboardId?: number | null;
  itemType: string;
  responseId: number;
  queryControls?: DashboardQueryControls | null;
};
```

### 5.2 dashboard item detail

在 `DashboardItemDetail` 中新增可选字段：

```ts
export type DashboardQueryControls = {
  version: 'dashboard-query-controls-v1';
  timeFilters?: DashboardTimeFilterControl[];
};

export type DashboardTimeFilterControl = {
  id: string;
  field: string;
  mode: 'fixed' | 'rolling_window';
  originalStartDate: string;
  originalEndDate: string;
  windowDays: number;
  anchor: 'today' | 'last_complete_day';
  timezone: string;
  sqlBinding: {
    kind: 'between' | 'gte_lte' | 'gte_lt';
    startLiteral: string;
    endLiteral: string;
  };
};
```

示例：

```json
{
  "version": "dashboard-query-controls-v1",
  "timeFilters": [
    {
      "id": "time_filter_1",
      "field": "order_date",
      "mode": "rolling_window",
      "originalStartDate": "2026-04-03",
      "originalEndDate": "2026-04-07",
      "windowDays": 5,
      "anchor": "last_complete_day",
      "timezone": "Asia/Shanghai",
      "sqlBinding": {
        "kind": "between",
        "startLiteral": "2026-04-03",
        "endLiteral": "2026-04-07"
      }
    }
  ]
}
```

### 5.3 字段说明

| 字段 | 说明 |
| --- | --- |
| `version` | 配置版本，方便未来迁移 |
| `field` | 日期字段，例如 `order_date` |
| `mode` | `fixed` 表示固定日期；`rolling_window` 表示滚动窗口 |
| `originalStartDate` | 用户问数时的原始开始日期 |
| `originalEndDate` | 用户问数时的原始结束日期 |
| `windowDays` | 滚动窗口长度，例如 5 天 |
| `anchor` | 滚动窗口锚点，`today` 或 `last_complete_day` |
| `timezone` | 计算日期窗口的时区 |
| `sqlBinding` | 用于安全替换 SQL 中原日期条件的绑定信息 |

## 6. SQL 编译策略

### 6.1 不直接覆盖原 SQL

dashboard item 继续保留原始 SQL：

```ts
detail.sql
```

每次看板预览或刷新时，根据 `queryControls` 生成运行时 SQL：

```text
detail.sql + detail.queryControls + 当前时间
→ compiledSql
→ queryService.preview(compiledSql)
```

### 6.2 编译示例

原始 SQL：

```sql
SELECT
  order_date,
  SUM(amount) AS amount
FROM orders
WHERE order_date BETWEEN '2026-04-03' AND '2026-04-07'
GROUP BY order_date
```

如果今天是 `2026-05-01`，配置为：

```json
{
  "mode": "rolling_window",
  "windowDays": 5,
  "anchor": "last_complete_day"
}
```

则编译后 SQL：

```sql
SELECT
  order_date,
  SUM(amount) AS amount
FROM orders
WHERE order_date BETWEEN '2026-04-26' AND '2026-04-30'
GROUP BY order_date
```

### 6.3 第一阶段支持范围

第一阶段只支持可安全识别的日期范围：

```sql
date_field BETWEEN 'YYYY-MM-DD' AND 'YYYY-MM-DD'
```

以及可选支持：

```sql
date_field >= 'YYYY-MM-DD' AND date_field <= 'YYYY-MM-DD'
date_field >= 'YYYY-MM-DD' AND date_field < 'YYYY-MM-DD'
```

不建议第一阶段支持：

1. 多层嵌套 SQL 中复杂日期表达式；
2. 数据库方言专属日期函数；
3. 非标准自然语言日期；
4. 多个日期字段同时滚动；
5. LLM 重新生成 SQL。

3. service 将 `queryControls` 写入 `detail`。

### 7.2 看板预览

涉及文件：

- `wren-ui/src/server/api/dashboardRestShared.ts`

改造前：

```ts
ctx.queryService.preview(item.detail.sql, ...)
```

改造后：

```ts
const compiledSql = compileDashboardItemSql({
  sql: item.detail.sql,
  queryControls: item.detail.queryControls,
  now: new Date(),
});

ctx.queryService.preview(compiledSql, ...)
```

### 7.3 定时刷新

涉及文件：

- `wren-ui/src/server/backgrounds/dashboardCacheBackgroundTracker.ts`

这里也需要使用同一套 `compileDashboardItemSql`，避免预览和后台刷新行为不一致。

### 7.4 新增工具模块

建议新增：

```text
wren-ui/src/server/utils/dashboardQueryControls.ts
```

职责：

1. 校验 `queryControls`；
2. 根据当前时间计算实际日期窗口；
3. 根据 `sqlBinding` 编译运行时 SQL；
4. 在无法安全编译时降级使用原始 SQL，并记录错误或 warning。

`ChartAnswer.tsx` 中推荐把 pin 流程改成：

```ts
const onSelectDashboard = async (dashboardId, dashboardName) => {
  setIsPinPopoverOpen(false);

  const detectedTimeFilter = detectDashboardTimeFilter(responseSql);

  if (!detectedTimeFilter) {
    await runPinToDashboard(dashboardId, dashboardName);

  setPendingPinTarget({ dashboardId, dashboardName, detectedTimeFilter });

```

```ts
await runPinToDashboard(dashboardId, dashboardName, {
  queryControls,
});
```

## 9. 用户体验细节

### 9.1 默认值

如果识别到日期范围，建议默认选择：

```text
随时间自动滚动
anchor = last_complete_day
```

原因：

1. 看板通常用于持续监控；
2. “到昨天”比“到今天”更稳定，避免当天数据不完整；
3. 更符合日报、周报、经营看板场景。

### 9.2 文案建议

按钮：

```text
固定到看板
```

配置弹窗标题：

```text
固定到看板
```

固定日期：

```text
固定当前日期范围
始终查询 2026-04-03 至 2026-04-07
```

滚动日期：

```text
随时间自动滚动
保持 5 天窗口，随看板刷新自动后移
```

锚点：

```text
到昨天：适合日报、经营看板，避免当天数据不完整
到今天：适合实时数据或当天可完整统计的场景
```

### 9.3 成功提示

固定成功后可以补充说明：

```text

```

固定模式下：

```text
已固定到看板「经营日报」，日期范围保持为 2026-04-03 至 2026-04-07。
```

## 11. 风险与边界

### 11.1 SQL 识别风险

SQL 可能非常复杂，第一阶段不要试图覆盖所有情况。只支持明确、可安全替换的日期范围。

识别不到时不报错，直接使用当前固定逻辑。

### 11.2 多日期字段风险

如果 SQL 中存在多个日期字段，第一阶段可以：

1. 默认只支持一个日期字段；
2. 多个候选时要求用户选择；
3. 或直接降级为固定日期。

### 11.3 数据库方言风险

不同数据源的日期语法可能不同。第一阶段优先支持字符串日期字面量：

```sql
'YYYY-MM-DD'
```

避免一开始支持方言函数，例如：

```sql
DATE_SUB(...)
CURRENT_DATE - INTERVAL ...
```

### 11.4 缓存一致性风险

看板预览和后台刷新必须使用同一个编译函数，否则用户在页面看到的结果和后台缓存结果可能不一致。

## 12. 推荐结论

推荐采用以下方案：

1. **保留并复用现有固定到看板 popover**，它继续负责选择目标看板。
2. **新增固定配置 modal/drawer**，专门处理日期滚动、固定日期、预览范围等配置。
3. **不再把“调整图表样式”作为当前入口暴露给用户**，第一阶段隐藏该按钮。
4. **dashboard item 保存 `queryControls`**，让看板刷新时可以根据当前时间编译出新的 SQL。
5. **第一阶段只支持安全可识别的日期范围**，复杂 SQL 降级为当前固定 SQL 行为。

这个方案既复用了现有固定入口，又避免把复杂配置塞进小 popover；同时为后续扩展“维度调整”“参数化看板卡片”“全局看板筛选器”保留了清晰的数据模型和执行路径。
